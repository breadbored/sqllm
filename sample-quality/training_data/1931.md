Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship

   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why

     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```

## Question
Build a unified user activity log that combines all user actions across the learning platform — course notes, badges, assessments, lesson progress, sentiment responses, course enrollments, knowledge checks, account creation, watchlist additions, client assignments, lab usage, lab servers, and data downloads — into a single event stream with a consistent schema, tagged by data source and action type.

--- AGENT ---

**Step 1 — Identify shapes**

The input is 17 separate staging tables, each representing a different type of user activity. Every table contributes `user_id`, `created`, and `updated` as common columns. Beyond those, each table has a subset of domain-specific columns (e.g. `course_id`, `lesson_id`, `assessment_id`, `question_id`, `knowledge_check_id`, `sentiment_question_id`, etc.).

The target shape is a single unified table with a fixed column set:

```
user_id, created, updated, data_source, action_type, action,
course_id, lesson_id, assessment_id, question_id,
sentiment_question_id, sentiment_question_option_id, learning_cue_id,
user_knowledge_check_answered, knowledge_checks, knowledge_check_questions,
knowledge_check_id
```

Each source table maps its native columns into this fixed schema, filling inapplicable columns with `NULL`. Two metadata columns — `data_source` (string literal identifying the source) and `action_type` / `action` (describing what the user did) — are synthesized per branch.

**Step 2 — Identify sources and relationships**

There are no joins in this query. Each of the 17 branches is an independent `SELECT` from a single staging table. The relationship between branches is purely structural — they are unioned into a common schema. The `UNION ALL` preserves all rows from all sources without deduplication.

The 17 source tables are:
1. `stg_course_notes` — provides `course_id`, `lesson_id`
2. `stg_user_badges` — provides `name` (mapped to `action`)
3. `stg_user_course_assessment_answers` — provides `course_id`, `assessment_id`, `question_id`
4. `stg_user_course_assessment_questions` — provides `course_id`, `assessment_id`, `question_id`
5. `stg_user_course_assessments` — provides `course_id`, `assessment_id`, `completed`, `score`
6. `stg_user_course_lesson_progress` — provides `course_id`, `lesson_id`, `progress_seconds`
7. `stg_user_course_sentiment_question_answers` — provides `sentiment_question_id`, `sentiment_question_option_id`, `learning_cue_id`
8. `stg_user_courses` — provides `course_id`, `status`
9. `stg_user_finished_course_lessons` — provides `course_id`, `lesson_id`
10. `stg_user_finished_knowledge_checks` — provides `course_id`, `lesson_id`, `knowledge_check_id`
11. `stg_user_knowledge_check_answers` — provides `course_id`, `lesson_id`, `knowledge_check_id`, `answer_option_id`, `correct`
12. `stg_users` — uses `id` as `user_id`; provides `created`, `updated`
13. `stg_watchlist` — no domain columns beyond the common set
14. `stg_client_users` — no domain columns; empty `action_type` and `action`
15. `stg_user_lab_usage` — no domain columns; empty `action_type` and `action`
16. `stg_user_course_lab_servers` — no domain columns; empty `action_type` and `action`
17. `stg_user_data_downloads` — no domain columns; empty `action_type` and `action`

**Step 3 — Identify aggregation depth**

There is no aggregation in this query. It is a pure row-level union — every row from every source table appears exactly once in the output. The transformation is structural (schema alignment), not analytical.

Arrow notation for completeness:

```
(no aggregation — row-level pass-through across 17 branches)
```

**Step 4 — Identify dependencies**

- No column depends on an aggregation or intermediate computation.
- The `action` column in several branches is derived from source-specific columns via `CAST` or `CONCAT`:
  - `stg_user_course_assessments`: `concat(completed, '|', score)`
  - `stg_user_course_lesson_progress`: `cast(progress_seconds AS string)`
  - `stg_user_course_sentiment_question_answers`: `cast(sentiment_question_option_id AS string)`
  - `stg_user_knowledge_check_answers`: `concat(answer_option_id, '|', correct)`
  - `stg_users`: `concat(created, '|', updated)`
- The `stg_users` branch remaps `id` to `user_id`, unlike all other branches which use `user_id` natively.
- The `data_source` and `action_type` columns are hardcoded string literals per branch — they are tagging constants, not derived from source data.

**Step 5 — Plan CTE layers bottom-up**

No CTEs are needed. The query is a flat `UNION ALL` of 17 `SELECT` statements, each performing schema alignment inline. Each branch is self-contained with no shared intermediate computation.

**Step 6 — Write each CTE with justification**

No CTEs exist in this query. Each branch handles its own column mapping, NULL-filling, and literal tagging inline.
