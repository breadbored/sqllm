## Question
For each resource-opened event, show the user's course progress at the time of the event alongside the resource details, current course progress snapshot, and associated client information.

--- AGENT ---

## 1. Identify Shapes

**Input tables and their observable columns:**

- `agg_resource_events` (e): `user_id, course_id, action, related_id, course_resource_id, lesson_resource_id, action_meta (JSON with $.resource_name), name, created`
- `stg_course_resources` (cr): `id, url, file`
- `stg_lesson_resources` (lr): `id, url, file`
- `stg_user_course_lesson_progress` (uclp): `user_id, course_id, lesson_id, progress_seconds, interval_seconds, created`
- `stg_events_app_speedy` (e inner): `user_id, course_id, created`
- `stg_lessons` (l): `id, video_id`
- `stg_videos` (v): `id, duration`
- `stg_course_lessons` (cl): `course_id, lesson_id`
- `agg_user_course_progress` (ucp): `user_id, course_id, container_id, course_name, course_status, course_progress_minutes, course_progress_percentage, course_length_minutes, time_on_course_minutes`
- `stg_clients_course_containers` (ccc): `course_container_id, client_id`
- `stg_clients` (clients): `id, name`

**Target shape (one row per resource-opened event):**

`user_id, course_id, container_id, course_name, action, resource_id, course_resource_id, lesson_resource_id, resource_name, resource_url, course_progress_minutes, course_length_minutes, course_progress_percentage, time_on_course_minutes, course_status, current_course_progress_minutes, current_course_progress_percentage, current_course_length_minutes, current_time_on_course_minutes, created, client_id (array), client_name (array)`

## 2. Identify Sources and Relationships

- `e -> cr`: `cr.id = e.course_resource_id` — resource metadata at the course level
- `e -> lr`: `lr.id = e.lesson_resource_id` — resource metadata at the lesson level
- `e -> ulp` (inline subquery): `e.user_id = ulp.user_id AND e.course_id = ulp.course_id` — user's historical course progress, pre-aggregated
- `e -> cd` (inline subquery): `e.course_id = cd.course_id` — total course duration
- `e -> ucp`: `e.user_id = ucp.user_id AND e.course_id = ucp.course_id` — current course progress snapshot
- `ucp -> ccc`: `ucp.container_id = ccc.course_container_id` — maps course containers to clients
- `ccc -> clients`: `clients.id = ccc.client_id` — client name lookup

**Critical indirect relationship inside `ulp`:**
- `stg_user_course_lesson_progress` (uclp) joins to `stg_events_app_speedy` (e inner) on `user_id, course_id` with a temporal constraint `uclp.created <= e.created`. This is a time-bounded join: progress records are only included if they existed at or before the event timestamp. This produces progress-at-time-of-event rather than current progress.

**Critical indirect relationship inside `cd`:**
- `stg_lessons` -> `stg_videos` via `l.video_id = v.id` — gets video duration per lesson
- `stg_lessons` -> `stg_course_lessons` via `l.id = cl.lesson_id` — maps lessons to courses
- These together produce total course duration by summing video durations per course.

**Note on the one-to-many at the outer level:** A single event can map to multiple clients through `ucp.container_id -> ccc -> clients`. This is why the outer query uses `GROUP BY` with `ARRAY_AGG(DISTINCT ...)` for client columns — it collapses the client fan-out back to one row per event.

## 3. Identify Aggregation Depth

There are three distinct aggregation levels, two inside inline subqueries and one at the outer query:

**Course duration (cd):**
```
course_id, lesson_id (implied by l.id), v.duration
-> course_id, SUM(duration) as course_duration_minutes
```
Single-level: sum video durations grouped by course.

**User lesson progress (ulp) — two levels:**
```
[uclp joined to events on user_id, course_id WHERE uclp.created <= e.created]
user_id, course_id, lesson_id, MAX(progress_seconds) as user_total_lesson_progress, SUM(interval_seconds) as time_on_lesson
-> user_id, course_id, SUM(user_total_lesson_progress)/60 as user_total_lesson_progress, SUM(time_on_lesson) as time_on_course
```
Level 1: per user/course/lesson, get max progress and total interval time (only for progress records that existed at or before the event).
Level 2: per user/course, sum lesson-level progress and time into course-level totals.

**Outer query — single level:**
```
(all event-grain columns), ANY_VALUE(ulp.user_total_lesson_progress) as course_progress_minutes, ANY_VALUE(cd.course_duration_minutes) as course_length_minutes, ARRAY_AGG(DISTINCT client_id), ARRAY_AGG(DISTINCT client_name)
```
This GROUP BY exists solely to collapse the client fan-out. The `ANY_VALUE` calls confirm that `ulp` and `cd` values are functionally determined by the group keys — they're already 1:1 with the event grain, but the join to clients creates duplicates that must be collapsed.

## 4. Identify Dependencies

**Grouping drivers:**
- `user_id, course_id, lesson_id` drive the inner lesson-progress aggregation
- `user_id, course_id` drive the course-level progress rollup
- `course_id` drives the course duration aggregation
- The full set of event-level columns drives the outer GROUP BY

**Derived columns:**
- `user_total_lesson_progress` — derived from `MAX(progress_seconds)` per lesson, then `SUM/60` per course
- `time_on_course` — derived from `SUM(interval_seconds)` per lesson, then `SUM` per course
- `course_duration_minutes` — derived from `SUM(v.duration)/60` per course
- `course_progress_percentage` — derived from `user_total_lesson_progress / course_duration_minutes`, capped at 1
- `resource_url` — `COALESCE` across lesson and course resource URLs/files
- `client_id`, `client_name` — `ARRAY_AGG(DISTINCT ...)` to flatten the many-clients-per-container relationship

**Intermediate step requirements:**
- `ulp` must be fully pre-aggregated before joining to events, because the temporal join (`uclp.created <= e.created`) must be resolved inside the subquery, not at the outer level
- `cd` must be pre-aggregated because course duration is a per-course scalar derived from a lessons-videos-course_lessons chain

## 5. Plan CTE Layers Bottom-Up

The original query uses inline subqueries rather than CTEs, but the logical layers are:

1. **`cd` (course duration)** — joins lessons to videos and course_lessons, sums video duration per course to produce `course_duration_minutes`.

2. **`lesson_progress` (inner subquery of ulp)** — joins `stg_user_course_lesson_progress` to `stg_events_app_speedy` with the temporal constraint `uclp.created <= e.created`, then aggregates per `user_id, course_id, lesson_id` to get `MAX(progress_seconds)` and `SUM(interval_seconds)` per lesson.

3. **`ulp` (user-level progress)** — aggregates `lesson_progress` up to `user_id, course_id`, summing lesson-level progress (converted to minutes) and time-on-lesson into course-level totals.

4. **Final query** — joins `agg_resource_events` to `cr`, `lr` (resource details), `ulp` (historical progress), `cd` (course duration), `ucp` (current progress snapshot), `ccc` and `clients` (client info). Groups by event-grain columns and uses `ARRAY_AGG(DISTINCT ...)` to flatten the client dimension.

## 6. Write Each CTE with Justification

**`cd` subquery:** Exists because course duration is not stored directly — it must be computed by summing video durations across all lessons in a course, requiring a three-table join (lessons → videos for duration, lessons → course_lessons for course mapping).

**`lesson_progress` inner subquery:** Exists because progress must be filtered to only records created at or before the event timestamp (`uclp.created <= e.created`), and then aggregated per lesson to get the max progress point and total time spent. This temporal filtering is the mechanism that produces "progress at time of event" rather than current progress.

**`ulp` outer subquery:** Exists because the final query needs course-level progress, not lesson-level. This rolls up the per-lesson maximums into a per-user-per-course total, converting seconds to minutes.

**Outer GROUP BY:** Exists because the join path `ucp -> ccc -> clients` can produce multiple rows per event (one course container can belong to multiple clients). The `ARRAY_AGG(DISTINCT ...)` collapses these back to one row per event while preserving all client associations.

## 7. Compose the Final Query

```sql
SELECT
  e.user_id,
  e.course_id,
  ucp.container_id,
  ucp.course_name,
  e.action,
  e.related_id AS resource_id,
  e.course_resource_id,
  e.lesson_resource_id,
  LOWER(JSON_EXTRACT_SCALAR(action_meta, '$.resource_name')) AS resource_name,
  COALESCE(lr.url, cr.url, lr.file, cr.file) AS resource_url,
  ANY_VALUE(ulp.user_total_lesson_progress) AS course_progress_minutes,
  ANY_VALUE(cd.course_duration_minutes) AS course_length_minutes,
  LEAST( ROUND(((ANY_VALUE(ulp.user_total_lesson_progress) / ANY_VALUE(cd.course_duration_minutes))), 2), 1 ) AS course_progress_percentage,
  ANY_VALUE(ROUND((ulp.time_on_course / 60), 3)) AS time_on_course_minutes,
  ucp.course_status,
  ucp.course_progress_minutes AS current_course_progress_minutes,
  ucp.course_progress_percentage AS current_course_progress_percentage,
  ucp.course_length_minutes AS current_course_length_minutes,
  ucp.time_on_course_minutes AS current_time_on_course_minutes,
  e.created,
  ARRAY_AGG(DISTINCT ccc.client_id) AS client_id,
  ARRAY_AGG(DISTINCT clients.name) AS client_name
FROM
  ${ref("agg_resource_events")} AS e
LEFT JOIN
  ${ref("stg_course_resources")}cr
ON
  cr.id = e.course_resource_id
LEFT JOIN
  ${ref("stg_lesson_resources")} AS lr
ON
  e.lesson_resource_id = lr.id
  -- get course progress at time of resource click
INNER JOIN (
  SELECT
    lesson_progress.user_id,
    lesson_progress.course_id,
    ROUND((SUM(lesson_progress.user_total_lesson_progress) / 60), 3) AS user_total_lesson_progress,
    SUM(time_on_lesson) AS time_on_course
  FROM (
    SELECT
      uclp.user_id,
      uclp.course_id,
      MAX(progress_seconds) AS user_total_lesson_progress,
      SUM(interval_seconds) AS time_on_lesson
    FROM
      ${ref("stg_user_course_lesson_progress")} AS uclp
    INNER JOIN
      ${ref("stg_events_app_speedy")} AS e
    ON
      uclp.user_id = e.user_id
      AND uclp.course_id = e.course_id
    WHERE
      uclp.user_id IS NOT NULL
      AND uclp.created <= e.created
    GROUP BY
      uclp.user_id,
      uclp.course_id,
      uclp.lesson_id ) AS lesson_progress
  GROUP BY
    lesson_progress.user_id,
    lesson_progress.course_id ) AS ulp
ON
  e.course_id = ulp.course_id
  AND e.user_id = ulp.user_id
  --get course duration
INNER JOIN (
  SELECT
    cl.course_id,
    ROUND((SUM(v.duration) / 60), 3) AS course_duration_minutes
  FROM
    ${ref("stg_lessons")} AS l
  LEFT JOIN
    ${ref("stg_videos")} AS v
  ON
    l.video_id = v.id
  LEFT JOIN
    ${ref("stg_course_lessons")} AS cl
  ON
    l.id = cl.lesson_id
  GROUP BY
    course_id
  ORDER BY
    course_id ) AS cd
ON
  e.course_id = cd.course_id
INNER JOIN
  ${ref("agg_user_course_progress")} AS ucp
ON
  e.course_id = ucp.course_id
  AND e.user_id = ucp.user_id
INNER JOIN
  ${ref("stg_clients_course_containers")} AS ccc
ON
  ucp.container_id = ccc.course_container_id
INNER JOIN
  ${ref("stg_clients")} AS clients
ON
  clients.id = ccc.client_id
WHERE
  e.name = 'opened_resource'
GROUP BY
  e.user_id,
  e.course_id,
  ucp.container_id,
  ucp.course_name,
  e.action,
  e.related_id,
  e.course_resource_id,
  e.lesson_resource_id,
  LOWER(JSON_EXTRACT_SCALAR(action_meta, '$.resource_name')),
  COALESCE(lr.url, cr.url, lr.file, cr.file),
  ucp.course_status,
  ucp.course_progress_minutes,
  ucp.course_progress_percentage,
  ucp.course_length_minutes,
  ucp.time_on_course_minutes,
  e.created
```
