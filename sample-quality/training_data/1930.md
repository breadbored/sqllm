--- USER ---

Enrich app events with their associated resource IDs, resolving each event to either a course resource or a lesson resource based on the event's related entity type. The join logic must handle both the legacy format (where `related_entity = 'resources'`) and the newer format introduced in July 2024 (where `related_entity` is `'course_resource'` or `'lesson_resource'`).

--- AGENT ---

## 1. Identify shapes

**Input tables (inferred from query):**

- `stg_events_app_speedy` (aliased `e`): contains at minimum `id`, `related_id`, `related_entity`, `course_id`, `action_meta`, plus additional columns captured by `e.*`
- `course_resources` (aliased `cr`): contains at minimum `id`, `course_id`
- `stg_lesson_resources` (aliased `lr`): contains at minimum `id`, `lesson_id`

**Target shape:**

All columns from `stg_events_app_speedy`, plus two nullable foreign keys:
- `course_resource_id` (non-null when the event relates to a course resource)
- `lesson_resource_id` (non-null when the event relates to a lesson resource)

Each output row is one event that successfully matched to exactly one resource type.

## 2. Identify sources and relationships

There are two independent join paths from events to resources:

**Path 1 — Course resources:**
`stg_events_app_speedy.related_id = course_resources.id`, with an additional guard:
- Legacy mode: `related_entity = 'resources'` AND `e.course_id = cr.course_id` (disambiguates because the same `related_id` could collide across resource types in the legacy scheme)
- New mode: `related_entity = 'course_resource'` (the entity type is unambiguous, but `e.course_id = cr.course_id` is still enforced as a top-level AND condition on the join)

**Path 2 — Lesson resources:**
`stg_events_app_speedy.related_id = stg_lesson_resources.id`, with:
- Legacy mode: `related_entity = 'resources'` AND `lr.lesson_id = SAFE_CAST(JSON_EXTRACT_SCALAR(e.action_meta, '$.lesson_id') AS INT64)` (the lesson_id is extracted from a JSON metadata field to disambiguate)
- New mode: `related_entity = 'lesson_resource'`

These two paths are mutually exclusive per event (an event relates to either a course resource or a lesson resource, never both), so UNION ALL is the correct combiner — no deduplication needed.

The final SELECT re-joins the resolved resource IDs back to the full event row via `event_id = e.id`.

## 3. Identify aggregation depth

There is no aggregation in this query. The transformation is purely structural: a fan-out into two join paths, union, then re-join to retrieve the full event columns.

Arrow notation for the shape transformation (no aggregation, but expressing the row-level mapping):

```
event_id, course_resource_id, lesson_resource_id (from UNION ALL of two INNER JOINs)
-> event_id joined back to e.* → e.*, course_resource_id, lesson_resource_id
```

No GROUP BY, no AGG functions. One input event produces at most one output row (assuming the join conditions are selective enough to be 1:1 per path).

## 4. Identify dependencies

- The two UNION ALL branches are independent — they share the same events table but join to different resource tables with mutually exclusive `related_entity` values.
- The final SELECT depends on the CTE completing first, because it needs `event_id` to re-join to the full event row.
- The `e.*` expansion requires the re-join because the CTE only carries `event_id` and the two resource ID columns — it does not carry all event columns through the union (which would require enumerating them and risk schema drift).

## 5. Plan CTE layers bottom-up

1. **`joined_resources`** — Resolves each event to its resource ID (course or lesson) via two UNION ALL branches, producing a narrow three-column mapping: `(event_id, course_resource_id, lesson_resource_id)`.

2. **Final SELECT** — Re-joins the narrow mapping back to the full events table to hydrate `e.*` alongside the resolved resource IDs.

## 6. Write each CTE with justification

### CTE: `joined_resources`

**Why this layer exists:** The events table stores a polymorphic foreign key (`related_id` + `related_entity`) that can point to either `course_resources` or `stg_lesson_resources`. This CTE resolves the polymorphism by attempting both join paths in separate branches and unioning the results, producing a clean `(event_id, course_resource_id, lesson_resource_id)` mapping where exactly one of the two resource ID columns is non-null per row.

**Branch 1 — Course Resources:**
- INNER JOIN on `e.related_id = cr.id` with `e.course_id = cr.course_id`
- Legacy guard: `related_entity = 'resources'` with course_id match
- New guard: `related_entity = 'course_resource'`
- Outputs: `event_id`, `course_resource_id`, `NULL AS lesson_resource_id`

**Branch 2 — Lesson Resources:**
- INNER JOIN on `e.related_id = lr.id`
- Legacy guard: `related_entity = 'resources'` with lesson_id extracted from `action_meta` JSON
- New guard: `related_entity = 'lesson_resource'`
- Outputs: `event_id`, `NULL AS course_resource_id`, `lesson_resource_id`

### Final SELECT

**Why this layer exists:** The CTE intentionally carries only the narrow mapping columns to avoid duplicating all event columns across both UNION ALL branches. The final join hydrates the full event row from the source table using the `event_id` key.

## 7. Compose the final query

```sql
WITH joined_resources AS (
  -- Course Resources
  SELECT
    e.id  AS event_id,
    cr.id AS course_resource_id,
    NULL  AS lesson_resource_id
  FROM
    ${ref("stg_events_app_speedy")} AS e
  INNER JOIN
    ${ref("course_resources")} AS cr
    ON
      e.related_id = cr.id
      AND
      (
      -- Backwards compatibility
        (
          e.related_entity = 'resources'
          AND
          e.course_id = cr.course_id
        )
        OR
        -- From event changes in July 2024
        e.related_entity = 'course_resource'
      )
      AND
      e.course_id = cr.course_id

  UNION ALL

  -- Lesson Resources
  SELECT
    e.id  AS event_id,
    NULL  AS course_resource_id,
    lr.id AS lesson_resource_id
  FROM
    ${ref("stg_events_app_speedy")} AS e
  INNER JOIN
    ${ref("stg_lesson_resources")} AS lr
    ON
      e.related_id = lr.id
      AND
      (
      -- Backwards compatibility
        (
          e.related_entity = 'resources'
          AND
          lr.lesson_id = safe_cast(json_extract_scalar(e.action_meta, '$.lesson_id') AS int64)
        )
        OR
        -- From event changes in July 2024
        e.related_entity = 'lesson_resource'
      )
)

SELECT
  e.*,
  jr.course_resource_id,
  jr.lesson_resource_id
FROM
  joined_resources AS jr
INNER JOIN
  ${ref("stg_events_app_speedy")} AS e
  ON
    jr.event_id = e.id
```

## 8. Verify output shape

**Target shape stated in step 1:** All columns from `stg_events_app_speedy` (`e.*`), plus `course_resource_id` (nullable INT64), plus `lesson_resource_id` (nullable INT64).

**Final SELECT produces:** `e.*` (all event columns), `jr.course_resource_id`, `jr.lesson_resource_id`.

✅ Output shape matches. Each row is one event that successfully resolved to a resource. Events that match neither resource table are excluded (INNER JOIN in both the CTE branches and the final join). Exactly one of `course_resource_id` or `lesson_resource_id` is non-null per row.
