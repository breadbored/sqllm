--- USER ---

For every user course assessment question record, produce a `client_ids` array indicating which client(s) have visibility over that record. Users who are linked to an external client should carry both the default client ID (`1`) and their external client ID; all other users should carry only the default client ID (`[1]`).

The output is a pass-through of all columns from `user_course_assessment_questions` with one derived column — `client_ids` — appended.

---

**Step 2: Identify sources and relationships**

Two tables are involved:

- `user_course_assessment_questions` — the driving table; one row per assessment question attempt per user.
- `users` — the lookup table; used only to retrieve `external_client_id`.

The natural join key is `user_id`: `user_course_assessment_questions.user_id` matches `users.id`. The join is a LEFT JOIN because not all users will have an `external_client_id`, and assessment question rows must still appear even when no external client is associated.

The `user_clients` CTE pre-filters `users` to only those with a non-null `external_client_id`, so the LEFT JOIN null-checks are handled by `COALESCE` rather than inline `CASE` expressions.

---

**Step 3: Identify aggregation depth**

There is no multi-level aggregation in this query. The CTE is a filtering and projection step, not an aggregation. The main query is a keyed LEFT JOIN enrichment. The full shape transformation in arrow notation:

```
user_id, [1, external_client_id] as client_ids        -- filtered: external_client_id IS NOT NULL
  -> user_course_assessment_questions.*, COALESCE(client_ids, [1]) as client_ids
```

One CTE, one final SELECT. No aggregation arrows required — the arrow here represents a join enrichment, not a GROUP BY reduction.

---

**Step 4: Identify dependencies**

- `client_ids` on the output depends on `users.external_client_id`, which is not present in `user_course_assessment_questions` and must be joined in.
- The COALESCE default (`[1]`) handles two cases: (a) users whose `external_client_id IS NULL` were excluded from the CTE, so the LEFT JOIN produces a NULL `client_ids`; (b) users not found in `users` at all. Both fall back to `[1]`.
- The array literal `[1, external_client_id]` expresses that every user implicitly belongs to client `1` (the default/internal client), and externally-linked users belong to that client as well.
- No grouping keys are needed — the relationship is one-to-many from `users` to `user_course_assessment_questions`, but the LEFT JOIN is on `user_id`, so each question row gets exactly one `client_ids` value.

---

**Step 5: Plan CTE layers bottom-up**

1. **`user_clients`** — Filters `users` to those with a non-null `external_client_id` and constructs the `client_ids` array as `[1, external_client_id]`, keyed by `user_id`. This is the lookup table the main query joins against.

---

**Step 6: Write each CTE with justification**

**`user_clients`** — This layer exists to pre-compute the two-element `client_ids` array for users who have an external client affiliation, so the main query can LEFT JOIN on `user_id` and fall back to the single-element default `[1]` via COALESCE for users who do not.

---

**Step 7: Compose the final query**

```sql
WITH
user_clients AS (
  SELECT
    u.id as user_id,
    [1, u.external_client_id] as client_ids
  FROM
    ${ref("users")} as u
  WHERE
    u.external_client_id IS NOT NULL
)
SELECT
  _utb.id,
  _utb.user_id,
  _utb.course_id,
  _utb.assessment_id,
  _utb.question_id,
  _utb.user_course_assessment_id,
  _utb.completed,
  _utb.score,
  _utb.created,
  _utb.updated,
  COALESCE(ucl.client_ids, [1]) as client_ids
FROM
  ${ref("user_course_assessment_questions")} as _utb
LEFT JOIN
  user_clients as ucl
ON
  ucl.user_id = _utb.user_id
```
