## Question
For each documentation page, show daily engagement metrics: how many unique sessions visited the page, the average time spent per session, and the total number of actions taken.

--- AGENT ---

## 1. Identify shapes

**Input shape (single table):**
`de.url_path, de.session_id, de.timestamp, de.action` — from `stg_dd_docs_events`, a documentation site event log.

**Target shape:**
`url_path, day, sessions, avg_session_duration, total_action_count` — one row per page per day with aggregated engagement metrics.

## 2. Identify sources and relationships

Single source table: `stg_dd_docs_events` (aliased `de`). No joins required. Each row is an event (an action taken by a session on a URL at a timestamp).

Key structural observations:
- `url_path` identifies the documentation page
- `session_id` groups events into browsing sessions
- `timestamp` provides the time dimension and is used both for day truncation and for computing session duration
- `action` is counted to measure engagement volume
- `url_path` can be NULL and those rows are excluded

## 3. Identify aggregation depth

Two levels of aggregation are required. The final output wants `avg(session_duration)` per page per day, but session duration is itself derived from `MAX(timestamp) - MIN(timestamp)` within a session. You cannot compute the average of a derived duration in a single pass — the duration must be materialized first at the session grain, then averaged at the page-day grain.

Arrow notation:

```
url_path, session_id, day, TIMESTAMP_DIFF(MAX(timestamp), MIN(timestamp), SECOND) as session_duration, COUNT(action) as action_count
-> url_path, day, COUNT(DISTINCT session_id) as sessions, AVG(session_duration) as avg_session_duration, SUM(action_count) as total_action_count
```

## 4. Identify dependencies

- **Grouping drivers (level 1):** `url_path`, `session_id`, `day` — the session-page-day grain
- **Derived at level 1:** `session_duration` (from min/max timestamp), `action_count` (from counting action rows)
- **Grouping drivers (level 2):** `url_path`, `day` — the page-day grain
- **Derived at level 2:** `sessions` (distinct count of session_id), `avg_session_duration` (average of the level-1 duration), `total_action_count` (sum of level-1 action counts)
- `session_id` is consumed at level 1 and dropped (counted distinctly) at level 2
- `day` is derived from `timestamp` via `DATETIME_TRUNC(CAST(timestamp AS TIMESTAMP), DAY)` — this truncation is part of the level-1 GROUP BY, meaning sessions that span midnight are split across days

## 5. Plan CTE layers bottom-up

1. **`session_durations`** — Aggregates raw events to the session-page-day grain, producing `session_duration` and `action_count` per session per page per day.
2. **Final SELECT** — Rolls up to the page-day grain, counting distinct sessions, averaging session duration, and summing action counts.

## 6. Write each CTE with justification

**`session_durations`:** Materializes session-level metrics (duration and action count) at the `url_path, session_id, day` grain. This is necessary because session duration is a MIN/MAX span that must exist as a concrete value before it can be averaged in the next layer. Also filters out NULL `url_path` rows.

**Final SELECT:** Collapses the session grain into page-day grain, producing the user-facing engagement metrics. Orders by session count descending to surface the most-visited pages first.

## 7. Compose the final query

```sql
WITH session_durations AS (
  SELECT
    de.url_path,
    de.session_id,
    datetime_trunc(cast(timestamp AS timestamp), DAY) AS day,
    timestamp_diff(
      timestamp(max(de.timestamp)),
      timestamp(min(de.timestamp)),
      SECOND
    )                                                 AS session_duration,
    count(de.action)                                  AS action_count
  FROM ${ref("stg_dd_docs_events")} AS de
  WHERE url_path IS NOT NULL
  GROUP BY de.url_path, de.session_id, datetime_trunc(cast(timestamp AS timestamp), DAY)
)

SELECT
  url_path,
  day,
  count(DISTINCT session_id)   AS sessions,
  round(avg(session_duration)) AS avg_session_duration,
  sum(action_count)            AS total_action_count
FROM session_durations
GROUP BY url_path, day
ORDER BY sessions DESC
```

## 8. Verify output shape

Target shape: `url_path, day, sessions, avg_session_duration, total_action_count`

Final SELECT produces: `url_path, day, sessions (COUNT DISTINCT), avg_session_duration (ROUND(AVG(...))), total_action_count (SUM(...))` — grouped by `url_path, day`, ordered by `sessions DESC`.

Output shape matches the target. One row per page per day. ✓
