Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```

     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     ```

## Tables
### agg_client_transactions
| column | type |
|---|---|
| user_id | INT64 |

## Question
Generate a monthly report per client showing unique users, new vs returning user counts, session counts, and sessions per user. A user is "new" on a given day if that day matches their first-ever transaction day for that client, and "returning" otherwise. Also flag users who appear in both new and returning categories within the same month.

--- AGENT ---

## Step 1: Identify Shapes

**Input shape:**

The primary table is `agg_client_transactions` with columns:
- `ct.user_id` — user identifier
- `ct.client_id` — a repeated/array field (since it is UNNESTed) containing client identifiers
- `ct.day` — the date of the transaction/session (backtick-quoted, so likely a reserved word or date column)
- `DAY` — a standalone column reference (not a function keyword here) present on `agg_client_transactions`, representing the current row's day

**Target shape (one row per month per client):**
- `unique_users` — distinct users
- `new_users` — distinct users whose first-ever day for this client equals the row's day
- `returning_users` — distinct users whose first-ever day differs
- `new_users_returning` — overlap count (users counted as both new and returning across days within the month)
- `monthly_new_users` — distinct users whose first-ever month matches the row's month
- `monthly_returning_users` — distinct users whose first-ever month differs
- `total_sessions` — total row count of transactions
- `returning_user_sessions` — session count for returning users
- `sessions_per_user` — total_sessions / unique_users, rounded
- `year_month` — formatted month label
- `month` — truncated month date
- `client_id` — the client identifier

## Step 2: Identify Sources and Relationships

**Tables involved:**
1. `agg_client_transactions` (aliased `ct`) — the fact table, used twice: once in the CTE and once in the final SELECT
2. `client_transactions_min` (CTE) — derived from `agg_client_transactions`, computes each user's earliest day per client
3. `meta_quest_min` (CTE) — passthrough of `client_transactions_min` (structurally identical, likely a naming convention or placeholder for additional filtering)

**Join relationships:**
- `agg_client_transactions.client_id` is an ARRAY field. It is UNNESTed in both the CTE (`UNNEST(ct.client_id) AS cid`) and the final query (`ct.client_id AS cl` — BigQuery's implicit UNNEST via comma-join syntax).
- The final query joins to `meta_quest_min` on **two columns**: `mqm.user_id = ct.user_id AND mqm.client_id = cl`. This is a tight join on both the user and the specific unnested client, ensuring the min_day lookup is scoped to the same user–client pair.

**Key observation:** The `DAY` reference in the final SELECT (e.g., `DATE_TRUNC(DAY, DAY)`, `DATE_TRUNC(DAY, MONTH)`) is a **column** on `agg_client_transactions`, not the SQL keyword. It represents the transaction date of the current row. This is how each row's date is compared against the user's first-ever date (`mqm.min_day`).

## Step 3: Identify Aggregation Depth

There are **two levels** of aggregation:

```
user_id, client_id (unnested), MIN(day) as min_day
-> [joined back to fact table on user_id, client_id]
   FORMAT_DATE('%b %y', DATE_TRUNC(DAY, MONTH)), DATE_TRUNC(DAY, MONTH), client_id,
   COUNT(DISTINCT user_id) as unique_users,
   COUNT(DISTINCT CASE WHEN min_day_trunc = day_trunc THEN user_id END) as new_users,
   COUNT(DISTINCT CASE WHEN min_day_trunc != day_trunc THEN user_id END) as returning_users,
   (new_users + returning_users) - unique_users as new_users_returning,
   COUNT(DISTINCT CASE WHEN min_month = day_month THEN user_id END) as monthly_new_users,
   COUNT(DISTINCT CASE WHEN min_month != day_month THEN user_id END) as monthly_returning_users,
   COUNT(user_id) as total_sessions,
   COUNT(CASE WHEN min_day_trunc != day_trunc THEN user_id END) as returning_user_sessions,
   ROUND(COUNT(user_id) / COUNT(DISTINCT user_id), 2) as sessions_per_user
```

The first level (pre-aggregation in the CTE) computes the per-user, per-client minimum day. The second level is the final monthly-client aggregation that uses that pre-computed minimum to classify each row.

## Step 4: Identify Dependencies

- **Grouping drivers:** The final GROUP BY is on `year_month` (formatted), `month` (truncated date), and `client_id` (unnested). These define the grain of the output.
- **Derived columns:** All COUNT DISTINCT metrics depend on comparing `mqm.min_day` (from the CTE) against `DAY` (from the fact row). This comparison drives the new/returning classification.
- **Intermediate step required:** `min_day` must be computed per user–client pair before the final join, because the classification logic (`min_day = DAY` vs `min_day != DAY`) requires knowing each user's historical first day at the time of the row-level join.
- **The `new_users_returning` metric** is an arithmetic expression: `(returning_users + new_users) - unique_users`. This captures users who were "new" on one day of the month but "returning" on another day — they get counted in both conditional DISTINCT counts but only once in the overall DISTINCT, so the difference reveals the overlap.

## Step 5: Plan CTE Layers Bottom-Up

1. **`client_transactions_min`** — Unnests the client_id array and computes the earliest transaction day (`MIN(day)`) per user–client pair across all history.

2. **`meta_quest_min`** — A passthrough select from `client_transactions_min`. Structurally a no-op in this query; likely exists as a naming convention or placeholder for additional filtering (e.g., restricting to a specific client type) that was removed or not yet applied.

3. **Final SELECT** — Joins the fact table (with unnested client_id) back to `meta_quest_min` on user_id + client_id, then aggregates by month + client to produce the user classification metrics.

## Step 6: Write Each CTE with Justification

**`client_transactions_min`** — Pre-aggregates the minimum transaction day per user–client pair, which is required before the final query can classify each transaction row as "new" or "returning" for that user.

**`meta_quest_min`** — Provides a named abstraction layer over the min-day data; in the current form it is a direct passthrough but preserves the option to apply additional filters or join to other metadata without modifying the base CTE.

## Step 7: Compose the Final Query

```sql
WITH
  client_transactions_min AS (
  SELECT
    ct.user_id,
    cid AS client_id,
    MIN(ct.`day`) AS min_day
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS cid
  GROUP BY
    ct.user_id,
    cid ),
  meta_quest_min AS (
  SELECT
    ctm.user_id,
    ctm.client_id,
    ctm.min_day
  FROM
    client_transactions_min AS ctm )
SELECT
  COUNT(DISTINCT ct.user_id) AS unique_users,
  COUNT( DISTINCT
    CASE
      WHEN DATE_TRUNC(mqm.min_day, DAY) = DATE_TRUNC(DAY, DAY) THEN ct.user_id
  END
    ) AS new_users,
  COUNT( DISTINCT
    CASE
      WHEN DATE_TRUNC(mqm.min_day, DAY) != DATE_TRUNC(DAY, DAY) THEN ct.user_id
  END
    ) AS returning_users,
  ( COUNT( DISTINCT
      CASE
        WHEN DATE_TRUNC(mqm.min_day, DAY) != DATE_TRUNC(DAY, DAY) THEN ct.user_id
    END
      ) + COUNT( DISTINCT
      CASE
        WHEN DATE_TRUNC(mqm.min_day, DAY) = DATE_TRUNC(DAY, DAY) THEN ct.user_id
    END
      ) ) - COUNT(DISTINCT ct.user_id) AS new_users_returning,
  COUNT( DISTINCT
    CASE
      WHEN DATE_TRUNC(mqm.min_day, MONTH) = DATE_TRUNC(DAY, MONTH) THEN ct.user_id
  END
    ) AS monthly_new_users,
  COUNT( DISTINCT
    CASE
      WHEN DATE_TRUNC(mqm.min_day, MONTH) != DATE_TRUNC(DAY, MONTH) THEN ct.user_id
  END
    ) AS monthly_returning_users,
  COUNT(ct.user_id) AS total_sessions,
  COUNT(
    CASE
      WHEN DATE_TRUNC(mqm.min_day, DAY) != DATE_TRUNC(DAY, DAY) THEN ct.user_id
  END
    ) AS returning_user_sessions,
  ROUND(COUNT(ct.user_id) / COUNT(DISTINCT ct.user_id), 2) AS sessions_per_user,
  FORMAT_DATE('%b %y', DATE_TRUNC(DAY, MONTH)) AS year_month,
  DATE_TRUNC(DAY, MONTH) AS month,
  cl AS client_id
FROM
  ${ref("agg_client_transactions")} AS ct,
  ct.client_id AS cl
INNER JOIN
  meta_quest_min AS mqm
ON
  mqm.user_id = ct.user_id
  AND mqm.client_id = cl
GROUP BY
  FORMAT_DATE('%b %y', DATE_TRUNC(DAY, MONTH)),
  DATE_TRUNC(DAY, MONTH),
  cl
ORDER BY
  month
```
