You are an expert analytical SQL reasoning assistant. When given a data transformation problem, you will reason through it step by step before writing any SQL.

Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table
   - For multiple tables, assume the tables are already joined for now and list all the columns as if they were already joined
     - `table1.X, table1.Y, table1.Z, table2.A, table2.B, table2.C`

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     -> person_id, session_id, SUM(page_time) as session_time
     -> person_id, SUM(session_time) as person_time
     ```
   - When a join depends on a pre-aggregated value, prefix the notation with a bracketed join clause:
     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     -> user_id, SUM(cues_reached) as total_cues_reached, SUM(lesson_progress) as course_progress
     ```

4. **Identify dependencies** - which columns drive grouping, which are derived, which require intermediate steps

5. **Plan CTE layers bottom-up** - name each CTE and describe in one sentence what it produces, using the arrow notation from step 3 as your guide

6. **Write each CTE with justification** - one sentence explaining why this layer exists

7. **Compose the final query**

8. **Verify output shape** - confirm the final output columns match the target shape stated in step 1

Do not skip steps. Do not write SQL before completing the reasoning. If the problem is ambiguous, state your assumptions explicitly before reasoning. Repeat the process from the beginning to refine the pseudocode outline if there are new realizations you arrive at while going down the list. The process must be followed at least once, and ideally a second time for validating the process.

## Question
Build a course container summary view that shows each container's course type, publish status, URL, and deduplicated nested arrays of course versions, associated badges, linked documentation, and assigned clients.

--- AGENT ---

## Step 1: Identify Shapes

**Input tables (columns observable from the query):**
- `stg_courses`: id, container_id, course_type, name, version, published
- `stg_clients_course_containers`: client_id, course_container_id
- `stg_clients`: id, name
- `stg_learning_path_course_containers`: course_container_id, learning_path_id
- `stg_learning_paths`: id, name, published, archived
- `stg_course_resources`: course_id, url
- `stg_course_lessons`: course_id, lesson_id
- `stg_lesson_resources`: lesson_id, url
- `agg_client_documents`: doc_id, name, type, path

**Target shape:**
One row per `(container_id, course_type)` with scalar columns `course_status`, `course_url`, and four deduplicated ARRAY<STRUCT> columns: `course_details`, `badge_details`, `doc_details`, `client_details`.

## Step 2: Identify Sources and Relationships

The join spine starts from `stg_clients_course_containers` (ccc) which links clients to course containers:

- `ccc.client_id = clients.id` — resolves client names
- `ccc.course_container_id = c.container_id` — brings in courses
- `c.container_id = mci.container_id` — filters to the max course ID per container (from the `max_course_ids` CTE), used to scope resource/lesson lookups to the latest course version only
- `ccc.course_container_id = lpcc.course_container_id` (LEFT JOIN) — links to learning paths (badges)
- `lpcc.learning_path_id = lp.id` (LEFT JOIN) — resolves badge/learning path details
- `mci.max_course_id = cr.course_id` (LEFT JOIN) — course-level resources for the latest version
- `mci.max_course_id = cl.course_id` (LEFT JOIN) — course-level lessons for the latest version
- `cl.lesson_id = lr.lesson_id` (LEFT JOIN) — lesson-level resources
- `cd.path = REGEXP_SUBSTR(cr.url, ...) OR cd.path = REGEXP_SUBSTR(lr.url, ...)` (LEFT JOIN) — matches document paths extracted from resource URLs at both course and lesson level

The multi-LEFT-JOIN fan-out (badges, resources via two paths, documents) creates row duplication, which is why ARRAY_AGG is used in the main CTE and then deduplicated via ARRAY subselects with GROUP BY in the final SELECT.

## Step 3: Identify Aggregation Depth

There are two logical aggregation levels:

**Level 0 (pre-filter CTE):**
```
container_id, MAX(id) as max_course_id  [where course_type != "vaas"]
```

**Level 1 (main CTE — `course_container_courses`):**
```
container_id, course_type, MAX(published) as course_status, ARRAY_AGG(course_details), ARRAY_AGG(badge_details), ARRAY_AGG(doc_details), ARRAY_AGG(client_details)
```

**Level 2 (final SELECT — deduplication):**
```
container_id, course_type, course_status, course_url, ARRAY(SELECT AS STRUCT ... GROUP BY ...) for each nested array
```

Arrow notation:
```
container_id, MAX(id) as max_course_id
-> container_id, course_type, MAX(published) as course_status, ARRAY_AGG(...) as course_details, ARRAY_AGG(...) as badge_details, ARRAY_AGG(...) as doc_details, ARRAY_AGG(...) as client_details
-> container_id, course_type, course_status, ARRAY(SELECT DISTINCT ...) as course_details, ARRAY(SELECT DISTINCT ...) as badge_details, ARRAY(SELECT DISTINCT ...) as doc_details, ARRAY(SELECT DISTINCT ...) as client_details
```

## Step 4: Identify Dependencies

- **Grouping drivers:** `container_id` and `course_type` define the grain of the output.
- **Derived scalar:** `course_status` is derived from `MAX(c.published)` — if any course version in the container is published, the container is "published".
- **Derived URL:** `course_url` is deterministic from `container_id`.
- **Pre-aggregation dependency:** `max_course_id` must be computed before the main join, because resource and lesson lookups are scoped to only the latest course version per container. Without this, resources from all versions would be joined in.
- **Deduplication dependency:** The multi-path LEFT JOINs (badges × resources × documents × clients) create a combinatorial fan-out. ARRAY_AGG captures everything, then the final SELECT deduplicates each array independently via `ARRAY(SELECT AS STRUCT ... GROUP BY ...)`.

## Step 5: Plan CTE Layers Bottom-Up

1. **`max_course_ids`** — For each non-VAAS container, find the latest (max) course ID so that resource/lesson joins are scoped to the most recent version only.

2. **`course_container_courses`** — Join the full spine (clients ↔ containers ↔ courses ↔ badges ↔ resources ↔ documents), group by `(container_id, course_type)`, and collect all related entities into ARRAY_AGG columns. This is the heavy aggregation layer.

3. **Final SELECT** — Deduplicate each ARRAY_AGG column by unnesting it with `ARRAY(SELECT AS STRUCT ... GROUP BY ...)`, which eliminates duplicate rows caused by the fan-out joins. Also enriches badge details with a constructed `badge_url`.

## Step 6: Write Each CTE with Justification

**`max_course_ids`:** This CTE pre-computes the latest course ID per container (excluding VAAS courses) so that downstream resource and lesson joins only pull data from the most recent course version — without this, the join would fan out across all historical versions.

**`course_container_courses`:** This is the main aggregation CTE. It joins the full relationship graph starting from the client-container bridge table, through courses, badges (via learning paths), and documents (via both course-level and lesson-level resource URLs matched by path). Grouping by `(container_id, course_type)` and using ARRAY_AGG collects all related entities. The `MAX(c.published)` pattern treats the container as published if any of its course versions is published.

**Final SELECT:** The ARRAY_AGG columns from the CTE contain duplicates from the fan-out joins. Each `ARRAY(SELECT AS STRUCT ... FROM UNNEST(...) GROUP BY ...)` clause deduplicates the array to distinct entities. The badge array is also enriched with a constructed `badge_url`.

## Step 7: Compose the Final Query

```sql
WITH
  max_course_ids AS (
  SELECT
    container_id,
    MAX(id) AS max_course_id
  FROM
    ${ref("stg_courses")}
  WHERE
    course_type != "vaas"
  GROUP BY
    container_id),
  course_container_courses AS (
  SELECT
    c.container_id,
    c.course_type,
    CONCAT("https://app.dataprotocol.com/container/", c.container_id) AS course_url,
    CASE
      WHEN MAX(c.published) = 1 THEN 'published'
      ELSE 'unpublished'
  END
    AS course_status,
    ARRAY_AGG(STRUCT (c.name AS course_name,
        c.id AS course_id,
        c.version AS version) IGNORE NULLS
    ORDER BY
      c.id) AS course_details,
    ARRAY_AGG(STRUCT (lp.id AS badge_id,
        lp.name AS badge_name,
        CASE
          WHEN lp.published = 1 THEN 'published'
          WHEN lp.published = 0 THEN 'unpublished'
          ELSE NULL
      END
        AS badge_status) IGNORE NULLS
    ORDER BY
      lp.id) AS badge_details,
    ARRAY_AGG(STRUCT (cd.doc_id AS doc_id,
        cd.name AS doc_name,
        cd.type AS doc_type,
        CONCAT("https://docs.dataprotocol.com", cd.path) AS doc_url ) IGNORE NULLS
    ORDER BY
      cd.doc_id) AS doc_details,
    ARRAY_AGG(STRUCT (ccc.client_id,
        clients.name AS client_name) IGNORE NULLS) AS client_details
  FROM
    ${ref("stg_clients_course_containers")} ccc
  JOIN
    ${ref("stg_clients")} clients
  ON
    clients.id = ccc.client_id
  JOIN
    ${ref("stg_courses")} c
  ON
    c.container_id = ccc.course_container_id
  LEFT JOIN
    ${ref("stg_learning_path_course_containers")} lpcc
  ON
    lpcc.course_container_id = ccc.course_container_id
  JOIN
    max_course_ids mci
  ON
    mci.container_id = c.container_id
  LEFT JOIN
    ${ref("stg_learning_paths")} lp
  ON
    lp.id = lpcc.learning_path_id
    -- and lp.archived = 0
  LEFT JOIN
    ${ref("stg_course_resources")} cr
  ON
    cr.course_id = mci.max_course_id
  LEFT JOIN
    ${ref("stg_course_lessons")} cl
  ON
    cl.course_id = mci.max_course_id
  LEFT JOIN
    ${ref("stg_lesson_resources")} lr
  ON
    lr.lesson_id = cl.lesson_id
  LEFT JOIN
    ${ref("agg_client_documents")} cd
  ON
    cd.path = REGEXP_SUBSTR(cr.url, r'https?://[^/]+(/.*)', 1, 1)
    OR cd.path = REGEXP_SUBSTR(lr.url, r'https?://[^/]+(/.*)', 1, 1)
  WHERE
    course_type != "vaas"
  GROUP BY
    c.container_id,
    c.course_type)
SELECT
  ccc.container_id,
  ccc.course_type,
  ccc.course_status,
  ccc.course_url,
  ARRAY(
  SELECT
    AS STRUCT cd.course_id, cd.course_name, cd.version
  FROM
    ccc.course_details AS cd
  GROUP BY
    cd.course_id, cd.course_name, cd.version) AS course_details,
  ARRAY(
  SELECT
    AS STRUCT bd.badge_id, bd.badge_name, bd.badge_status, CONCAT("https://app.dataprotocol.com/badges/", bd.badge_id) AS badge_url
  FROM
    ccc.badge_details AS bd
  GROUP BY
    bd.badge_id, bd.badge_name, bd.badge_status) AS badge_details,
  ARRAY(
  SELECT
    AS STRUCT dd.doc_id, dd.doc_name, dd.doc_type, dd.doc_url
  FROM
    ccc.doc_details AS dd
  GROUP BY
    dd.doc_id, dd.doc_name, dd.doc_type, dd.doc_url) AS doc_details,
  ARRAY(
  SELECT
    AS STRUCT cd.client_id, cd.client_name
  FROM
    ccc.client_details AS cd
  GROUP BY
    cd.client_id, cd.client_name) AS client_details
FROM
  course_container_courses AS ccc
```
