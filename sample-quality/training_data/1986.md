## Question
Give me all document badges with their associated client IDs. If a document has no clients, default to client ID 1.

--- AGENT ---

## 1. Identify Shapes

**Input shape:**
- `document_badges`: `id, document_id, learning_path_id, order, created, updated`
- `documents_clients`: `doc_id, client_id`

**Target shape:**
`id, document_id, learning_path_id, order, created, updated, client_ids`

Where `client_ids` is an array of distinct client IDs associated with each document badge's document, defaulting to `[1]` when no clients exist.

## 2. Identify Sources and Relationships

Two tables are involved:
- `document_badges` — the primary entity, one row per badge
- `documents_clients` — a many-to-many mapping between documents and clients

The join relationship is `documents_clients.doc_id = document_badges.document_id`. This is a one-to-many relationship from the badge's document to its clients.

Because a single document can have many client rows in `documents_clients`, joining directly would fan out `document_badges` rows. The client rows must be pre-aggregated into an array per document before joining, so the badge grain is preserved.

The join is a LEFT JOIN because not every document necessarily has client entries — the COALESCE to `[1]` handles that default.

## 3. Identify Aggregation Depth

Single-level aggregation, performed before the join:

```
doc_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

No further aggregation after the join — the final SELECT is at the `document_badges` row grain.

Arrow notation:

```
[pre-aggregate clients per document, then LEFT JOIN to document_badges on doc_id = document_id]
doc_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
-> (join to document_badges, no further aggregation — one row per badge)
```

## 4. Identify Dependencies

- **Grouping column:** `doc_id` in `documents_clients` drives the pre-aggregation
- **Derived column:** `client_ids` — the array of distinct client IDs per document
- **Intermediate step required:** The `documents_clients` table must be collapsed to one row per `doc_id` before joining to `document_badges`, otherwise the join fans out badge rows
- **Default logic:** `COALESCE(dc.client_ids, [1])` applies after the LEFT JOIN to handle documents with no client associations

## 5. Plan CTE Layers Bottom-Up

**CTE `doc_clients`:** Aggregates `documents_clients` to one row per `doc_id`, collecting all distinct non-null `client_id` values into an array.

**Final SELECT:** Joins `document_badges` to `doc_clients` on `document_id = doc_id`, selects all badge columns plus the client array (defaulting to `[1]`).

## 6. Write Each CTE with Justification

**`doc_clients`:** Pre-aggregates the client-to-document mapping so the join to `document_badges` does not multiply rows. `ARRAY_AGG(DISTINCT ... IGNORE NULLS)` ensures each client appears once and null client IDs are excluded.

## 7. Compose the Final Query

```sql
WITH
doc_clients AS (
  SELECT
    dc.doc_id as doc_id,
    ARRAY_AGG(DISTINCT dc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("documents_clients")} as dc
  GROUP BY
    dc.doc_id
)
SELECT
  db.id,
  db.document_id,
  db.learning_path_id,
  db.`order`,
  db.created,
  db.updated,
  COALESCE(dc.client_ids, [1]) AS client_ids
FROM
  ${ref("document_badges")} as db
LEFT JOIN
  doc_clients as dc
ON
  dc.doc_id = db.document_id
```

## 8. Verify Output Shape

Target shape: `id, document_id, learning_path_id, order, created, updated, client_ids`

Final SELECT produces: `db.id, db.document_id, db.learning_path_id, db.order, db.created, db.updated, COALESCE(dc.client_ids, [1]) AS client_ids`

✓ All seven columns present. Row grain is one row per `document_badges` entry, preserved by the pre-aggregation CTE. The LEFT JOIN ensures badges without client mappings still appear, with `client_ids` defaulting to `[1]`.
