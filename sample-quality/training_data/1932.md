--- USER ---

Give me each user's learning path completion status by combining two sources: current course-based progress (where a learning path is complete when all its courses are completed) and legacy badge records. When both sources have an entry for the same user and learning path, keep the earliest completion date and its associated details.

--- AGENT ---

## 1. Identify Shapes

**Input tables:**
- `stg_users`: `id, test_user`
- `stg_user_courses`: `user_id, course_id, status, updated`
- `stg_courses`: `id, container_id`
- `stg_learning_path_course_containers`: `id, learning_path_id, course_container_id`
- `stg_learning_paths`: `id, name`
- `stg_user_badges`: `user_id, learning_path_id, name, created`

**Target shape:**
```
user_id, learning_path_id, learning_path_name, completed_date, did_complete, legacy_badge
```
One row per user per learning path, reflecting the earliest completion record across both current and legacy sources.

## 2. Identify Sources and Relationships

There are two independent data paths that produce the same output shape, then get unioned:

**Branch 1 ŌĆö Current course-based completions:**
- `stg_users.id` ŌåÆ `stg_user_courses.user_id` (direct)
- `stg_user_courses.course_id` ŌåÆ `stg_courses.id` (direct) ŌĆö this gives us `container_id`
- `stg_courses.container_id` ŌåÆ `stg_learning_path_course_containers.course_container_id` (direct) ŌĆö this maps containers to learning paths
- `stg_learning_path_course_containers.learning_path_id` ŌåÆ `stg_learning_paths.id` (direct)
- A separate subquery counts the total number of course containers per learning path (from `stg_learning_path_course_containers` joined to `stg_learning_paths`) ŌĆö this is needed to determine whether a user has completed *all* courses in a path

Key insight: courses are grouped by `container_id` before joining to learning paths. Multiple courses can share a `container_id`, and the subquery `ucc` picks one status per user per container (using `MIN(status)` ŌĆö alphabetically "completed" < other statuses, so this effectively checks if the best status is completed). The `MIN_BY(updated, status)` picks the `updated` timestamp corresponding to that minimum status.

**Branch 2 ŌĆö Legacy badge entries:**
- `stg_users.id` ŌåÆ `stg_user_badges.user_id` (direct)
- `stg_user_badges` already contains `learning_path_id` and `name` directly ŌĆö no chain needed

**Outer deduplication:**
- The two branches are UNIONed and grouped by `user_id, learning_path_id`
- `MIN(completed_date)` picks the earliest record
- `MIN_BY(col, completed_date)` picks associated columns from that earliest record

## 3. Identify Aggregation Depth

This query has three levels of aggregation, plus the union deduplication:

**Branch 1 (current courses):**
```
user_id, container_id, MIN(status) as status, MIN_BY(updated, status) as updated
-> user_id, learning_path_id, learning_path_name, MAX(updated) as completed_date, SUM(CASE completed) = MAX(num_courses) as did_complete
```

**Branch 2 (legacy badges):**
```
user_id, learning_path_id, learning_path_name, MAX(created) as completed_date, true as did_complete, true as legacy_badge
```

**Outer (deduplication across both branches):**
```
[UNION ALL of Branch 1 and Branch 2]
user_id, learning_path_id, MIN(completed_date) as completed_date, MIN_BY(learning_path_name, completed_date), MIN_BY(did_complete, completed_date), MIN_BY(legacy_badge, completed_date)
```

Also a reference subquery for course counts:
```
learning_path_id, COUNT(course_container_id) as num_courses
```

## 4. Identify Dependencies

- **Grouping drivers:** `user_id` and `learning_path_id` are the final grain. Everything else is derived.
- **`container_id`** is an intermediate grouping key ŌĆö courses must be collapsed to containers before joining to learning paths, because learning paths reference containers, not individual courses.
- **`num_courses`** is a reference dimension ŌĆö it must be computed independently per learning path and joined in so the `did_complete` boolean can compare the count of completed containers against the expected total.
- **`did_complete`** in Branch 1 depends on both the per-container aggregation (to get one status per container) and the `num_courses` reference value.
- **The outer `MIN_BY` pattern** depends on both branches producing the same column shape with `completed_date` as the ordering column.

## 5. Plan CTE Layers Bottom-Up

This query uses inline subqueries rather than named CTEs, but the logical layers are:

1. **`ucc` (inline subquery)** ŌĆö Collapse `user_courses` joined to `courses` down to one row per `(user_id, container_id)`, picking the best status and its associated timestamp.
2. **`lp_num_courses` (inline subquery)** ŌĆö Count the total number of course containers per learning path, used as a denominator for the completion check.
3. **Branch 1 (inline, first half of UNION ALL)** ŌĆö Join users ŌåÆ ucc ŌåÆ learning_path_course_containers ŌåÆ learning_paths ŌåÆ lp_num_courses, group by user/learning_path, compute `did_complete` and `completed_date`. Mark `legacy_badge = false`.
4. **Branch 2 (inline, second half of UNION ALL)** ŌĆö Join users ŌåÆ user_badges, group by user/learning_path, set `did_complete = true` and `legacy_badge = true`.
5. **Outer query** ŌĆö Group the union by `(user_id, learning_path_id)`, pick the earliest `completed_date` and carry associated columns via `MIN_BY`.

## 6. Write Each CTE With Justification

**`ucc` subquery:** Courses can share a `container_id`, and learning paths reference containers ŌĆö so we must collapse courses to the container grain before joining upward. `MIN(status)` picks the best status alphabetically (where "completed" sorts first), and `MIN_BY(updated, status)` gets the timestamp associated with that status.

**`lp_num_courses` subquery:** The `did_complete` flag requires comparing how many containers a user has completed against how many containers the learning path expects. This reference subquery provides the denominator ŌĆö the total container count per learning path.

**Branch 1:** Assembles the full chain from users through containers to learning paths, checks completion by comparing `SUM(CASE completed)` to `num_courses`, and uses `MAX(ucc.updated)` as the completion timestamp (the latest container update represents when the path was fully addressed).

**Branch 2:** Legacy badge records already represent completed learning paths. They are grouped by user and learning path with `MAX(created)` as the timestamp, and hardcoded `did_complete = true`, `legacy_badge = true`.

**Outer query:** A single user may appear in both branches for the same learning path. The outer GROUP BY deduplicates, keeping the earliest `completed_date` via `MIN()` and pulling associated fields from that earliest record via `MIN_BY()`.

## 7. Compose the Final Query

```sql
SELECT
  both.user_id,
  both.learning_path_id,
  MIN_BY(both.learning_path_name, both.completed_date) AS learning_path_name,
  MIN(both.completed_date) AS completed_date,
  MIN_BY(both.did_complete, both.completed_date) AS did_complete,
  MIN_BY(both.legacy_badge, both.completed_date) AS legacy_badge
FROM
  (
    -- Badge completions by current courses
    SELECT
      u.id                                AS user_id,
      lp.id                               AS learning_path_id,
      lp.name                             AS learning_path_name,
      max(ucc.updated)                    AS completed_date, -- This is assuming that the user_courses entry will be updated only to change the status to 'completed'
      sum(
        CASE WHEN ucc.status = 'completed' THEN 1 ELSE 0 END
      ) = max(lp_num_courses.num_courses) AS did_complete,
      false AS legacy_badge
    FROM
      ${ref("stg_users")} AS u
    INNER JOIN
      (
        SELECT
          _uc.user_id,
          min(_uc.status)                 AS status,
          min_by(_uc.updated, _uc.status) AS updated,
          _c.container_id
        FROM
          ${ref("stg_user_courses")} AS _uc
        INNER JOIN
          ${ref("stg_courses")} AS _c
          ON
            _uc.course_id = _c.id
        GROUP BY
          _uc.user_id,
          _c.container_id
      ) AS ucc
      ON
        u.id = ucc.user_id
    INNER JOIN
      ${ref("stg_learning_path_course_containers")} AS lpcc
      ON
        ucc.container_id = lpcc.course_container_id
    INNER JOIN
      ${ref("stg_learning_paths")} AS lp
      ON
        lpcc.learning_path_id = lp.id
    INNER JOIN
      (
        SELECT
          _lp.id,
          count(_lpcc.id) AS num_courses
        FROM
          ${ref("stg_learning_path_course_containers")} AS _lpcc
        INNER JOIN
          ${ref("stg_learning_paths")} AS _lp
          ON
            _lpcc.learning_path_id = _lp.id
        GROUP BY
          _lp.id
      ) AS lp_num_courses
      ON
        lp.id = lp_num_courses.id
    WHERE
      u.test_user = 0
    GROUP BY
      u.id,
      lp.id,
      lp.name

    UNION ALL

    -- Existing badge entries
    SELECT
      u.id                                AS user_id,
      ub.learning_path_id                 AS learning_path_id,
      ub.name                             AS learning_path_name,
      max(ub.created)                     AS completed_date,
      true AS did_complete,
      true AS legacy_badge
    FROM
      ${ref("stg_users")} AS u
    JOIN
      ${ref("stg_user_badges")} as ub
    ON
      u.id = ub.user_id
    WHERE
      u.test_user = 0
    GROUP BY
      u.id,
      ub.learning_path_id,
      ub.name
  ) as both
GROUP BY
  both.user_id,
  both.learning_path_id

```
