--- USER ---

Give me a list of all documents with their metadata, and for each document include an array of all associated client IDs. If a document has no client associations, default to client ID 1.

--- AGENT ---

## 1. Identify shapes

**Input shape:**
- `documents`: `id, name, path, type, description, image, created, updated, minutes_to_read, hidden, category_id, client_id`
- `documents_clients`: `doc_id, client_id`

**Target shape:**
- `id, name, path, type, description, image, created, updated, minutes_to_read, hidden, category_id, client_id, client_ids`

The output is one row per document, with all document metadata columns preserved and a new `client_ids` array column containing the distinct client IDs associated with each document.

## 2. Identify sources and relationships

Two tables are involved:

- `documents` — the primary entity table, one row per document
- `documents_clients` — a many-to-many junction table linking documents to clients via `doc_id` and `client_id`

The join relationship is `documents.id = documents_clients.doc_id`.

The join is a LEFT JOIN because not every document necessarily has entries in `documents_clients`, and we want to preserve all documents in the output. Documents with no client associations receive a default `client_ids` of `[1]` via `COALESCE`.

Note: `documents` has its own `client_id` scalar column, which is a separate concept from the `client_ids` array being constructed from the junction table. Both are preserved in the output.

## 3. Identify aggregation depth

There is one level of aggregation — collapsing the many-to-many `documents_clients` rows into a single array per document before joining back:

```
doc_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

This is a single-level aggregation. No further rollup is needed. The arrow notation is:

```
doc_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

The aggregation is pre-computed in a CTE so that the join to `documents` is one-to-one rather than one-to-many, which avoids row duplication in the final output.

## 4. Identify dependencies

- **Grouping driver:** `doc_id` in the CTE — this is the foreign key that links back to `documents.id`
- **Derived column:** `client_ids` — an array aggregation of distinct non-null `client_id` values per document
- **Default logic:** `COALESCE(dc.client_ids, [1])` — when the LEFT JOIN produces NULL (no matching rows in `documents_clients`), the default array `[1]` is substituted
- No intermediate steps are required beyond the single pre-aggregation CTE

## 5. Plan CTE layers bottom-up

1. **`doc_clients`** — Pre-aggregates `documents_clients` into one row per `doc_id` with an array of distinct client IDs, so the subsequent join to `documents` is one-to-one.

## 6. Write each CTE with justification

**`doc_clients`:** This CTE exists to collapse the many-to-many `documents_clients` junction table into one row per document before joining, preventing row duplication in the final SELECT that would occur from a direct join to the raw junction table.

```sql
doc_clients AS (
  SELECT
    dc.doc_id as doc_id,
    ARRAY_AGG(DISTINCT dc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("documents_clients")} as dc
  GROUP BY
    dc.doc_id
)
```

`DISTINCT` ensures no duplicate client IDs in the array. `IGNORE NULLS` prevents NULL values from appearing in the array output (a BigQuery-specific ARRAY_AGG modifier).

## 7. Compose the final query

```sql
WITH
doc_clients AS (
  SELECT
    dc.doc_id as doc_id,
    ARRAY_AGG(DISTINCT dc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("documents_clients")} as dc
  GROUP BY
    dc.doc_id
)
SELECT
  d.id,
  d.name,
  d.path,
  d.type,
  d.description,
  d.image,
  d.created,
  d.updated,
  d.minutes_to_read,
  d.hidden,
  d.category_id,
  d.client_id,
  COALESCE(dc.client_ids, [1]) AS client_ids
FROM
  ${ref("documents")} as d
LEFT JOIN
  doc_clients as dc
ON
  dc.doc_id = d.id
```

## 8. Verify output shape

Target shape: `id, name, path, type, description, image, created, updated, minutes_to_read, hidden, category_id, client_id, client_ids`

Final SELECT produces: `d.id, d.name, d.path, d.type, d.description, d.image, d.created, d.updated, d.minutes_to_read, d.hidden, d.category_id, d.client_id, client_ids` — **13 columns, one row per document.** ✓

The LEFT JOIN ensures all documents appear even without junction table entries. The COALESCE guarantees `client_ids` is never NULL — unmatched documents receive `[1]`. The pre-aggregation in the CTE guarantees no row duplication from the join. Output shape matches target.
