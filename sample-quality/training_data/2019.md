--- USER ---

For each row in the `user_course_lab_servers` table, produce the full record enriched with a `client_ids` array indicating which client IDs the row's user belongs to. Users who have an external client association should receive an array containing both the default client (1) and their external client ID; users with no external client association should default to the array `[1]` (default client only).

**Step 1: Identify shapes**

Input tables (as if already joined):

- `users.id` (aliased as `user_id`), `users.external_client_id`
- `user_course_lab_servers.id`, `user_course_lab_servers.user_id`, `user_course_lab_servers.course_id`, `user_course_lab_servers.lab_id`, `user_course_lab_servers.server_id`, `user_course_lab_servers.lab_instance_id`, `user_course_lab_servers.created`, `user_course_lab_servers.updated`

Target shape (one row per row in `user_course_lab_servers`):

`id, user_id, course_id, lab_id, server_id, lab_instance_id, created, updated, client_ids`

where `client_ids` is an array — either `[1, external_client_id]` or the fallback `[1]`.

---

**Step 2: Identify sources and relationships**

The driving table is `user_course_lab_servers` (aliased `_utb`). The enrichment source is `users`, filtered to only those rows where `external_client_id IS NOT NULL`, and pre-shaped into a `user_clients` CTE.

The join relationship is `ucl.user_id = _utb.user_id` — a natural foreign key link between `user_course_lab_servers.user_id` and the `user_id` derived from `users.id` in the CTE.

The join is a LEFT JOIN, which is the correct strategy here because not every user in `user_course_lab_servers` will have an `external_client_id`. A INNER JOIN would silently drop rows for users with no external client — the LEFT JOIN preserves all lab server rows regardless.

---

**Step 3: Identify aggregation depth**

There is no GROUP BY aggregation in this query. The CTE performs a row-level scalar transformation (array construction), and the final query is a pure enrichment join. The full transformation is a single flat projection.

Arrow notation (single level, no aggregation arrow needed):

```
_utb.id, _utb.user_id, _utb.course_id, _utb.lab_id, _utb.server_id,
_utb.lab_instance_id, _utb.created, _utb.updated,
COALESCE(ucl.client_ids, [1]) as client_ids
```

No `->` levels exist because there is no multi-level aggregation — the CTE is a pre-filter and scalar reshape, not an aggregation step.

**Step 4: Identify dependencies**

- `client_ids` depends on the presence of `external_client_id` in the `users` table. Users without it are excluded from the CTE (`WHERE external_client_id IS NOT NULL`), so they produce no match in the LEFT JOIN and `ucl.client_ids` is `NULL`.
- `COALESCE(ucl.client_ids, [1])` handles the NULL case: if no external client exists, the row falls back to the default client array `[1]`.
- The array literal `[1, u.external_client_id]` uses square-bracket syntax, confirming this is BigQuery SQL. The integer `1` is the default/base client ID, hardcoded as a constant — its presence in both the array and the COALESCE fallback is deliberate: every user, regardless of external association, belongs to client 1.
- The CTE must be pre-computed before the join because the array construction (`[1, u.external_client_id]`) needs to happen at the user grain, not at the lab server grain — it would be incorrect to construct the array after the join (it would be identical every time, but semantically the separation is correct).

---

**Step 5: Plan CTE layers bottom-up**

Only one CTE is required:

1. **`user_clients`** — Filters `users` to rows with a non-null `external_client_id` and constructs a two-element BigQuery array `[1, external_client_id]` as `client_ids`, keyed by `user_id`. This pre-shapes the client membership data at the user grain so the main query can enrich each lab server row with a single LEFT JOIN lookup.

---

**Step 6: Write each CTE with justification**

**`user_clients`**: This CTE exists because the client ID array must be built at the `user_id` grain — filtering out users with no external client and constructing `[1, external_client_id]` — so that the main query can perform a clean scalar enrichment join rather than embedding conditional array logic inline on a potentially large join result.

---

**Step 7: Compose the final query**

```sql
WITH
user_clients AS (
  SELECT
    u.id as user_id,
    [1, u.external_client_id] as client_ids
  FROM
    ${ref("users")} as u
  WHERE
    u.external_client_id IS NOT NULL
)
SELECT
  _utb.id,
  _utb.user_id,
  _utb.course_id,
  _utb.lab_id,
  _utb.server_id,
  _utb.lab_instance_id,
  _utb.created,
  _utb.updated,
  COALESCE(ucl.client_ids, [1]) as client_ids
FROM
  ${ref("user_course_lab_servers")} as _utb
LEFT JOIN
  user_clients as ucl
ON
  ucl.user_id = _utb.user_id
```
