--- SYSTEM ---

You are an expert analytical SQL reasoning assistant. When given a data transformation problem, you will reason through it step by step before writing any SQL.

Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table
   - For multiple tables, assume the tables are already joined for now and list all the columns as if they were already joined
     - `table1.X, table1.Y, table1.Z, table2.A, table2.B, table2.C`

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     -> person_id, session_id, SUM(page_time) as session_time
     -> person_id, SUM(session_time) as person_time
     ```
   - When a join depends on a pre-aggregated value, prefix the notation with a bracketed join clause:
     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     -> user_id, SUM(cues_reached) as total_cues_reached, SUM(lesson_progress) as course_progress
     ```

4. **Identify dependencies** - which columns drive grouping, which are derived, which require intermediate steps

5. **Plan CTE layers bottom-up** - name each CTE and describe in one sentence what it produces, using the arrow notation from step 3 as your guide

6. **Write each CTE with justification** - one sentence explaining why this layer exists

7. **Compose the final query**

8. **Verify output shape** - confirm the final output columns match the target shape stated in step 1

Do not skip steps. Do not write SQL before completing the reasoning. If the problem is ambiguous, state your assumptions explicitly before reasoning. Repeat the process from the beginning to refine the pseudocode outline if there are new realizations you arrive at while going down the list. The process must be followed at least once, and ideally a second time for validating the process.

--- USER ---

Generate a quarterly engagement report per client showing unique users, new vs returning user counts, users who appear in both categories, quarterly-scoped new/returning breakdowns, total sessions, returning user sessions, and average sessions per user. The report should be based on client transaction data where each transaction may be associated with multiple clients.

--- AGENT ---

## 1. Identify Shapes

**Input shape:**
- `agg_client_transactions`: `user_id`, `client_id` (repeated/array field), `day`

**Target output shape:**
- `unique_users` — distinct users active in the quarter for a given client
- `new_users` — distinct users whose first-ever transaction day matches the row's day (i.e. their first appearance globally for that client)
- `returning_users` — distinct users whose first-ever day does NOT match the row's day
- `new_users_returning` — users who appear as both new and returning within the same quarter (the overlap between the two sets above)
- `quarterly_new_users` — distinct users whose first-ever day falls within the same quarter
- `quarterly_returning_users` — distinct users whose first-ever day falls outside the current quarter
- `total_sessions` — total row count (one per user per day per client)
- `returning_user_sessions` — session count restricted to returning users
- `sessions_per_user` — total_sessions / unique_users, rounded to 2 decimal places
- `quarter_start` — the first day of the calendar quarter
- `client_id`

Grouped by: `quarter_start, client_id`

## 2. Identify Sources and Relationships

There is a single source table `agg_client_transactions` used twice — once to compute each user's first-ever transaction day per client, and once to enumerate every transaction day per quarter per client.

The `client_id` column is an array (REPEATED field in BigQuery), so it must be unnested with `UNNEST(ct.client_id)` to produce one row per user × day × client.

The join relationship between the two uses of this table is:
- `user_id` — the same user
- `client_id` — the same expanded client

This is a self-join pattern: the first use computes the user-level minimum day per client, the second use fans out every transaction into quarterly periods. They join on `(user_id, client_id)`.

## 3. Identify Aggregation Depth

There are two levels:

**Level 1 — Pre-aggregation CTEs (two parallel branches from the same table):**

Branch A — first-ever day per user per client:
```
user_id, client_id, MIN(day) as min_day
```

Branch B — transaction detail with quarter derived:
```
user_id, client_id, day, DATE(YEAR, 1 + (QUARTER-1)*3, 1) as quarter_start
```
(No aggregation — this is a row-level transformation.)

**Level 2 — Final aggregation after joining Branch A to Branch B:**
```
quarter_start, client_id,
  COUNT(DISTINCT user_id) as unique_users,
  COUNT(DISTINCT CASE WHEN min_day = day THEN user_id END) as new_users,
  COUNT(DISTINCT CASE WHEN min_day != day THEN user_id END) as returning_users,
  (new_users + returning_users) - unique_users as new_users_returning,
  COUNT(DISTINCT CASE WHEN min_day within quarter THEN user_id END) as quarterly_new_users,
  COUNT(DISTINCT CASE WHEN min_day outside quarter THEN user_id END) as quarterly_returning_users,
  COUNT(user_id) as total_sessions,
  COUNT(CASE WHEN min_day != day THEN user_id END) as returning_user_sessions,
  ROUND(total_sessions / unique_users, 2) as sessions_per_user
```

Arrow notation:
```
user_id, client_id, MIN(day) as min_day
-> [joined to quarterly_periods on user_id, client_id]
   quarter_start, client_id,
   COUNT(DISTINCT user_id) as unique_users,
   COUNT(DISTINCT CASE new) as new_users,
   COUNT(DISTINCT CASE returning) as returning_users,
   (new_users + returning_users - unique_users) as new_users_returning,
   COUNT(DISTINCT CASE quarterly_new) as quarterly_new_users,
   COUNT(DISTINCT CASE quarterly_returning) as quarterly_returning_users,
   COUNT(user_id) as total_sessions,
   COUNT(CASE returning) as returning_user_sessions,
   ROUND(total_sessions / unique_users, 2) as sessions_per_user
```

## 4. Identify Dependencies

- **Grouping drivers:** `quarter_start` and `client_id` define the output grain.
- **Derived — `min_day`:** Must be pre-computed per `(user_id, client_id)` before the final aggregation. This is the critical dependency — every conditional COUNT in the final SELECT references `min_day` to classify users as new or returning.
- **Derived — `quarter_start`:** Computed from `day` as a row-level expression in the `quarterly_periods` CTE. No aggregation needed, just a date transformation.
- **Derived — `new_users_returning`:** This is an arithmetic expression over two other aggregates: `(returning_users + new_users) - unique_users`. It captures users who transacted on their first-ever day AND on at least one other day within the same quarter — they appear in both the new and returning DISTINCT counts, so their sum exceeds the total unique count by the overlap size.
- **`meta_quest_min` CTE:** This is a pass-through of `client_transactions_min` with no transformation. It serves as a naming alias (possibly a vestige of a more complex pipeline where additional filters were applied).

## 5. Plan CTE Layers Bottom-Up

1. **`client_transactions_min`** — Computes the earliest transaction day (`min_day`) per `(user_id, client_id)` after unnesting the client_id array. This is the first-ever appearance of each user for each client.

2. **`meta_quest_min`** — Pass-through alias of `client_transactions_min`. Selects `user_id, client_id, min_day` unchanged. (Likely a legacy naming convention or placeholder for a filter that was removed.)

3. **`quarterly_periods`** — Unnests the client_id array and derives `quarter_start` from each transaction's `day` column. Produces one row per `(user_id, client_id, day)` with the quarter boundary attached. No aggregation.

4. **Final SELECT** — Joins `quarterly_periods` to `meta_quest_min` on `(user_id, client_id)`, then groups by `(quarter_start, client_id)` to produce all output metrics using conditional DISTINCT counts that compare each row's `day` against the user's `min_day`.

## 6. Write Each CTE with Justification

**`client_transactions_min`** — Exists because the new/returning classification requires knowing each user's first-ever transaction day per client, which must be computed once across all time before being joined back to per-transaction rows.

```sql
SELECT
  ct.user_id,
  client_id_expanded AS client_id,
  MIN(ct.`day`) AS min_day
FROM ${ref("agg_client_transactions")} AS ct,
  UNNEST(ct.client_id) AS client_id_expanded
GROUP BY ct.user_id, client_id_expanded
```

**`meta_quest_min`** — Exists as a naming indirection over `client_transactions_min`; structurally a no-op passthrough that may have historically carried additional logic or filters.

```sql
SELECT ctm.user_id, ctm.client_id, ctm.min_day
FROM client_transactions_min AS ctm
```

**`quarterly_periods`** — Exists to attach the derived `quarter_start` date to every transaction row after unnesting the client array, so the final aggregation can group by calendar quarter without recomputing the date expression inline.

```sql
SELECT
  ct.user_id,
  client_id_expanded AS client_id,
  ct.`day`,
  DATE(EXTRACT(YEAR FROM `day`), 1 + (EXTRACT(QUARTER FROM `day`) - 1) * 3, 1) AS quarter_start
FROM ${ref("agg_client_transactions")} AS ct,
  UNNEST(ct.client_id) AS client_id_expanded
```

## 7. Compose the Final Query

```sql
WITH
  client_transactions_min AS (
  SELECT
    ct.user_id,
    client_id_expanded AS client_id,
    MIN(ct.`day`) AS min_day
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS client_id_expanded
  GROUP BY
    ct.user_id,
    client_id_expanded ),
  meta_quest_min AS (
  SELECT
    ctm.user_id,
    ctm.client_id,
    ctm.min_day
  FROM
    client_transactions_min AS ctm ),
  quarterly_periods AS (
  SELECT
    ct.user_id,
    client_id_expanded AS client_id,
    ct.`day`,
    DATE(EXTRACT(YEAR
      FROM
        `day`), 1 + (EXTRACT(QUARTER
        FROM
          `day`) - 1) * 3, 1) AS quarter_start
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS client_id_expanded )
SELECT
  COUNT(DISTINCT ct.user_id) AS unique_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day = ct.`day` THEN ct.user_id
  END
    ) AS new_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day != ct.`day` THEN ct.user_id
  END
    ) AS returning_users,
  ( COUNT( DISTINCT
      CASE
        WHEN mqm.min_day != ct.`day` THEN ct.user_id
    END
      ) + COUNT( DISTINCT
      CASE
        WHEN mqm.min_day = ct.`day` THEN ct.user_id
    END
      ) ) - COUNT(DISTINCT ct.user_id) AS new_users_returning,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day >= quarter_start AND mqm.min_day < DATE_ADD(quarter_start, INTERVAL 3 MONTH) THEN ct.user_id
  END
    ) AS quarterly_new_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day < quarter_start OR mqm.min_day >= DATE_ADD(quarter_start, INTERVAL 3 MONTH) THEN ct.user_id
  END
    ) AS quarterly_returning_users,
  COUNT(ct.user_id) AS total_sessions,
  COUNT(
    CASE
      WHEN mqm.min_day != ct.`day` THEN ct.user_id
  END
    ) AS returning_user_sessions,
  ROUND(COUNT(ct.user_id) / COUNT(DISTINCT ct.user_id), 2) AS sessions_per_user,
  quarter_start,
  ct.client_id AS client_id
FROM
  quarterly_periods AS ct
INNER JOIN
  meta_quest_min AS mqm
ON
  mqm.user_id = ct.user_id
  AND mqm.client_id = ct.client_id
GROUP BY
  quarter_start,
  ct.client_id
ORDER BY
  quarter_start
```
