   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:

     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)

--- USER ---

Give me a daily breakdown per client of total users, new users, returning users, total sessions, and returning user sessions. A user is "new" on a given day if that day is their first-ever transaction day for that client, and "returning" otherwise.

--- AGENT ---

## 1. Identify Shapes

**Input shape:**
`agg_client_transactions` has at minimum: `user_id`, `day`, `client_id` (repeated/array field — each row can belong to multiple clients).

**Target shape:**
```
date, client_id, total_users, new_users, returning_users, total_sessions, returning_user_sessions
```

One row per day per client, with user counts (distinct) and session counts broken out by new vs returning status.

## 2. Identify Sources and Relationships

There is only one source table — `agg_client_transactions` — but `client_id` is an array column that must be unnested. This means each row fans out to one row per client membership.

The CTE (`client_transactions_min`) and the main query both read from the same table and both unnest `client_id`. They join on `(user_id, client_id)`, connecting each transaction row to that user's earliest transaction day *for that specific client*.

Join relationship:
- `ct.user_id = ctm.user_id`
- `cl = ctm.client_id` (where `cl` is the unnested client_id in the main query, and `ctm.client_id` is the unnested client_id in the CTE)

This is a many-to-one join: many transaction rows per user-client pair map to one min-day row per user-client pair.

## 3. Identify Aggregation Depth

Two levels of aggregation are required:

**Level 1 (CTE):** Find the earliest day per user per client — this is needed before the main query can classify rows as new vs returning.

**Level 2 (Main query):** Aggregate daily metrics per client using the new/returning classification.

Arrow notation:

```
[pre-aggregation: user's first day per client needed to classify new vs returning]
user_id, client_id, MIN(day) as min_day
-> date, client_id, COUNT(DISTINCT user_id) as total_users,
   COUNT(DISTINCT CASE WHEN min_day = day THEN user_id END) as new_users,
   COUNT(DISTINCT CASE WHEN min_day != day THEN user_id END) as returning_users,
   COUNT(user_id) as total_sessions,
   COUNT(CASE WHEN min_day != day THEN user_id END) as returning_user_sessions
```

## 4. Identify Dependencies

- **Grouping drivers:** `date` (truncated day) and `client_id` drive the final output grain.
- **Derived columns:** `new_users` and `returning_users` depend on comparing each row's `day` to `min_day` — which requires the CTE to have already computed `min_day` per `(user_id, client_id)`.
- **Intermediate step:** The CTE must complete before the main query can join against it. The unnesting of `client_id` must happen independently in both the CTE and the main query so the join key `client_id` is a scalar in both contexts.

## 5. Plan CTE Layers Bottom-Up

1. **`client_transactions_min`** — Produces the first-ever transaction day (`min_day`) for each `(user_id, client_id)` pair by unnesting the client_id array and grouping.

2. **Final SELECT** — Joins unnested transactions back to `client_transactions_min` on `(user_id, client_id)`, then groups by `(date, client_id)` to produce daily per-client metrics with new/returning classification based on whether the row's day matches `min_day`.

## 6. Write Each CTE with Justification

**`client_transactions_min`:** This CTE exists because the new/returning classification requires knowing each user's first transaction day per client *before* the main aggregation can evaluate it row by row. Without pre-aggregating `MIN(day)`, there is no scalar to compare against in the main query's CASE expressions.

## 7. Compose the Final Query

```sql
WITH
  client_transactions_min AS (
  SELECT
    ct.user_id,
    cid AS client_id,
    MIN(ct.`day`) AS min_day
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS cid
  GROUP BY
    ct.user_id,
    cid )
SELECT
  COUNT(DISTINCT ct.user_id) AS total_users,
  COUNT( DISTINCT
    CASE
      WHEN DATE_TRUNC(ctm.min_day, DAY) = DATE_TRUNC(ct.`day`, DAY) THEN ct.user_id
  END
    ) AS new_users,
  COUNT( DISTINCT
    CASE
      WHEN DATE_TRUNC(ctm.min_day, DAY) != DATE_TRUNC(ct.`day`, DAY) THEN ct.user_id
  END
    ) AS returning_users,
  COUNT(ct.user_id) AS total_sessions,
  COUNT(
    CASE
      WHEN DATE_TRUNC(ctm.min_day, DAY) != DATE_TRUNC(ct.`day`, DAY) THEN ct.user_id
  END
    ) AS returning_user_sessions,
  DATE_TRUNC(ct.`day`, DAY) AS date,
  cl AS client_id
FROM
  ${ref("agg_client_transactions")} AS ct
CROSS JOIN
  UNNEST(ct.client_id) AS cl
INNER JOIN
  client_transactions_min AS ctm
ON
  ct.user_id = ctm.user_id
  AND ctm.client_id = cl
GROUP BY
  DATE_TRUNC(ct.`day`, DAY),
  cl
ORDER BY
  date
```

## 8. Verify Output Shape

Target shape: `date, client_id, total_users, new_users, returning_users, total_sessions, returning_user_sessions`

Final SELECT produces:
- `total_users` — `COUNT(DISTINCT user_id)` ✓
- `new_users` — `COUNT(DISTINCT CASE ... user_id)` where day = min_day ✓
- `returning_users` — `COUNT(DISTINCT CASE ... user_id)` where day ≠ min_day ✓
- `total_sessions` — `COUNT(user_id)` (non-distinct, counts all rows) ✓
- `returning_user_sessions` — `COUNT(CASE ... user_id)` where day ≠ min_day (non-distinct) ✓
- `date` — `DATE_TRUNC(ct.day, DAY)` ✓
- `client_id` — `cl` (unnested scalar) ✓

Grain: one row per `(date, client_id)`. Output shape matches target. ✓

**Notable dialect detail:** `DATE_TRUNC(ctm.min_day, DAY)` is applied to both sides of the comparison even though `min_day` is already the result of `MIN(day)`. This is defensive — it normalizes any sub-day precision in the `day` column (if it's a TIMESTAMP rather than a DATE) so the comparison is strictly date-level. The `CROSS JOIN UNNEST` syntax with a comma-style implicit cross join in the CTE vs explicit `CROSS JOIN` in the main query are both valid BigQuery patterns for array unnesting.
