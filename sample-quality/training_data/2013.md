--- USER ---

For each sentiment question, produce a record containing its core metadata and an array of client IDs that have access to it through their course container hierarchy. If a sentiment question's associated learning cue does not appear in any client's course, default the client array to `[1]` (the global/default client).

--- AGENT ---

**Step 1: Identify shapes**

Input tables, treated as if already joined through the CTE chain:

`ccc.client_id, ccc.course_container_id, c.id, c.container_id, cl.course_id, cl.lesson_id, l.id, lc.id, lc.lesson_id`

Plus the driving table for the final output:

`sq.id, sq.name, sq.learning_cue_id, sq.created, sq.updated`

Target shape (one row per sentiment question):

`id, name, learning_cue_id, created, updated, client_ids`

where `client_ids` is an array of integers.

---

**Step 2: Identify sources and relationships**

The relationship between `clients_course_containers` and `learning_cues` is entirely indirect — no single table contains both `client_id` and `learning_cue_id`. The chain must be resolved through a sequence of glue tables:

- `clients_course_containers.course_container_id` → `courses.container_id` (client owns a container, a container holds courses)
- `courses.id` → `course_lessons.course_id` (a course has many lessons via the junction table)
- `course_lessons.lesson_id` → `lessons.id` (resolve lesson identity)
- `lessons.id` → `learning_cues.lesson_id` (a lesson has many learning cues)

This 5-hop chain means a single `client_id` can fan out to many `learning_cue_id` rows before aggregation, which is why `ARRAY_AGG(DISTINCT ... IGNORE NULLS)` is required — the joins can produce duplicate `client_id` values per `learning_cue_id` (e.g., a client has multiple courses under the same container, or a course has multiple lessons each with the same cues).

The final join connects `sentiment_questions` to the pre-aggregated CTE via `learning_cue_id`:

- `sentiment_questions.learning_cue_id` → `stg_sentiment_question.learning_cue_id`

This is a LEFT JOIN because `sentiment_questions` is the authoritative list — questions must appear even if their learning cue is not reachable from any client's course hierarchy. The `COALESCE(..., [1])` handles the NULL case, falling back to an array containing only client ID `1`, which structurally appears to be a reserved global or default client.

---

**Step 3: Identify aggregation depth**

Single level of aggregation in the CTE, followed by a non-aggregating join in the final query:

```
learning_cue_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

The final SELECT does not aggregate — it is a one-to-one enrichment of `sentiment_questions` rows with the pre-computed `client_ids` array.

---

**Step 4: Identify dependencies**

- `client_ids` per `learning_cue_id` cannot be computed in the final query directly because the source relationship requires traversing five tables before the client-to-cue mapping is known.
- The `ARRAY_AGG(DISTINCT ...)` must happen at the `learning_cue_id` level before joining to `sentiment_questions`, or the fan-out from the chain join would cause row duplication and incorrect results in the final output.
- `IGNORE NULLS` is defensive — any NULLs introduced by the join chain are silently excluded from the array rather than surfacing as NULL elements.
- `COALESCE(ssq.client_ids, [1])` is a post-join default, applied after the LEFT JOIN resolves. It requires the CTE to be pre-aggregated so the COALESCE operates on a single array value, not on individual rows.

---

**Step 5: Plan CTE layers bottom-up**

**`stg_sentiment_question`** — Traverses the five-table chain from `clients_course_containers` through to `learning_cues`, then collapses all reachable `client_id` values per `learning_cue_id` into a deduplicated array. This pre-aggregation is required so the final join to `sentiment_questions` is safe from fan-out.

---

**Step 6: Write each CTE with justification**

**`stg_sentiment_question`** — This layer exists because the client-to-learning-cue relationship is indirect across five tables, and without collapsing to one row per `learning_cue_id` before the join, the LEFT JOIN to `sentiment_questions` would produce duplicate rows or require a secondary aggregation that would force all `sentiment_questions` metadata into a GROUP BY.

---

**Step 7: Compose the final query**

```sql
WITH
stg_sentiment_question AS (
    SELECT
        lc.id as learning_cue_id,
        ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
    FROM
        ${ref("clients_course_containers")} as ccc
    JOIN
        ${ref("courses")} as c
    ON
        c.container_id = ccc.course_container_id
    JOIN
        ${ref("course_lessons")} as cl
    ON
        cl.course_id = c.id
    JOIN
        ${ref("lessons")} as l
    ON 
        l.id = cl.lesson_id
    JOIN
        ${ref("learning_cues")} as lc
    ON
        lc.lesson_id = l.id
    GROUP BY
        lc.id
)


SELECT
    sq.id,
    sq.name,
    sq.learning_cue_id,
    sq.created,
    sq.updated,
    COALESCE(ssq.client_ids, [1]) AS client_ids
FROM
    ${ref("sentiment_questions")} as sq
LEFT JOIN
    stg_sentiment_question as ssq
ON
    ssq.learning_cue_id = sq.learning_cue_id
```
