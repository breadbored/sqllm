You are an expert analytical SQL reasoning assistant. When given a data transformation problem, you will reason through it step by step before writing any SQL.

Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table
   - For multiple tables, assume the tables are already joined for now and list all the columns as if they were already joined
     - `table1.X, table1.Y, table1.Z, table2.A, table2.B, table2.C`

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     -> person_id, session_id, SUM(page_time) as session_time
     -> person_id, SUM(session_time) as person_time
     ```
   - When a join depends on a pre-aggregated value, prefix the notation with a bracketed join clause:
     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     -> user_id, SUM(cues_reached) as total_cues_reached, SUM(lesson_progress) as course_progress
     ```

4. **Identify dependencies** - which columns drive grouping, which are derived, which require intermediate steps

5. **Plan CTE layers bottom-up** - name each CTE and describe in one sentence what it produces, using the arrow notation from step 3 as your guide

6. **Write each CTE with justification** - one sentence explaining why this layer exists

7. **Compose the final query**

8. **Verify output shape** - confirm the final output columns match the target shape stated in step 1

Do not skip steps. Do not write SQL before completing the reasoning. If the problem is ambiguous, state your assumptions explicitly before reasoning. Repeat the process from the beginning to refine the pseudocode outline if there are new realizations you arrive at while going down the list. ## Question
Give me the individual survey responses from the meta horizon survey, with one row per user per question per answer choice. If a question allows multiple answers, unnest them into separate rows. Include the session context, any free-text "other" responses, and the earliest timestamp for each response.

--- AGENT ---

## Step 1: Identify shapes

**Input shape (joined):**
`e.session_id, e.sub_session_id, e.user_id, e.name, e.action_meta (JSON), e.created, u.id, answer (unnested from JSON array)`

The `action_meta` JSON contains at minimum: `$.question`, `$.answers` (array), `$.other`, `$.client_id`.

**Target shape:**
`session_id, sub_session_id, user_id, question, answer, other_text, client_id, created`

One row per unique combination of session, sub-session, user, question, answer choice, other text, and client — with `created` collapsed to the earliest timestamp.

## Step 2: Identify sources and relationships

Two tables plus one unnest:

- **`app_events` (e)** — the event log, filtered to `name = 'meta_horizon_survey'`. Contains the JSON payload in `action_meta` and the session/user context.
- **`stg_users` (u)** — a staged users table. Joined on `u.id = e.user_id`. This is an INNER JOIN, so it acts as an existence filter — only events from known/valid users are included.
- **`UNNEST(JSON_EXTRACT_ARRAY(e.action_meta, '$.answers'))` as `answer`** — LEFT JOIN unnest of the answers array from the JSON payload. LEFT JOIN ensures that survey events with no answers array (or an empty one) are still retained, with `answer` as NULL (which COALESCE converts to `'other'`).

Join keys:
- `e.user_id = u.id` (events to users)
- The UNNEST is a lateral join on the same row's JSON column — no key needed.

## Step 3: Identify aggregation depth

Single-level aggregation. The only aggregate is `MIN(created)` to deduplicate or collapse repeated event firings for the same logical response.

```
session_id, sub_session_id, user_id, question, answer, other_text, client_id, MIN(created) as created
```

No multi-level aggregation — one arrow level only. The `HAVING question IS NOT NULL` post-filters to discard rows where the JSON extraction for question yielded NULL (malformed or irrelevant events).

## Step 4: Identify dependencies

- **Grouping columns:** `session_id`, `sub_session_id`, `user_id`, `question`, `answer`, `other_text`, `client_id` — all extracted or derived before aggregation.
- **Derived columns:**
  - `question` — extracted from JSON via `JSON_EXTRACT_SCALAR(e.action_meta, '$.question')`
  - `answer` — unnested from `JSON_EXTRACT_ARRAY(e.action_meta, '$.answers')`, then wrapped in `COALESCE(..., 'other')` to default NULLs
  - `other_text` — extracted from JSON via `JSON_EXTRACT_SCALAR(e.action_meta, '$.other')`
  - `client_id` — extracted from JSON via `JSON_EXTRACT_SCALAR(e.action_meta, '$.client_id')`
- **Aggregate column:** `MIN(e.created)` — earliest event timestamp per group.
- **Filter dependencies:** `e.name = 'meta_horizon_survey'` (WHERE), `question IS NOT NULL` (HAVING — must be post-aggregation since `question` is a group key derived from JSON extraction).
- The INNER JOIN to `stg_users` must execute before aggregation to exclude events from unknown users.

## Step 5: Plan CTE layers bottom-up

No CTEs are needed. This is a single-level query:

1. **Base query (no CTE)** — Filter `app_events` to survey events, LEFT JOIN unnest the answers array, INNER JOIN to `stg_users` for user validation, extract JSON fields, group by all dimensions, take `MIN(created)`, and filter out NULL questions via HAVING.

## Step 6: Write each CTE with justification

No CTEs required. The entire transformation is expressible in a single SELECT with a GROUP BY. The JSON extraction, unnesting, filtering, and aggregation all operate at the same grain without requiring intermediate materialization.

## Step 7: Compose the final query

```sql
SELECT
  e.session_id as session_id,
  e.sub_session_id as sub_session_id,
  e.user_id as user_id,
  JSON_EXTRACT_SCALAR(e.action_meta, '$.question') as question,
  COALESCE(JSON_EXTRACT_SCALAR(answer), 'other') as answer,
  JSON_EXTRACT_SCALAR(e.action_meta, '$.other') as other_text,
  JSON_EXTRACT_SCALAR(e.action_meta, '$.client_id') as client_id,
  MIN(e.created) as created
FROM
  ${ref("app_events")} as e
LEFT JOIN
  UNNEST(JSON_EXTRACT_ARRAY(e.action_meta, '$.answers')) as answer
JOIN
  ${ref("stg_users")} as u
ON
  u.id = e.user_id
WHERE
  e.name = 'meta_horizon_survey'
GROUP BY
  session_id,
  sub_session_id,
  user_id,
  question,
  answer,
  other_text,
  client_id
HAVING
  question IS NOT NULL
```

## Step 8: Verify output shape

Target shape: `session_id, sub_session_id, user_id, question, answer, other_text, client_id, created`

Final SELECT produces: `session_id, sub_session_id, user_id, question, answer, other_text, client_id, created` ✓

Grain: one row per unique (session, sub-session, user, question, answer choice, other text, client) combination. Multi-answer questions produce multiple rows via the UNNEST. Questions with no answers array produce one row with `answer = 'other'` via the LEFT JOIN + COALESCE. Rows with NULL question are excluded by HAVING. Only events from validated users survive the INNER JOIN to `stg_users`. ✓
