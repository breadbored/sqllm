--- USER ---

For each top banner in the system, return all of its standard fields along with the array of client IDs that are associated with it through its linked learning cue. If a banner's learning cue has no client associations (or the banner has no learning cue), default the client list to `[1]`.

--- AGENT ---

## Step 1: Identify Shapes

**Input columns** — treating all tables as if pre-joined:

```
clients_course_containers.client_id, clients_course_containers.course_container_id,
courses.id, courses.container_id,
course_lessons.course_id, course_lessons.lesson_id,
lessons.id,
learning_cues.id, learning_cues.lesson_id,
top_banners.id, top_banners.learning_cue_id, top_banners.course_container_id,
top_banners.display_seconds, top_banners.created, top_banners.updated,
top_banners.lesson_resource_id
```

**Target shape** — one row per top banner:

```
id, learning_cue_id, course_container_id, display_seconds, created, updated,
lesson_resource_id, client_ids (ARRAY)
```

---

## Step 2: Identify Sources and Relationships

The core challenge is that `top_banners` references `learning_cues` via `top_banners.learning_cue_id`, but the client visibility dimension lives in `clients_course_containers` — five hops away. There is no direct column bridging the two. The chain must be traversed in order:

1. `clients_course_containers.course_container_id` → `courses.container_id` *(client-to-course link)*
2. `courses.id` → `course_lessons.course_id` *(course-to-lesson link)*
3. `course_lessons.lesson_id` → `lessons.id` *(lesson identity)*
4. `lessons.id` → `learning_cues.lesson_id` *(lesson-to-cue link)*

This is a pure 4-hop chain with no glue ambiguity — each join has a single unambiguous key. The result of traversing it is a set of `(learning_cue_id, client_id)` pairs, which is what needs to be collected into an array per learning cue before joining to `top_banners`.

The final join is `top_banners.learning_cue_id = learning_cue_clients.learning_cue_id`. It is a LEFT JOIN because not every top banner is guaranteed to have an associated learning cue (or one with any client associations), and the COALESCE to `[1]` handles that NULL case.

---

## Step 3: Identify Aggregation Depth

There is one level of aggregation — in the CTE — and no further aggregation in the final query. The CTE collapses the many `client_id` rows per `learning_cue_id` into a single array:

```
learning_cue_id, client_id
-> learning_cue_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

The final query is a lookup join on `learning_cue_id` with no additional grouping. One row in, one row out per `top_banner`.

---

## Step 4: Identify Dependencies

- `client_ids` cannot be computed inline in the final SELECT without either multiplying `top_banners` rows (one per client per cue) or wrapping a correlated subquery. A pre-aggregated CTE is the correct structural choice.
- The COALESCE default `[1]` depends on the LEFT JOIN producing a NULL `client_ids` for unmatched banners — which is only safe after the array has already been collapsed to one row per `learning_cue_id`. If the join were done before aggregation, a missing learning cue would produce no rows at all rather than a NULL.
- `ARRAY_AGG(DISTINCT ... IGNORE NULLS)` is required because the 4-hop join may produce duplicate `client_id` values (one per course containing the lesson) and may produce NULLs if any intermediate join leg has optional rows.

---

## Step 5: Plan CTE Layers Bottom-Up

**`learning_cue_clients`** — traverses the 5-table chain from `clients_course_containers` through `courses → course_lessons → lessons → learning_cues`, then collapses each `learning_cue_id` to one row containing an array of all distinct associated `client_id` values.

One CTE. One arrow. The final SELECT is a pass-through lookup join.

---

## Step 6: Write Each CTE with Justification

**`learning_cue_clients`** — This CTE exists because the relationship between a `learning_cue` and its visible clients is indirect (4 hops), and the many-to-many fan-out from that join must be collapsed to one row per `learning_cue_id` before it can be safely joined to `top_banners` without duplicating banner rows.

---

## Step 7: Compose the Final Query

```sql
WITH
learning_cue_clients as (
    SELECT
        lc.id as learning_cue_id,
        ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
    FROM
        ${ref("clients_course_containers")} as ccc
    JOIN
        ${ref("courses")} as c
    ON
        c.container_id = ccc.course_container_id
    JOIN
        ${ref("course_lessons")} as cl
    ON
        cl.course_id = c.id
    JOIN
        ${ref("lessons")} as l
    ON 
        l.id = cl.lesson_id
    JOIN
        ${ref("learning_cues")} as lc
    ON
        lc.lesson_id = l.id
    GROUP BY
        lc.id
)
SELECT
  tb.id,
  tb.learning_cue_id,
  tb.course_container_id,
  tb.display_seconds,
  tb.created,
  tb.updated,
  tb.lesson_resource_id,
  COALESCE(slr.client_ids, [1]) AS client_ids
FROM
  ${ref("top_banners")} as tb
LEFT JOIN
    learning_cue_clients as slr
ON
    tb.learning_cue_id = slr.learning_cue_id
```
