## Question
For each day, show the number of total active users, returning users, and new users based on transaction activity. A "new" user is one whose account was created on the same day as the transaction; a "returning" user is one whose account was created on a prior day.

--- AGENT ---

## 1. Identify shapes

**Input shape (joined):**
- `agg_transactions`: `created`, `user_id` (plus other columns not referenced)
- `stg_users`: `id`, `created` (plus other columns not referenced)

**Target shape:**
| date | active_users | returning_users | new_users |
|------|-------------|-----------------|-----------|

One row per calendar date. All three user counts are distinct-user counts for that date.

## 2. Identify sources and relationships

Two tables are involved:

- `agg_transactions` (aliased `t`) — contains transaction events with `user_id` and `created` timestamp
- `stg_users` (aliased `u`) — contains user records with `id` and `created` timestamp

**Join relationship:** `t.user_id = u.id` — the transaction's user foreign key joins to the user's primary key.

The query uses the same two tables twice:
1. **Outer query** — all transactions joined to users, counting all distinct active users per day
2. **Subquery (`returning`)** — same join, but filtered to only transactions where `date(u.created) != date(t.created)`, meaning the user's account was not created on the transaction date. This isolates returning users.

The subquery result is LEFT JOINed back to the outer query on `date`, so that each day gets its returning user count attached. New users are derived arithmetically: `active_users - returning_users`.

## 3. Identify aggregation depth

There are two independent aggregation paths that merge via a join, not a multi-level rollup:

**Path A (outer — all active users):**
```
date(t.created), COUNT(DISTINCT t.user_id) as active_users
```

**Path B (subquery — returning users only):**
```
[filtered: date(u.created) != date(t.created)]
date(t.created), COUNT(DISTINCT t.user_id) as active_users
```

**Merge:**
```
Path A LEFT JOIN Path B on date -> date, active_users, returning_users, (active_users - returning_users) as new_users
```

This is a single level of aggregation in two parallel branches, not a multi-level rollup.

## 4. Identify dependencies

- **Grouping driver:** `date(t.created)` — the calendar date extracted from the transaction timestamp
- **Derived columns:**
  - `active_users` — `COUNT(DISTINCT t.user_id)` across all transactions for that date
  - `returning_users` — `COUNT(DISTINCT t.user_id)` where the user was not created on that same date
  - `new_users` — arithmetic difference: `active_users - returning_users`
- **Filter dependency:** The returning-user subquery depends on `u.created` to distinguish new vs. returning, which requires the join to `stg_users` to be evaluated before the filter
- **COALESCE dependency:** Days with zero returning users will produce NULL from the LEFT JOIN; `COALESCE(..., 0)` handles this, and the `new_users` subtraction depends on this coalesced value

## 5. Plan CTE layers bottom-up

This query uses no CTEs — it uses an inline subquery instead. The logical layers are:

1. **Subquery `returning`** — aggregates returning users per date by filtering out same-day-created users, then counting distinct user IDs grouped by transaction date
2. **Outer query** — aggregates all active users per date, LEFT JOINs the returning count, and derives new users by subtraction

## 6. Write each CTE with justification

**Subquery `returning`:** Exists because returning users require a filter (`date(u.created) != date(t.created)`) that cannot be applied to the main aggregation without excluding new users from the total active count. A separate aggregation pass isolates this subset.

**Outer query:** Computes the unfiltered total of distinct active users per day, then attaches the returning count via LEFT JOIN. The LEFT JOIN ensures days with zero returning users still appear. The arithmetic `active - returning = new` avoids a third aggregation pass.

## 7. Compose the final query

```sql
SELECT
  date(t.created)                                                 AS date,
  count(DISTINCT t.user_id)                                       AS active_users,
  coalesce(returning.active_users, 0)                             AS returning_users,
  count(DISTINCT t.user_id) - coalesce(returning.active_users, 0) AS new_users
FROM ${ref("agg_transactions")} AS t
INNER JOIN
  ${ref("stg_users")} AS u
  ON t.user_id = u.id
LEFT JOIN
  (
    SELECT
      date(t.created)           AS date,
      count(DISTINCT t.user_id) AS active_users
    FROM ${ref("agg_transactions")} AS t
    INNER JOIN
      ${ref("stg_users")} AS u
      ON t.user_id = u.id
    WHERE date(u.created) != date(t.created)
    GROUP BY date(t.created)
  ) AS returning
  ON returning.date = date(t.created)
GROUP BY
  date(t.created), returning.active_users
ORDER BY date
```

Target shape: `date, active_users, returning_users, new_users` — one row per calendar date.

- `date` — `date(t.created)`, grouped ✓
- `active_users` — `COUNT(DISTINCT t.user_id)`, aggregate ✓
- `returning_users` — `COALESCE(returning.active_users, 0)`, scalar from LEFT JOIN (included in GROUP BY via `returning.active_users`) ✓
- `new_users` — arithmetic of the above two ✓

One row per date is guaranteed by `GROUP BY date(t.created), returning.active_users` — since `returning.active_users` is functionally dependent on `date` (the subquery groups by date and joins on date), the composite GROUP BY does not multiply rows. Output shape matches target. ✓
