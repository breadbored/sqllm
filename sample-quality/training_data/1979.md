## Question
Give me all course-lesson records with their associated client IDs. If a course has no client associations, default to client ID 1.

--- AGENT ---

## 1. Identify shapes

**Input tables:**
- `clients_course_containers` (ccc): `client_id`, `course_container_id`
- `courses` (c): `id`, `container_id`
- `course_lessons` (cl): `id`, `course_id`, `lesson_id`, `created`, `updated`

**Target shape:**
```
id, course_id, lesson_id, created, updated, client_ids
```

One row per course-lesson, with an array of client IDs attached. Courses with no client associations get a default array of `[1]`.

## 2. Identify sources and relationships

- `courses.container_id` joins to `clients_course_containers.course_container_id` — this links courses to clients through a container abstraction.
- `course_lessons.course_id` joins to `courses.id` — this links lessons to courses.
- The indirect chain is: `course_lessons` → (via `course_id`) → `courses` → (via `container_id = course_container_id`) → `clients_course_containers`.

The client IDs must be pre-aggregated per course before joining to `course_lessons`, otherwise the join would fan out lesson rows across multiple clients.

## 3. Identify aggregation depth

Single-level aggregation, pre-joined before the final SELECT:

```
course_id, ARRAY_AGG(DISTINCT client_id) as client_ids
```

No further aggregation after this — the final query is a flat join with no GROUP BY.

## 4. Identify dependencies

- **Grouping driver:** `course_id` — client IDs must be collapsed into an array per course before the join to `course_lessons`.
- **Derived:** `client_ids` — the array aggregate of distinct client IDs per course.
- **Intermediate step required:** The `courses` table is needed as a bridge between `clients_course_containers.course_container_id` and the `course_id` used in `course_lessons`. This bridge join and the aggregation must happen before the final join.
- **Default handling:** `COALESCE(..., [1])` applied in the final SELECT handles courses with no client container associations.

## 5. Plan CTE layers bottom-up

1. **`course_clients`** — Joins `courses` to `clients_course_containers` via the container ID, then aggregates distinct client IDs into an array per `course_id`.

That single CTE is sufficient. The final query LEFT JOINs `course_lessons` to the pre-aggregated `course_clients` and applies the COALESCE default.

## 6. Write each CTE with justification

**`course_clients`**: Pre-aggregates client IDs per course so the final join to `course_lessons` does not fan out rows. The join through `courses.container_id = ccc.course_container_id` resolves the indirect relationship between course IDs and client IDs. `ARRAY_AGG(DISTINCT ... IGNORE NULLS)` ensures each client appears once and nulls are excluded.

## 7. Compose the final query

```sql
WITH
course_clients AS (
  SELECT
    c.id as course_id,
    ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("clients_course_containers")} as ccc
  JOIN
    ${ref("courses")} as c
  ON
    c.container_id = ccc.course_container_id
  GROUP BY
    c.id
)
SELECT
  cl.id,
  cl.course_id,
  cl.lesson_id,
  cl.created,
  cl.updated,
  COALESCE(cc.client_ids, [1]) AS client_ids
FROM
  ${ref("course_lessons")} as cl
LEFT JOIN
  course_clients as cc
ON
  cc.course_id = cl.course_id
```

Target: `id, course_id, lesson_id, created, updated, client_ids`

- `cl.lesson_id` ✓

Grain: one row per course-lesson record. The LEFT JOIN ensures all course-lessons appear even if no client mapping exists, and COALESCE provides the `[1]` default in that case. Output shape matches target.
