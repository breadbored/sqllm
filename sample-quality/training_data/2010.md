--- USER ---

For each row in the `question_sets` table, return its core fields along with an array of client IDs that have access to the question through the course content hierarchy. If a question has no associated clients in the hierarchy, default the client array to `[1]`.

--- AGENT ---

## Step 1: Identify Shapes

**Input tables (joined chain within CTE):**
`clients_course_containers.client_id, clients_course_containers.course_container_id, courses.id, courses.container_id, course_lessons.course_id, course_lessons.lesson_id, learning_cues.id, learning_cues.lesson_id, knowledge_checks.id, knowledge_checks.learning_cue_id, knowledge_check_questions.id, knowledge_check_questions.knowledge_check_id`

**Final driving table:** `question_sets.id, question_sets.question_id, question_sets.correct_option_id, question_sets.created, question_sets.updated`

**Target output shape:**
`id, question_id, correct_option_id, created, updated, client_ids (ARRAY<INT64>)`
One row per row in `question_sets`, with `client_ids` being an array of all client IDs that reach each question through the course hierarchy — defaulting to `[1]` when none exist.

---

## Step 2: Identify Sources and Relationships

The problem requires linking `question_sets` to `clients_course_containers` — two tables with no direct shared column. The path is resolved through a 5-hop glue chain:

| Left table | Join key (left) | Right table | Join key (right) |
|---|---|---|---|
| `clients_course_containers` | `course_container_id` | `courses` | `container_id` |
| `courses` | `id` | `course_lessons` | `course_id` |
| `course_lessons` | `lesson_id` | `learning_cues` | `lesson_id` |
| `learning_cues` | `id` | `knowledge_checks` | `learning_cue_id` |
| `knowledge_checks` | `id` | `knowledge_check_questions` | `knowledge_check_id` |

This chain resolves `client_id → course_container → course → lesson → learning_cue → knowledge_check → question`. The result of walking this chain is a set of `(question_id, client_id)` pairs.

The final join connects `question_sets.question_id` to `knowledge_check_questions.id` (aliased as `question_id` in the CTE), meaning `question_sets` references individual knowledge check questions by their ID.

The final join is a `LEFT JOIN` — not all `question_sets` rows are guaranteed to be reachable through the client-course hierarchy, and the `COALESCE(..., [1])` handles those gaps explicitly.

---

## Step 3: Identify Aggregation Depth

The join chain produces one row per `(question_id, client_id)` combination — a fan-out from the multi-client, multi-course structure. These must be collapsed to one row per `question_id` before joining back to `question_sets`, otherwise the LEFT JOIN would multiply `question_sets` rows.

**Single aggregation level — one CTE required:**

```
question_id (kcq.id), ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

The final SELECT does no further aggregation — it is a 1:1 LEFT JOIN from `question_sets` to the pre-collapsed CTE output.

---

## Step 4: Identify Dependencies

- `client_ids` per question requires traversing the full 5-hop hierarchy from `clients_course_containers` to `knowledge_check_questions`. This cannot be inlined into the final SELECT without multiplying `question_sets` rows, so it must be pre-aggregated in a CTE.
- `ARRAY_AGG(DISTINCT ... IGNORE NULLS)` is required because the join chain can produce the same `client_id` via multiple course paths (a client may own multiple courses sharing the same questions), and outer-join nulls should not pollute the array.
- `COALESCE(sqs.client_ids, [1])` — when a question has no client-course path at all (LEFT JOIN produces NULL), the array literal `[1]` is substituted. The integer `1` is structurally a default client identifier; its business meaning is not derivable from the query alone.
- `question_sets` is the driving table. All its rows must be preserved — the CTE enriches them, not filters them.

---

## Step 5: Plan CTE Layers Bottom-Up

**One CTE layer required:**

`stg_question_set` — walks the 5-hop client-to-question hierarchy and produces one row per `knowledge_check_questions.id`, carrying the full array of distinct client IDs that can reach that question through any course path.

---

## Step 6: Write Each CTE with Justification

**`stg_question_set`** — Pre-aggregates the client ID fanout created by the 5-hop hierarchy join, collapsing to one row per question ID so the downstream LEFT JOIN to `question_sets` is a safe 1:≤1 relationship.

## Step 7: Compose the Final Query

```sql
WITH
stg_question_set AS (
    SELECT
        kcq.id as question_id,
        ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
    FROM
        ${ref("clients_course_containers")} as ccc
    JOIN
        ${ref("courses")} as c
    ON
        c.container_id = ccc.course_container_id
    JOIN
        ${ref("course_lessons")} as cl
    ON
        cl.course_id = c.id
    JOIN
        ${ref("learning_cues")} as lc
    ON
        lc.lesson_id = cl.lesson_id
    JOIN
        ${ref("knowledge_checks")} as kc
    ON
        kc.learning_cue_id = lc.id
    JOIN
        ${ref("knowledge_check_questions")} as kcq
    ON
        kcq.knowledge_check_id = kc.id
    GROUP BY
        kcq.id
)

SELECT
    qs.id,
    qs.question_id,
    qs.correct_option_id,
    qs.created,
    qs.updated,
    COALESCE(sqs.client_ids, [1]) AS client_ids
FROM
    ${ref("question_sets")} as qs
LEFT JOIN
    stg_question_set as sqs
ON
    sqs.question_id = qs.question_id
```
