--- USER ---

For each day, show the total number of active users, how many are returning users, and how many are new users. A "new" user is one whose account was created on the same day as their transaction; a "returning" user is one whose account was created on a prior day.

--- AGENT ---

## 1. Identify shapes

**Input shape (joined):**
- `agg_transactions`: `created`, `user_id` (plus other columns not referenced)
- `stg_users`: `id`, `created`

**Target shape:**
```
date, active_users, returning_users, new_users
```

One row per calendar date. `active_users` is the total distinct users transacting that day, `returning_users` is the subset whose account predates that day, and `new_users` is the remainder.

## 2. Identify sources and relationships

Two tables are involved:

- `agg_transactions` (aliased `t`) — contains transaction events with `user_id` and `created` timestamp
- `stg_users` (aliased `u`) — contains user records with `id` and `created` timestamp

**Join relationship:** `CAST(u.id AS STRING) = CAST(t.user_id AS STRING)` — a type-coerced equality join. The cast suggests `u.id` and `t.user_id` are stored in different types (likely one numeric, one string). This is used identically in both the outer query and the subquery.

The subquery `returning` is a self-contained aggregation over the same two tables, filtered to rows where the user's creation date differs from the transaction date. It is joined back to the outer query on `date`.

## 3. Identify aggregation depth

There are two parallel aggregation paths, not stacked levels:

**Path A — outer query (all active users):**
```
date(t.created) AS date, COUNT(DISTINCT t.user_id) AS active_users
```

**Path B — subquery (returning users only):**
```
[WHERE date(u.created) != date(t.created)]
date(t.created) AS date, COUNT(DISTINCT t.user_id) AS active_users (as returning.active_users)
```

**Composition (not a second aggregation level — a lateral join of two parallel aggregations):**
```
date, active_users, COALESCE(returning.active_users, 0) AS returning_users, active_users - returning_users AS new_users
```

Arrow notation for each path:

Path A:
```
date(t.created), COUNT(DISTINCT t.user_id) AS active_users
```

Path B:
```
[filtered to date(u.created) != date(t.created)]
date(t.created), COUNT(DISTINCT t.user_id) AS active_users (aliased returning_users)
```

Final composition:
```
date, active_users, COALESCE(returning.active_users, 0) AS returning_users, active_users - returning_users AS new_users
```

## 4. Identify dependencies

- **Grouping driver:** `date(t.created)` — the calendar date extracted from the transaction timestamp drives both the outer and subquery grouping.
- **Derived columns:**
  - `active_users` — `COUNT(DISTINCT t.user_id)` in the outer query (all users transacting that day)
  - `returning_users` — `COUNT(DISTINCT t.user_id)` in the subquery, filtered to `date(u.created) != date(t.created)`
  - `new_users` — arithmetic: `active_users - returning_users`
- **Intermediate step:** The subquery must be computed independently before the outer query can reference `returning.active_users`. The LEFT JOIN ensures dates with zero returning users still appear (with `COALESCE` to 0).
- **The `GROUP BY` includes `returning.active_users`** because it is referenced directly in the SELECT without aggregation. Since the LEFT JOIN is on `date` (which is unique in the subquery output), this does not change the grain — it is functionally determined by `date`.

## 5. Plan CTE layers bottom-up

This query uses no CTEs — it uses an inline subquery instead. The logical layers are:

1. **Subquery `returning`** — joins transactions to users, filters to rows where the user existed before the transaction date, counts distinct users per date.
2. **Outer query** — joins transactions to users (no date filter), counts all distinct users per date, LEFT JOINs the `returning` subquery to attach the returning user count, and derives new users by subtraction.

## 6. Write each CTE with justification

**Subquery `returning`:** Computes the count of distinct users per day whose account creation date is strictly different from the transaction date. This isolates the "returning" population — users who were not created on the day they transacted.

**Outer query:** Computes total distinct active users per day across all transactions (regardless of account age), then joins the returning count to derive new users as the difference. The LEFT JOIN ensures days with only new users (no returning users) still appear with `returning_users = 0`.

## 7. Compose the final query

```sql
SELECT
  date(t.created)                                                 AS date,
  count(DISTINCT t.user_id)                                       AS active_users,
  coalesce(returning.active_users, 0)                             AS returning_users,
  count(DISTINCT t.user_id) - coalesce(returning.active_users, 0) AS new_users
FROM ${ref("agg_transactions")} AS t
INNER JOIN
  ${ref("stg_users")} AS u
  ON cast(u.id AS string) = cast(t.user_id AS string)
LEFT JOIN
  (
    SELECT
      date(t.created)           AS date,
      count(DISTINCT t.user_id) AS active_users
    FROM ${ref("agg_transactions")} AS t
    INNER JOIN
      ${ref("stg_users")} AS u
      ON cast(u.id AS string) = cast(t.user_id AS string)
    WHERE date(u.created) != date(t.created)
    GROUP BY date(t.created)
  ) AS returning
  ON returning.date = date(t.created)
GROUP BY
  date(t.created), returning.active_users
ORDER BY date
```

## 8. Verify output shape

Target shape: `date, active_users, returning_users, new_users`

Final SELECT produces:
- `date` — `date(t.created)` ✓
- `active_users` — `COUNT(DISTINCT t.user_id)` ✓
- `returning_users` — `COALESCE(returning.active_users, 0)` ✓
- `new_users` — `active_users - returning_users` ✓

**Grain:** one row per calendar date, ordered by date. Confirmed — the GROUP BY is on `date(t.created)` (plus the functionally dependent `returning.active_users`), and the subquery also groups by `date(t.created)`, so the LEFT JOIN is 1:1 on date.

**Note on the INNER JOIN to `stg_users`:** The outer query joins to `stg_users` but does not filter on `u.created`. This join still serves a purpose — it restricts the active user count to users that exist in `stg_users`, effectively excluding transactions from users not present in that table (e.g., deleted or unstaged users). Without this join, orphaned `user_id` values in `agg_transactions` would inflate the active user count.
