--- SYSTEM ---

You are an expert analytical SQL reasoning assistant. When given a data transformation problem, you will reason through it step by step before writing any SQL.

Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table
   - For multiple tables, assume the tables are already joined for now and list all the columns as if they were already joined
     - `table1.X, table1.Y, table1.Z, table2.A, table2.B, table2.C`

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     -> person_id, session_id, SUM(page_time) as session_time
     -> person_id, SUM(session_time) as person_time
     ```
   - When a join depends on a pre-aggregated value, prefix the notation with a bracketed join clause:
     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     -> user_id, SUM(cues_reached) as total_cues_reached, SUM(lesson_progress) as course_progress
     ```

4. **Identify dependencies** - which columns drive grouping, which are derived, which require intermediate steps

5. **Plan CTE layers bottom-up** - name each CTE and describe in one sentence what it produces, using the arrow notation from step 3 as your guide

6. **Write each CTE with justification** - one sentence explaining why this layer exists

7. **Compose the final query**

8. **Verify output shape** - confirm the final output columns match the target shape stated in step 1

Do not skip steps. Do not write SQL before completing the reasoning. If the problem is ambiguous, state your assumptions explicitly before reasoning. Repeat the process from the beginning to refine the pseudocode outline if there are new realizations you arrive at while going down the list. The process must be followed at least once, and ideally a second time for validating the process.

## Question
Give me a daily breakdown of total, new, and returning sessions per client, where a "new" session is one whose first-ever activity falls on that day and a "returning" session is one that was first seen on a previous day.

--- AGENT ---

## 1. Identify shapes

**Input tables:**
- `stg_events_app_speedy` (aliased `e`): `session_id`, `created`, `doc_id`
- `agg_client_documents` (aliased `cd`): `doc_id`, `client_id` (array)

**Target output shape:**
- `total_sessions`, `new_sessions`, `returning_sessions`, `day`, `client_id`

This is a per-day, per-client summary with three distinct session counts.

## 2. Identify sources and relationships

- `e.doc_id = cd.doc_id` — events are joined to client-document mapping on `doc_id`.
- `cd.client_id` is an array (repeated field), so it is unnested with `UNNEST(cd.client_id) AS unnested_client_id`. This fans out each event row to one row per client associated with the document.
- The relationship chain is: **session → event → document → client**. A session touches documents, and documents belong to clients, so sessions are attributed to clients through document ownership.

## 3. Identify aggregation depth

There is a window function layer followed by a single aggregation layer.

The window function `MIN(day) OVER (PARTITION BY session_id)` computes each session's first-ever date. This is not a GROUP BY aggregation — it decorates every row with the session's origin date so the final layer can classify sessions as new or returning.

Arrow notation for the actual aggregation:

```
day, client_id, session_id, first_date,
  COUNT(DISTINCT session_id) as total_sessions,
  COUNT(DISTINCT CASE WHEN day = first_date THEN session_id END) as new_sessions,
  COUNT(DISTINCT CASE WHEN day != first_date THEN session_id END) as returning_sessions
-> (final output, single level)
```

This is a single-level aggregation. The window function is a pre-computation step (expressed in the CTE), not a separate aggregation level.

## 4. Identify dependencies

- **Grouping columns:** `day`, `client_id` — these define the grain of the output.
- **Derived columns:**
  - `first_date` — derived via `MIN(DATETIME_TRUNC(created, DAY)) OVER (PARTITION BY session_id)`. This must be computed before the final GROUP BY so each row knows whether its session is new or returning on that day.
  - `total_sessions` — `COUNT(DISTINCT session_id)` per group.
  - `new_sessions` — conditional distinct count where the session's activity day matches its first-ever date.
  - `returning_sessions` — conditional distinct count where the day does not match the first-ever date.
- **Intermediate step required:** The window function that computes `first_date` must run before the final aggregation. This is why the CTE exists — it prepares a row-level dataset annotated with each session's origin date.

## 5. Plan CTE layers bottom-up

**CTE `docs_sessions`:** Joins events to client-document mappings (unnesting the client array), truncates timestamps to day granularity, and computes each session's first-ever date via a window function. Produces one row per event × client combination, annotated with `day`, `first_date`, `session_id`, and `unnested_client_id`.

**Final SELECT:** Groups by `day` and `client_id`, counts total distinct sessions, and uses conditional `COUNT(DISTINCT ...)` to split sessions into new vs. returning based on whether `day = first_date`.

## 6. Write each CTE with justification

**`docs_sessions`:** This CTE exists to (a) resolve the session-to-client relationship through the document join and array unnest, (b) truncate timestamps to day granularity, and (c) compute each session's first-ever date via a window function — all of which must happen before the final GROUP BY can classify and count sessions.

## 7. Compose the final query

```sql
WITH
  docs_sessions AS (
  SELECT
    e.session_id,
    DATETIME_TRUNC(e.created, DAY) AS day,
    MIN(DATETIME_TRUNC(e.created, DAY)) OVER (PARTITION BY e.session_id) AS first_date,
    unnested_client_id
  FROM
    ${ref("stg_events_app_speedy")} e
  JOIN
    ${ref("agg_client_documents")} cd
  ON
    e.doc_id = cd.doc_id,
    UNNEST(cd.client_id) AS unnested_client_id)
SELECT
  COUNT(DISTINCT session_id) AS total_sessions,
  COUNT(DISTINCT
    CASE
      WHEN day = first_date THEN session_id
  END
    ) AS new_sessions,
  COUNT(DISTINCT
    CASE
      WHEN day != first_date THEN session_id
  END
    ) AS returning_sessions,
  day,
  unnested_client_id AS client_id
FROM
  docs_sessions
GROUP BY
  day,
  client_id
ORDER BY
  day,
  client_id
```

## 8. Verify output shape

Target shape: `total_sessions`, `new_sessions`, `returning_sessions`, `day`, `client_id`

Final SELECT produces exactly these five columns:
- `COUNT(DISTINCT session_id) AS total_sessions` ✓
- `COUNT(DISTINCT CASE WHEN day = first_date ...) AS new_sessions` ✓
- `COUNT(DISTINCT CASE WHEN day != first_date ...) AS returning_sessions` ✓
- `day` ✓
- `unnested_client_id AS client_id` ✓

Grain is one row per `(day, client_id)` pair. Output shape matches the target. ✓
