Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```

   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time

     -> person_id, SUM(session_time) as person_time
     ```

     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     ```

## Tables
### agg_client_transactions
| column | type |
|---|---|
| user_id | INT64 |
| user_created | DATETIME |
| day | DATETIME |
| client_id | ARRAY<INT64> |

## Question
Generate a yearly client-level user activity report showing unique users, new users, returning users, users who appear in both categories, year-scoped new and returning users, total sessions, returning user sessions, and sessions per user — broken down by client and year.

--- AGENT ---

## Step 1: Identify shapes

**Input shape:**
The query reads from a single source table `agg_client_transactions` with at least these columns:
- `user_id` — identifies a user
- `client_id` — an ARRAY column (it is UNNESTed), identifying one or more clients per row
- `day` — a DATE column representing the date of a transaction/session

**Target shape (output):**
```
unique_users, new_users, returning_users, new_users_returning,
year_new_users, year_returning_users, total_sessions,
returning_user_sessions, sessions_per_user, year, client_id
```

One row per (year, client_id) combination, containing various user classification counts and session metrics.

## Step 2: Identify sources and relationships

There is only one source table (`agg_client_transactions`), but it is read twice — once to compute each user's first-ever transaction day per client (`client_transactions_min`), and once to produce the per-day transaction rows with year boundaries (`year_periods`).

The join relationship between `year_periods` and `client_transactions_min` (aliased through `meta_quest_min`) is on:
- `user_id = user_id`
- `client_id = client_id`

This attaches each user's first-ever day (per client) to every transaction row for that user+client, enabling new-vs-returning classification.

## Step 3: Identify aggregation depth

There are two conceptual levels, but the query expresses them as one pre-computation CTE + one final aggregation:

**Pre-computation (CTE level):**
```
user_id, client_id, MIN(day) as min_day
```
This is not a "level" feeding into a higher aggregation — it produces a lookup dimension (the user's first day per client).

**Final aggregation:**
```
[year_periods joined to meta_quest_min on user_id, client_id]
FORMAT_DATE('%Y', year_start), client_id,
  COUNT(DISTINCT user_id) as unique_users,
  COUNT(DISTINCT CASE WHEN min_day = day THEN user_id END) as new_users,
  COUNT(DISTINCT CASE WHEN min_day != day THEN user_id END) as returning_users,
  (new_users + returning_users) - unique_users as new_users_returning,
  COUNT(DISTINCT CASE WHEN min_day within year THEN user_id END) as year_new_users,
  COUNT(DISTINCT CASE WHEN min_day outside year THEN user_id END) as year_returning_users,
  COUNT(user_id) as total_sessions,
  COUNT(CASE WHEN min_day != day THEN user_id END) as returning_user_sessions,
  ROUND(total_sessions / unique_users, 2) as sessions_per_user
```

Arrow notation:
```
user_id, client_id, MIN(day) as min_day
-> [joined to year_periods] year, client_id, COUNT(DISTINCT user_id) as unique_users, COUNT(DISTINCT CASE new) as new_users, COUNT(DISTINCT CASE returning) as returning_users, COUNT(user_id) as total_sessions, ...
```

Two CTE layers for preparation, one final GROUP BY — but only one true aggregation depth since `min_day` is a lookup, not a value that gets re-aggregated.

## Step 4: Identify dependencies

- **Grouping drivers:** `year_start` (derived from `day`) and `client_id` drive the final GROUP BY.
- **Derived lookup:** `min_day` must be computed first (per user, per client) before the final aggregation can classify users as new or returning.
- **Classification columns:** `new_users`, `returning_users`, `year_new_users`, `year_returning_users` all depend on comparing `min_day` against either the row's `day` or the row's `year_start` boundaries.
- **`new_users_returning`** is an arithmetic expression over two DISTINCT counts — it captures users who had their first day within the period but also have other sessions, so they appear in both the new and returning counts. The formula `(new + returning) - unique` isolates the overlap.

## Step 5: Plan CTE layers bottom-up

1. **`client_transactions_min`** — UNNESTs the client_id array and computes each user's earliest transaction day per client.
2. **`meta_quest_min`** — A pass-through alias of `client_transactions_min` (no transformation). Likely a naming convention artifact or placeholder for a filter that was removed.
3. **`year_periods`** — UNNESTs the client_id array again and computes the `year_start` (January 1 of the transaction's year) for each transaction row, producing the fact table for the final aggregation.
4. **Final SELECT** — Joins `year_periods` to `meta_quest_min`, groups by year and client, and computes all classification counts and session metrics.

## Step 6: Write each CTE with justification

**`client_transactions_min`:** Exists to determine each user's first-ever transaction day per client, which is the reference point for all new-vs-returning classifications.

**`meta_quest_min`:** A direct passthrough of `client_transactions_min` with no added logic. Likely retained as a naming abstraction (the name suggests this may have originally filtered to a specific client or platform). It serves as the lookup dimension joined into the final query.

**`year_periods`:** Exists to UNNEST the client_id array and derive the `year_start` date for each transaction row, enabling year-boundary grouping and year-scoped new/returning classification in the final aggregation.

## Step 7: Compose the final query

```sql
WITH
  client_transactions_min AS (
  SELECT
    ct.user_id,
    client_id_expanded AS client_id,
    MIN(ct.`day`) AS min_day
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS client_id_expanded
  GROUP BY
    ct.user_id,
    client_id_expanded),
  meta_quest_min AS (
  SELECT
    ctm.user_id,
    ctm.client_id,
    ctm.min_day
  FROM
    client_transactions_min AS ctm),
  year_periods AS (
  SELECT
    ct.user_id,
    client_id_expanded AS client_id,
    ct.`day`,
    DATE(EXTRACT(YEAR
      FROM
        `day`), 1, 1) AS year_start
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS client_id_expanded)
SELECT
  COUNT(DISTINCT ct.user_id) AS unique_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day = ct.`day` THEN ct.user_id
  END
    ) AS new_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day != ct.`day` THEN ct.user_id
  END
    ) AS returning_users,
  ( COUNT( DISTINCT
      CASE
        WHEN mqm.min_day != ct.`day` THEN ct.user_id
    END
      ) + COUNT( DISTINCT
      CASE
        WHEN mqm.min_day = ct.`day` THEN ct.user_id
    END
      ) ) - COUNT(DISTINCT ct.user_id) AS new_users_returning,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day >= year_start AND mqm.min_day < DATE_ADD(year_start, INTERVAL 1 YEAR) THEN ct.user_id
  END
    ) AS year_new_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day < year_start OR mqm.min_day >= DATE_ADD(year_start, INTERVAL 1 YEAR) THEN ct.user_id
  END
    ) AS year_returning_users,
  COUNT(ct.user_id) AS total_sessions,
  COUNT(
    CASE
      WHEN mqm.min_day != ct.`day` THEN ct.user_id
  END
    ) AS returning_user_sessions,
  ROUND(COUNT(ct.user_id) / COUNT(DISTINCT ct.user_id), 2) AS sessions_per_user,
  FORMAT_DATE('%Y', year_start) AS year,
  ct.client_id AS client_id
FROM
  year_periods AS ct
INNER JOIN
  meta_quest_min AS mqm
ON
  mqm.user_id = ct.user_id
  AND mqm.client_id = ct.client_id
GROUP BY
  FORMAT_DATE('%Y', year_start),
  year_start,
  ct.client_id
ORDER BY
  year_start
```
