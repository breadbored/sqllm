## Question
For each search event in the document viewer, calculate which search result number the user is currently viewing. Users navigate between search results using "next" and "prev" actions, and the result number should wrap around when it exceeds the total result count or goes below 1.

--- AGENT ---

## 1. Identify shapes

**Input shape (joined):**
- `e.session_id`, `e.sub_session_id`, `e.name`, `e.type`, `e.action`, `e.url_path`, `e.doc_id`, `e.action_meta` (JSON), `e.created` — from `stg_events_docs_speedy`
- `d.id`, `d.name` — from `stg_documents`

**Target shape:**
- `session_id`, `sub_session_id`, `name`, `type`, `action`, `url_path`, `doc_name`, `doc_id`, `search_term`, `result_count`, `result_num`, `created`

One row per search event, enriched with the document name and a computed `result_num` that reflects which search result the user is currently looking at, accounting for next/prev navigation and wraparound.

## 2. Identify sources and relationships

Two sources:

- **`stg_events_docs_speedy` (aliased `e`)** — the event stream, filtered to `name = 'search'`. Contains JSON metadata (`action_meta`) with `search_term` and `result_count`. The `action` column encodes navigation direction (`next`, `prev`, or the initial search action).
- **`stg_documents` (aliased `d`)** — document dimension table. Joined on `e.doc_id = d.id` to resolve the document name.

The join is a simple foreign-key lookup: `doc_id` → `d.id`. This is a 1:1 enrichment join — it does not change the grain.

## 3. Identify aggregation depth

This query has **no GROUP BY aggregation**. The grain remains one row per search event throughout.

However, there is a **window function** that performs a running sum to track the user's position within search results. The window aggregation shape:

```
[window] session_id, sub_session_id, name, type, search_term, url_path, doc_id | ORDER BY created ASC,
  SUM(CASE next→+1, prev→-1, else→0) as cumulative_offset
```

The cumulative offset is then adjusted (+1 for 1-based indexing) to produce `result_num` in the CTE. The final SELECT applies modular arithmetic to wrap `result_num` within `[1, result_count]`.

No multi-level aggregation — the arrow notation for the single window pass:

```
(no grouping — window) partition_keys, SUM(direction_delta) OVER (... ORDER BY created) + 1 as result_num
-> (no further aggregation — final select applies MOD wrapping)
```

## 4. Identify dependencies

- **`search_term`** and **`result_count`**: derived by extracting from the JSON `action_meta` field. These are scalar extractions, not aggregations.
- **`result_num` (CTE version)**: depends on the running sum of direction deltas (`next` = +1, `prev` = -1) within a partition. The partition key is the combination of `session_id, sub_session_id, name, type, search_term, url_path, doc_id` — this defines a single "search context" where a user is navigating results for a specific term in a specific document. The `CASE WHEN result_count = 0` guard short-circuits to 0 when there are no results (avoids meaningless navigation tracking).
- **`result_num` (final version)**: depends on the CTE's `result_num` and `result_count`. Applies `MOD` to wrap around. The `CASE WHEN result_num < 0` handles negative offsets (more `prev` presses than `next`) by reflecting them into positive modular space before applying `MOD`.
- **`doc_name`**: depends on the join to `stg_documents`.

## 5. Plan CTE layers bottom-up

**CTE 1: `search_events`** — Filters the event stream to search events only, extracts JSON fields, and computes a raw running result position using a cumulative window sum of navigation deltas (+1 for next, -1 for prev), offset by +1 for 1-based indexing. Guards against zero-result searches.

**Final SELECT** — Joins `search_events` to `stg_documents` for the document name, then applies modular arithmetic to wrap `result_num` into the valid range `[1, result_count]`, handling both overflow (past last result) and underflow (before first result).

## 6. Write each CTE with justification

**`search_events`**: This CTE exists to isolate the window function logic. The running sum must be computed before the wraparound modular arithmetic can be applied, because `MOD` is a scalar operation on the accumulated position — it cannot be folded into the window function. Separating the cumulative tracking from the wraparound also keeps each layer's logic readable.

**Final SELECT**: Exists to apply the wrapping logic and join in the document name. The wrapping uses `MOD(adjusted_result_num, result_count) + 1` where the adjustment handles negative values by reflecting them: `result_count - result_num` converts a negative offset into its positive equivalent within the modular space.

## 7. Compose the final query

```sql
WITH search_events AS (
  SELECT
    e.session_id,
    e.sub_session_id,
    e.name,
    e.type,
    e.action,
    e.url_path,
    e.doc_id,
    JSON_EXTRACT_SCALAR(e.action_meta, '$.search_term') as search_term,
    SAFE_CAST(JSON_EXTRACT_SCALAR(e.action_meta, '$.result_count') AS INT64) as result_count,
    CASE WHEN SAFE_CAST(JSON_EXTRACT_SCALAR(e.action_meta, '$.result_count') AS INT64) = 0 THEN 0
    ELSE
      SUM(
        CASE 
          WHEN action = 'next' THEN 1
          WHEN action = 'prev' THEN -1
          ELSE 0
        END
      ) OVER (
        PARTITION BY 
          e.session_id,
          e.sub_session_id,
          e.name,
          e.type,
          -- e.action
          JSON_EXTRACT_SCALAR(e.action_meta, '$.search_term'),
          e.url_path,
          e.doc_id
        ORDER BY
          e.created ASC
      ) + 1
    END AS result_num,
    e.created
  FROM
    ${ref('stg_events_docs_speedy')} as e
  WHERE
    e.name = 'search'
  ORDER BY
    e.session_id ASC,
    e.sub_session_id ASC,
    e.created ASC
)
SELECT
  se.session_id,
  se.sub_session_id,
  se.name,
  se.type,
  se.action,
  se.url_path,
  d.name as doc_name,
  se.doc_id,
  se.search_term,
  se.result_count,
  CASE WHEN se.result_count <= 0 
    THEN 0
    ELSE
        MOD(
          CASE WHEN se.result_num < 0 THEN se.result_count - se.result_num ELSE se.result_num END,
          se.result_count
        ) + 1
  END AS result_num,
  se.created
FROM
  search_events as se
JOIN
  ${ref('stg_documents')} as d
ON
  se.doc_id = d.id
ORDER BY
  se.created ASC
```
