--- USER ---

Give me a full course catalog with all course metadata fields, including which clients each course belongs to. If a course has no client associations, default to client ID 1.

--- AGENT ---

## Step 1: Identify shapes

**Input shape:**
- `courses`: `id, creator_admin_id, updated_admin_id, container_id, name, description, headline, version, published, created, updated, expiration_days, thumbnail, background_image, pricing, has_been_published, vaas_id, archived, level, publish_date, assessment_id, lab_id, is_public, promoted, mailgun_template_id, mailgun_subject, discover_page, facebook_sso_required, cover_image, mailgun_data, course_cover_mp4, course_cover_webm, disable_final_assessment, hide_assessment_score, meta_quest_sso_required, google_sso_required, course_type, show_latest_version, seo`
- `clients_course_containers`: `course_container_id, client_id`

**Target shape:**
All course columns (with certain text fields lowercased), plus a `client_ids` array column — one row per course.

## Step 2: Identify sources and relationships

Two tables are involved:

- `courses` (aliased `c`) — the base entity, one row per course.
- `clients_course_containers` (aliased `ccc`) — a many-to-many association between clients and course containers.

The join relationship is indirect: `courses.container_id` relates to `clients_course_containers.course_container_id`. The `container_id` column on `courses` is the foreign key that links a course to its container, and `clients_course_containers` maps containers to clients.

The join is a LEFT JOIN because not every course necessarily has client associations — the `COALESCE(..., [1])` default confirms this.

## Step 3: Identify aggregation depth

There is one level of aggregation — collapsing the many client rows per container into a single array:

```
container_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

This is a single-level aggregation with no further rollup. Arrow notation:

```
container_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

No second arrow — the aggregated result is joined directly to the course table, producing one row per course.

## Step 4: Identify dependencies

- **Grouping driver:** `course_container_id` in `clients_course_containers` — this is the grain at which client IDs must be collapsed before joining.
- **Derived column:** `client_ids` — the array of distinct client IDs per container.
- **Intermediate step required:** The aggregation must happen *before* the join to `courses`, because `courses` is at one-row-per-course grain. Joining the raw many-to-many table directly would fan out course rows. Pre-aggregating into one row per `container_id` keeps the join 1:1.
- **Default logic:** `COALESCE(cc.client_ids, [1])` handles the LEFT JOIN null case — courses with no container-client mapping get a default array of `[1]`.

## Step 5: Plan CTE layers bottom-up

1. **`course_clients`** — aggregates `clients_course_containers` to one row per `container_id`, producing a `client_ids` array of distinct client IDs.

That is the only CTE. The final SELECT joins `courses` to `course_clients` on `container_id` and applies text normalization (LOWER) on select string fields.

## Step 6: Write each CTE with justification

**`course_clients`**: Pre-aggregates the many-to-many client-container association into one row per container with an array of client IDs, so the subsequent join to `courses` remains 1:1 and does not fan out rows.

```sql
course_clients AS (
    SELECT
        ccc.course_container_id as container_id,
        ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
    FROM
        ${ref("clients_course_containers")} as ccc
    GROUP BY
        ccc.course_container_id
)
```

## Step 7: Compose the final query

```sql
WITH
course_clients AS (
    SELECT
        ccc.course_container_id as container_id,
        ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
    FROM
        ${ref("clients_course_containers")} as ccc
    GROUP BY
        ccc.course_container_id
)
SELECT
  c.id,
  c.creator_admin_id,
  c.updated_admin_id,
  c.container_id,
  c.name,
  LOWER(c.description) AS description,
  LOWER(c.headline) AS headline,
  c.version,
  c.published,
  c.created,
  c.updated,
  c.expiration_days,
  c.thumbnail,
  c.background_image,
  LOWER(c.pricing) AS pricing,
  c.has_been_published,
  c.vaas_id,
  c.archived,
  LOWER(c.level) AS level,
  c.publish_date,
  c.assessment_id,
  c.lab_id,
  c.is_public,
  c.promoted,
  LOWER(c.mailgun_template_id) AS mailgun_template_id,
  LOWER(c.mailgun_subject) AS mailgun_subject,
  c.discover_page,
  c.facebook_sso_required,
  c.cover_image,
  LOWER(c.mailgun_data) AS mailgun_data,
  c.course_cover_mp4,
  c.course_cover_webm,
  c.disable_final_assessment,
  c.hide_assessment_score,
  c.meta_quest_sso_required,
  c.google_sso_required,
  LOWER(c.course_type) AS course_type,
  c.show_latest_version,
  LOWER(c.seo) AS seo,
  COALESCE(cc.client_ids, [1]) AS client_ids
FROM
  ${ref("courses")} as c
LEFT JOIN
    course_clients as cc
ON
    cc.container_id = c.container_id
```

**Target:** One row per course, with all course metadata fields and a `client_ids` array.

- All course columns are present, with text fields (`description`, `headline`, `pricing`, `level`, `mailgun_template_id`, `mailgun_subject`, `mailgun_data`, `course_type`, `seo`) lowercased via `LOWER()`.
- `client_ids` is an array: either the aggregated distinct client IDs from the container mapping, or `[1]` as a default when no mapping exists.
