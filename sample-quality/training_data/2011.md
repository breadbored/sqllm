You are an expert analytical SQL reasoning assistant. When given a data transformation problem, you will reason through it step by step before writing any SQL.

Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table
   - For multiple tables, assume the tables are already joined for now and list all the columns as if they were already joined
     - `table1.X, table1.Y, table1.Z, table2.A, table2.B, table2.C`

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     -> person_id, session_id, SUM(page_time) as session_time
     -> person_id, SUM(session_time) as person_time
     ```
   - When a join depends on a pre-aggregated value, prefix the notation with a bracketed join clause:
     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     -> user_id, SUM(cues_reached) as total_cues_reached, SUM(lesson_progress) as course_progress
     ```

4. **Identify dependencies** - which columns drive grouping, which are derived, which require intermediate steps

5. **Plan CTE layers bottom-up** - name each CTE and describe in one sentence what it produces, using the arrow notation from step 3 as your guide

6. **Write each CTE with justification** - one sentence explaining why this layer exists

7. **Compose the final query**

8. **Verify output shape** - confirm the final output columns match the target shape stated in step 1

Do not skip steps. Do not write SQL before completing the reasoning. If the problem is ambiguous, state your assumptions explicitly before reasoning. Repeat the process from the beginning to refine the pseudocode outline if there are new realizations you arrive at while going down the list. ## Question
For every question in the question registry, return its metadata along with an array of client IDs that have access to it through their course container assignments. Questions that are not reachable from any client's course containers should still be returned, with their client array defaulting to `[1]`.

--- AGENT ---

## Step 1: Identify Shapes

**Input tables (as if pre-joined):**
`clients_course_containers.client_id, clients_course_containers.course_container_id, courses.id, courses.container_id, course_lessons.course_id, course_lessons.lesson_id, learning_cues.id, learning_cues.lesson_id, knowledge_checks.id, knowledge_checks.learning_cue_id, knowledge_check_questions.id, knowledge_check_questions.knowledge_check_id, questions.id, questions.name, questions.type, questions.explanation, questions.attempts, questions.created, questions.updated, questions.image, questions.image_title`

**Target shape:**
`id, name, type, explanation, attempts, created, updated, image, image_title, client_ids`

One row per question. `client_ids` is an array column — a collection of all client IDs that can reach this question through the course hierarchy.

---

## Step 2: Identify Sources and Relationships

There is no direct foreign key between `questions` and `clients_course_containers`. The relationship must be traversed through a 5-hop chain:

1. `clients_course_containers.course_container_id` → `courses.container_id` — links a client's contracted course container to the courses within it
2. `courses.id` → `course_lessons.course_id` — links a course to the lessons it contains
3. `course_lessons.lesson_id` → `learning_cues.lesson_id` — links lessons to their embedded learning cues
4. `learning_cues.id` → `knowledge_checks.learning_cue_id` — links a cue to a knowledge check gating it
5. `knowledge_checks.id` → `knowledge_check_questions.knowledge_check_id` — links a knowledge check to the individual questions it uses

The `questions` table is the canonical question registry, joined via `questions.id = knowledge_check_questions.id` (aliased as `question_id`) in the outer query. The LEFT JOIN in the final step ensures questions that exist in the registry but are not reachable through any client's container are still returned.

---

## Step 3: Identify Aggregation Depth

The join chain fans out: a single question can be reached from multiple clients (through multiple containers, courses, or lessons). This means there are many rows per `question_id` before aggregation. The goal is one row per question with a deduplicated array of client IDs.

**Single aggregation level:**

```
question_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

Only one CTE is needed. The final SELECT is a flat join with no further aggregation.

---

## Step 4: Identify Dependencies

- `client_ids` is a derived array column — it cannot be expressed without first traversing the 5-hop join chain and collapsing the fan-out with `ARRAY_AGG(DISTINCT ...)`.
- The `IGNORE NULLS` modifier exists because the join chain includes `LEFT JOIN`-capable paths where `client_id` could be null if intermediate rows are absent.
- `COALESCE(sq.client_ids, [1])` in the outer query means: if a question has no CTE match (i.e., it is not reachable from any client), default its `client_ids` to the array `[1]`. This implies client `1` is a global or default client.
- `LOWER()` on `type`, `explanation`, and `image_title` is a normalization concern, not a join or aggregation dependency.

---

## Step 5: Plan CTE Layers Bottom-Up

**`stg_questions`** — traverses the 5-hop join chain from `clients_course_containers` through to `knowledge_check_questions`, then collapses the result to one row per question with an array of distinct client IDs.

One CTE. One level of aggregation. The final SELECT is a straight enrich-and-return join against the question registry.

---

## Step 6: Write Each CTE With Justification

**`stg_questions`**: This CTE exists because `questions` and `clients_course_containers` share no direct key — the client-to-question reachability relationship must be assembled by chaining five intermediate tables. Pre-aggregating to `(question_id → client_ids[])` before the outer join prevents the fan-out from duplicating question registry rows in the final result.

---

## Step 7: Compose the Final Query

```sql
WITH
stg_questions AS (
    SELECT
        kcq.id as question_id,
        ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
    FROM
        ${ref("clients_course_containers")} as ccc
    JOIN
        ${ref("courses")} as c
    ON
        c.container_id = ccc.course_container_id
    JOIN
        ${ref("course_lessons")} as cl
    ON
        cl.course_id = c.id
    JOIN
        ${ref("learning_cues")} as lc
    ON
        lc.lesson_id = cl.lesson_id
    JOIN
        ${ref("knowledge_checks")} as kc
    ON
        kc.learning_cue_id = lc.id
    JOIN
        ${ref("knowledge_check_questions")} as kcq
    ON
        kcq.knowledge_check_id = kc.id
    GROUP BY
        kcq.id
)


SELECT
  q.id,
  q.name,
  LOWER(type)        AS type,
  LOWER(explanation) AS explanation,
  q.attempts,
  q.created,
  q.updated,
  q.image,
  LOWER(image_title) AS image_title,
  COALESCE(sq.client_ids, [1]) AS client_ids
FROM 
  ${ref("questions")} as q
LEFT JOIN
  stg_questions as sq
ON
  sq.question_id = q.id
```
