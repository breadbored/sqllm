## Question
For each month, show the total number of unique active users, how many of those were new users (first active in that month), and how many were returning users (active before that month).

--- AGENT ---

## 1. Identify shapes

**Input tables (inferred from query):**
- `agg_transactions`: `user_id`, `created` (timestamp), plus other transaction columns
- `stg_users`: `id`, `created` (timestamp — user registration date), plus other user columns

**Target output shape:**
- `date` (month-truncated)
- `monthly_unique_users` (count distinct)
- `monthly_new_users` (derived: total minus returning)
- `monthly_returning_users` (count distinct)

## 2. Identify sources and relationships

- `agg_transactions.user_id` joins to `stg_users.id` — this is the natural key linking transactions to users
- The user's `created` timestamp from `stg_users` is compared against the transaction's `created` timestamp to determine whether a user is "new" or "returning" in a given month: `datetime_trunc(u.created, MONTH) != datetime_trunc(dt.created, MONTH)`
- Two CTEs produce daily user sets: one for all active users, one for returning-only users. They are left-joined on `created` (date), meaning the join key is the activity date, not the user — this is a date-level join where returning user dates are a subset of total user dates

## 3. Identify aggregation depth

Two levels of aggregation:

```
[total_daily_users]
user_id, date(created), GROUP BY user_id, date(created) -> deduplicated (user_id, created) pairs

[returning_monthly_users]
user_id, date(created), WHERE month(u.created) != month(dt.created), GROUP BY user_id, date(created) -> deduplicated (user_id, created) pairs for returning users only

[final — left join total_daily_users to returning_monthly_users on created date]
datetime_trunc(created, MONTH), COUNT(DISTINCT du.user_id) as monthly_unique_users, COUNT(DISTINCT ru.user_id) as monthly_returning_users, (monthly_unique_users - monthly_returning_users) as monthly_new_users
```

Arrow notation for the aggregation plan:

```
du.user_id, date(du.created) as created -> (dedup grain: one row per user per day, all users)
ru.user_id, date(dt.created) as created -> (dedup grain: one row per user per day, returning only)
[du LEFT JOIN ru ON du.created = ru.created]
datetime_trunc(du.created, MONTH) as date, COUNT(DISTINCT du.user_id) as monthly_unique_users, COUNT(DISTINCT ru.user_id) as monthly_returning_users
```

## 4. Identify dependencies

- **Grouping drivers:** `user_id` and `date(created)` define the grain of both CTEs; `datetime_trunc(created, MONTH)` defines the final output grain
- **Derived columns:** `monthly_new_users` is derived as `monthly_unique_users - monthly_returning_users` — it is not independently aggregated
- **Intermediate steps required:** The returning user filter (`month(u.created) != month(dt.created)`) must be applied before deduplication, so the returning user set is built as a separate CTE rather than a conditional aggregate
- **The left join on date (not user):** This is critical — the join produces a fan-out where each total daily user row gets matched to returning daily user rows for the same date. The `COUNT(DISTINCT)` in the final SELECT then correctly counts unique users across the month from each side of the join

## 5. Plan CTE layers bottom-up

1. **`total_daily_users`** — Deduplicate `agg_transactions` to one row per user per day (all users who had any transaction), joined to `stg_users` to confirm user validity
2. **`returning_monthly_users`** — Same deduplication but filtered to only users whose registration month differs from the transaction month (i.e., they are not new that month)
3. **Final SELECT** — Left join the two daily sets on date, aggregate to monthly grain with `COUNT(DISTINCT)`, and derive new users by subtraction

## 6. Write each CTE with justification

**`total_daily_users`:** Establishes the complete set of active (user, day) pairs. The `GROUP BY user_id, date(created)` deduplicates multiple transactions on the same day by the same user. The inner join to `stg_users` ensures only valid/known users are counted.

**`returning_monthly_users`:** Mirrors `total_daily_users` but adds a WHERE clause filtering to users whose `stg_users.created` month does not match the transaction month. This isolates users who existed before the month they transacted in — i.e., returning users. Same deduplication to (user, day) grain.

**Final SELECT:** Left joins total users to returning users on the activity date. This is a left join because not every active day has returning users (some days may only have new users). The `COUNT(DISTINCT du.user_id)` counts all unique users in the month; `COUNT(DISTINCT ru.user_id)` counts only the returning subset. New users are derived by subtraction, which is valid because the sets are mutually exclusive: a user in a given month is either new (registration month matches) or returning (it doesn't).

## 7. Compose the final query

```sql
WITH
total_daily_users AS (
  SELECT
    dt.user_id,
    date(dt.created) AS created
  FROM
    ${ref("agg_transactions")} AS dt
  INNER JOIN
    ${ref("stg_users")} AS u
    ON
      dt.user_id = u.id
  GROUP BY
    dt.user_id,
    date(dt.created)
),

returning_monthly_users AS (
  SELECT
    dt.user_id,
    date(dt.created) AS created
  FROM
    ${ref("agg_transactions")} AS dt

  INNER JOIN
    ${ref("stg_users")} AS u
    ON
      dt.user_id = u.id
  WHERE
    datetime_trunc(u.created, MONTH) != datetime_trunc(dt.created, MONTH)
  GROUP BY
    dt.user_id,
    date(dt.created)
)

SELECT
  datetime_trunc(du.created, MONTH)                       AS date,
  count(DISTINCT du.user_id)                              AS monthly_unique_users,
  count(DISTINCT du.user_id) - count(DISTINCT ru.user_id) AS monthly_new_users,
  count(DISTINCT ru.user_id)                              AS monthly_returning_users
FROM
  total_daily_users AS du
LEFT JOIN
  returning_monthly_users AS ru
  ON
    du.created = ru.created
GROUP BY
  date
ORDER BY
  date
```

## 8. Verify output shape

Target shape from step 1:
- `date` ✓ — month-truncated timestamp
- `monthly_unique_users` ✓ — count distinct of all active users
- `monthly_new_users` ✓ — derived as total minus returning
- `monthly_returning_users` ✓ — count distinct of returning users

Output grain: one row per calendar month. The `GROUP BY date` (where `date` is `datetime_trunc(created, MONTH)`) guarantees this. The `ORDER BY date` provides chronological ordering. All four columns are present and correctly typed.
