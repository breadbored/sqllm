## Question
For each user and course, produce a daily timeline of learning progress showing daily progress made, cumulative total progress, time spent on course each day, cumulative time on course, and progress percentages relative to total course length.

--- AGENT ---

## 1. Identify shapes

**Input tables (inferred from query):**
- `stg_user_course_lesson_progress`: `user_id, course_id, lesson_id, progress_seconds, interval_seconds, created`
- `stg_users`: `id` (and other user columns — only `id` is used)
- `stg_lessons`: `id, video_id`
- `stg_videos`: `id, duration`
- `stg_course_lessons`: `course_id, lesson_id`

**Target shape:**
```
user_id, course_id, progress_date, daily_progress_seconds, current_progress_seconds,
total_progress_seconds, daily_time_on_course, total_time_on_course, course_length,
progress_percentage_made, total_course_progress_percentage, current_course_progress_percentage
```

One row per user × course × day.

## 2. Identify sources and relationships

- `stg_user_course_lesson_progress` → `stg_users`: join on `u.id = uclp.user_id`. This join filters progress to only valid users (inner join acts as existence check).
- `stg_lessons` → `stg_videos`: join on `l.video_id = v.id`. Gets video duration per lesson.
- `stg_lessons` → `stg_course_lessons`: join on `l.id = cl.lesson_id`. Maps lessons to courses.
- The two major substructures (`course_progress` and `course_length`) are joined on `course_id` at the final SELECT.

The `stg_lessons` → `stg_course_lessons` → `stg_videos` chain is the glue that produces total course duration by summing video durations across all lessons belonging to a course.

## 3. Identify aggregation depth

This query has **four levels of aggregation/transformation**, plus a separate single-level aggregation for course length:

**Progress pipeline:**
```
user_id, course_id, lesson_id, DATE_TRUNC(created, DAY) as progress_date,
  MAX(progress_seconds) as user_total_lesson_progress, SUM(interval_seconds) as time_on_lesson
-> [window: LAG(user_total_lesson_progress) per user_id, course_id, lesson_id order by progress_date
    to compute daily delta as current - previous day's max]
-> user_id, course_id, progress_date,
   SUM(user_total_lesson_progress) as daily_progress_seconds,
   SUM(time_on_lesson) as daily_time_on_course,
   [window: cumulative SUM(daily_progress_seconds) as total_progress_seconds],
   [window: cumulative SUM(daily_progress_seconds) unclamped as current_progress_seconds],
   [window: cumulative SUM(daily_time_on_course) as total_time_on_course]
```

**Course length pipeline (independent):**
```
course_id, SUM(duration) as course_length
```

**Final join:**
```
course_progress joined to course_length on course_id
-> derive percentage columns as progress / course_length, capped at 1.0
```

## 4. Identify dependencies

- **Grouping drivers:** `user_id, course_id, lesson_id, progress_date` at the finest grain; `user_id, course_id, progress_date` at the daily output grain; `course_id` for course length.
- **Derived columns requiring intermediate steps:**
  - `user_total_lesson_progress` (the daily delta) depends on LAG over the pre-aggregated per-day-per-lesson MAX — this is why `pre_lesson_progress` must exist before `lesson_progress`.
  - `total_progress_seconds` and `current_progress_seconds` are cumulative window sums over the daily aggregate — requires the daily SUM to be computed first, then windowed. This is expressed as `SUM(...) OVER(...)` applied to `SUM(...)` — a nested aggregate-then-window pattern.
  - Percentage columns depend on both `course_progress` and `course_length` being fully computed before the final SELECT.

**Key subtlety — the daily delta pattern:**
`progress_seconds` appears to be a high-water-mark (furthest point reached), not an interval. Multiple tracking events per user/lesson/day are collapsed via MAX. The daily *delta* is then `MAX(today) - MAX(yesterday)` using LAG. This converts a cumulative position into a daily increment. The `GREATEST(..., 0)` clamp handles cases where the high-water mark appears to decrease (possibly due to data corrections or replays).

**Second subtlety — `total_progress_seconds` vs `current_progress_seconds`:**
- `total_progress_seconds` = cumulative sum of `GREATEST(daily_delta, 0)` — never decreases, ignores negative deltas.
- `current_progress_seconds` = cumulative sum of raw daily deltas — can decrease if a day's delta is negative.

This distinction lets the downstream percentage calculations show both "best progress ever" and "current actual position."

## 5. Plan CTE layers bottom-up

1. **`pre_lesson_progress`** (nested inside `course_progress`): Aggregates raw progress events to one row per user × course × lesson × day, capturing MAX position and total interval time.

2. **`lesson_progress`** (nested inside `course_progress`): Applies LAG window to compute the daily delta in progress per user × course × lesson, converting cumulative position to daily increment.

3. **`course_progress`** (outer): Aggregates lesson-level daily deltas up to user × course × day, then applies cumulative window sums for running totals of progress and time.

4. **`course_length`**: Joins lessons → videos → course_lessons to compute total video duration per course.

5. **Final SELECT**: Joins `course_progress` to `course_length` on `course_id` and derives percentage columns capped at 1.0.

## 6. Write each CTE with justification

**`pre_lesson_progress`**: Exists because raw progress events have many rows per user/lesson/day. Must collapse to one row per user × course × lesson × day to establish a daily high-water mark (`MAX(progress_seconds)`) and daily time spent (`SUM(interval_seconds)`). The join to `stg_users` filters to valid users only.

**`lesson_progress`**: Exists because `progress_seconds` is a cumulative position, not a daily increment. LAG over the daily high-water mark computes `today's max - yesterday's max` to produce the actual progress made on each day. `GREATEST(COALESCE(LAG(...), 0), 0)` handles the first day (no previous row → COALESCE to 0) and ensures the subtracted value is never negative.

**`course_progress`** (the wrapping SELECT): Exists to collapse lesson-level daily deltas into course-level daily totals (`SUM` across lessons), then compute running cumulative totals via window functions. The nested `SUM(SUM(...)) OVER(...)` pattern works because the inner SUM is the GROUP BY aggregate and the outer SUM is the window function — BigQuery evaluates aggregates before window functions in the same SELECT.

**`course_length`**: Exists to compute the total duration of each course by summing video durations across all lessons assigned to the course. LEFT JOINs handle lessons without videos or without course assignments gracefully.

## 7. Compose the final query

```sql
WITH
  course_progress AS (
  WITH
    pre_lesson_progress AS (
    SELECT
      user_id,
      course_id,
      lesson_id,
      MAX(progress_seconds) AS user_total_lesson_progress,
      SUM(interval_seconds) AS time_on_lesson,
      DATE_TRUNC(uclp.created, DAY) AS progress_date,
    FROM
      ${ref("stg_user_course_lesson_progress")} AS uclp
    JOIN
      ${ref("stg_users")} u
    ON
      u.id = uclp.user_id
    GROUP BY
      user_id,
      course_id,
      lesson_id,
      progress_date
    ORDER BY
      user_id,
      course_id,
      lesson_id,
      progress_date),
    lesson_progress AS (
    SELECT
      lp.user_id,
      lp.course_id,
      lp.lesson_id,
      lp.user_total_lesson_progress - (GREATEST( COALESCE( LAG( lp.user_total_lesson_progress) OVER (PARTITION BY lp.user_id, lp.course_id, lp.lesson_id ORDER BY lp.progress_date ASC), 0), 0)) AS user_total_lesson_progress,
      lp.user_total_lesson_progress AS old_user_total_lesson_progress,
      lp.time_on_lesson,
      lp.progress_date,
    FROM
      pre_lesson_progress AS lp)
  SELECT
    lp.user_id AS user_id,
    lp.course_id AS course_id,
    lp.progress_date AS progress_date,
    SUM(lp.user_total_lesson_progress) AS daily_progress_seconds,
    SUM( GREATEST(SUM(lp.user_total_lesson_progress), 0)) OVER (PARTITION BY lp.user_id, lp.course_id ORDER BY lp.progress_date ASC) AS total_progress_seconds,
    SUM( SUM(lp.user_total_lesson_progress)) OVER (PARTITION BY lp.user_id, lp.course_id ORDER BY lp.progress_date ASC) AS current_progress_seconds,
    SUM(time_on_lesson) AS daily_time_on_course,
    SUM(SUM(time_on_lesson)) OVER (PARTITION BY user_id, course_id ORDER BY lp.progress_date) AS total_time_on_course
  FROM
    lesson_progress AS lp
  GROUP BY
    lp.user_id,
    lp.course_id,
    lp.progress_date
  ORDER BY
    lp.user_id,
    lp.course_id,
    lp.progress_date),
  course_length AS (
  SELECT
    cl.course_id,
    ROUND(SUM(v.duration), 3) AS course_length
  FROM
    ${ref("stg_lessons")} AS l
  LEFT JOIN
    ${ref("stg_videos")} AS v
  ON
    l.video_id = v.id
  LEFT JOIN
    ${ref("stg_course_lessons")} AS cl
  ON
    l.id = cl.lesson_id
  GROUP BY
    course_id
  ORDER BY
    course_id)
SELECT
  cp.user_id,
  cp.course_id,
  cp.progress_date,
  cp.daily_progress_seconds,
  cp.current_progress_seconds,
  cp.total_progress_seconds,
  cp.daily_time_on_course,
  cp.total_time_on_course,
  CAST(FORMAT('%g', cd.course_length) AS FLOAT64) AS course_length,
  CAST(FORMAT('%g', LEAST(ROUND((cp.daily_progress_seconds / cd.course_length), 2), 1)) AS FLOAT64) AS progress_percentage_made,
  CAST(FORMAT('%g', LEAST(ROUND((cp.total_progress_seconds / cd.course_length), 2), 1)) AS FLOAT64) AS total_course_progress_percentage,
  CAST(FORMAT('%g', LEAST(ROUND((GREATEST(cp.current_progress_seconds, 0) / cd.course_length), 2), 1)) AS FLOAT64) AS current_course_progress_percentage
FROM
  course_progress cp
JOIN
  course_length cd
ON
  cd.course_id = cp.course_id
```
