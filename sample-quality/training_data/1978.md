## Question
Give me a list of all course containers with their basic metadata (id, creator, timestamps), and include an array of associated client IDs for each container. If a container has no associated clients, default to client ID 1.

--- AGENT ---

## 1. Identify shapes

**Input shape:**
- `course_container`: `id, creator_admin_id, created, updated`
- `clients_course_containers`: `course_container_id, client_id`

**Target shape:**
- `id, creator_admin_id, created, updated, client_ids`

One row per course container, with `client_ids` as an array column.

## 2. Identify sources and relationships

Two tables are involved:

- `course_container` — the primary entity table, one row per container.
- `clients_course_containers` — a many-to-many bridge table linking `client_id` to `course_container_id`.

The join relationship is `clients_course_containers.course_container_id = course_container.id`.

The join is a LEFT JOIN because every course container should appear in the output regardless of whether it has any client associations.

## 3. Identify aggregation depth

The bridge table `clients_course_containers` has potentially many rows per `course_container_id`. These must be collapsed into a single array before joining to `course_container`, otherwise the join would fan out the container rows.

Arrow notation:

```
course_container_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

Single-level aggregation. The array aggregation happens at the `course_container_id` grain, producing one row per container. No further aggregation is needed after the join — the final SELECT is a simple 1:1 projection.

## 4. Identify dependencies

- `client_ids` is derived from `clients_course_containers` and must be pre-aggregated to the container grain before the join.
- `id, creator_admin_id, created, updated` come directly from `course_container` with no transformation.
- `COALESCE(_c.client_ids, [1])` handles the LEFT JOIN null case — containers with no client associations get a default array of `[1]`.

## 5. Plan CTE layers bottom-up

1. **`course_clients`** — Aggregates the bridge table to one row per `course_container_id`, producing a distinct array of `client_id` values. This prevents fan-out when joined to the container table.

Final SELECT joins `course_container` to `course_clients` and applies the COALESCE default.

## 6. Write each CTE with justification

**`course_clients`**: Pre-aggregates the many-to-many bridge table so that each container maps to exactly one row with an array of its client IDs. Without this, joining `clients_course_containers` directly to `course_container` would duplicate container rows — one per client association.

`DISTINCT` inside `ARRAY_AGG` deduplicates client IDs. `IGNORE NULLS` excludes any null `client_id` values from the array.

## 7. Compose the final query

```sql
WITH
course_clients AS (
  SELECT
    ccc.course_container_id as container_id,
    ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("clients_course_containers")} as ccc
  GROUP BY
    ccc.course_container_id
)
SELECT
  id,
  creator_admin_id,
  created,
  updated,
  COALESCE(_c.client_ids, [1]) AS client_ids
FROM
  ${ref("course_container")} as cc
LEFT JOIN
  course_clients as _c
ON
  _c.container_id = cc.id
```

## 8. Verify output shape

Target shape: `id, creator_admin_id, created, updated, client_ids`

Final SELECT produces: `id, creator_admin_id, created, updated, client_ids` ✓

Grain: one row per `course_container.id`. The LEFT JOIN to `course_clients` is 1:1 (pre-aggregated CTE guarantees one row per container). No fan-out, no collapse. ✓

NULL handling: containers with no client associations get `client_ids = [1]` via COALESCE. ✓
