## Question
For each month, show the total number of unique users, how many were new users (first appeared that month), and how many were returning users (had activity in a prior month).

--- AGENT ---

## 1. Identify shapes

**Input shape (joined columns):**
- `agg_transactions.user_id`, `agg_transactions.created`
- `stg_users.id`, `stg_users.created`

**Target shape:**
- `date` (month), `monthly_unique_users`, `monthly_new_users`, `monthly_returning_users`

One row per month. Three distinct-count metrics segmenting users by new vs. returning.

## 2. Identify sources and relationships

Two source tables:

- `agg_transactions` (aliased `dt`) — transaction-level data with `user_id` and `created` timestamp. Likely multiple rows per user per day.
- `stg_users` (aliased `u`) — user dimension with `id` and `created` (account creation timestamp).

**Join relationship:** `dt.user_id = u.id` — the transaction's user foreign key joins to the user's primary key.

The user table's `created` field is critical: it determines whether a user is "new" or "returning" in a given month. A user is returning if `datetime_trunc(u.created, MONTH) != datetime_trunc(dt.created, MONTH)` — i.e., their account was created in a different month than the transaction month. A user is new if these months match.

## 3. Identify aggregation depth

Two CTE layers flatten transactions to daily user grain, then the final SELECT aggregates daily grain to monthly grain with distinct counts.

Arrow notation:

```
CTE total_daily_users:
  user_id, date(created), GROUP BY user_id, date(created) -> user_id, created (one row per user per day)

CTE returning_monthly_users:
  user_id, date(created), WHERE user_signup_month != transaction_month, GROUP BY user_id, date(created) -> user_id, created (one row per returning user per day)

Final:
  [total_daily_users LEFT JOIN returning_monthly_users ON created date]
  datetime_trunc(created, MONTH) AS date, COUNT(DISTINCT du.user_id) AS monthly_unique_users, COUNT(DISTINCT ru.user_id) AS monthly_returning_users, difference AS monthly_new_users
```

Expressed compactly:

```
dt.user_id, date(dt.created) -> (daily user grain, all users)
dt.user_id, date(dt.created) [WHERE signup_month != txn_month] -> (daily user grain, returning only)
[daily_all LEFT JOIN daily_returning ON created]
datetime_trunc(du.created, MONTH), COUNT(DISTINCT du.user_id) as monthly_unique_users, COUNT(DISTINCT ru.user_id) as monthly_returning_users, monthly_unique_users - monthly_returning_users as monthly_new_users
```

## 4. Identify dependencies

- **Grouping driver:** `datetime_trunc(created, MONTH)` in the final SELECT determines the output grain (one row per month).
- **Derived columns:**
  - `monthly_unique_users` — distinct count of all users with transactions that month.
  - `monthly_returning_users` — distinct count of users whose signup month differs from the transaction month.
  - `monthly_new_users` — arithmetic difference: total minus returning. This avoids a separate new-user CTE.
- **Intermediate dependency:** Both CTEs must deduplicate transactions down to user-day pairs first. Without this, the LEFT JOIN between two transaction-level tables would produce a Cartesian explosion on days where a user has multiple transactions.
- **The LEFT JOIN condition** is on `created` (the date), not on `user_id`. This is intentional — `returning_monthly_users` is a strict subset of `total_daily_users` (same user_id, same date, just filtered). The join on date alone means `ru.user_id` will be non-null for any returning user row that shares that date. The `COUNT(DISTINCT)` then correctly counts unique returning users per month.

**Important subtlety:** The LEFT JOIN is on `du.created = ru.created` (date only, no user_id). This works because:
- Both CTEs produce (user_id, date) pairs.
- The join on date alone creates a fan-out: each total-daily-user row for a given date joins to *every* returning-daily-user row for that same date.
- But since the final aggregation uses `COUNT(DISTINCT)`, the fan-out is collapsed — each distinct `du.user_id` is counted once for total, and each distinct `ru.user_id` is counted once for returning, regardless of how many join matches occurred.
- This is a deliberate pattern: use a loose join to bring two populations into the same row space, then rely on `COUNT(DISTINCT)` to neutralize the Cartesian effect.

## 5. Plan CTE layers bottom-up

1. **`total_daily_users`** — Deduplicates `agg_transactions` to one row per user per day (all users who had any transaction), filtered to valid users via inner join to `stg_users`.

2. **`returning_monthly_users`** — Same deduplication as above, but filtered to only users whose account creation month differs from the transaction month (returning users only).

3. **Final SELECT** — Left joins the two daily-grain CTEs on date, then aggregates to monthly grain using `COUNT(DISTINCT)` for total and returning users, and subtraction for new users.

## 6. Write each CTE with justification

**`total_daily_users`:** Collapses potentially many transaction rows per user per day into a single (user_id, date) pair. This is necessary to prevent row multiplication in the downstream join. The inner join to `stg_users` ensures only valid/existing users are included.

**`returning_monthly_users`:** Same structure as `total_daily_users` but adds a WHERE filter: `datetime_trunc(u.created, MONTH) != datetime_trunc(dt.created, MONTH)`. This isolates users whose signup month is different from their transaction month — i.e., they are not new in that month. The deduplication to daily grain serves the same anti-Cartesian purpose.

**Final SELECT:** Groups by month (`datetime_trunc(du.created, MONTH)`), counts distinct users from each population, and derives new users by subtraction rather than requiring a third CTE.

## 7. Compose the final query

```sql
WITH
total_daily_users AS (
  SELECT
    dt.user_id,
    date(dt.created) AS created
  FROM
    ${ref("agg_transactions")} AS dt
  INNER JOIN
    ${ref("stg_users")} AS u
    ON
      dt.user_id = u.id
  GROUP BY
    dt.user_id,
    date(dt.created)
),

returning_monthly_users AS (
  SELECT
    dt.user_id,
    date(dt.created) AS created
  FROM
    ${ref("agg_transactions")} AS dt

  INNER JOIN
    ${ref("stg_users")} AS u
    ON
      dt.user_id = u.id
  WHERE
    datetime_trunc(u.created, MONTH) != datetime_trunc(dt.created, MONTH)
  GROUP BY
    dt.user_id,
    date(dt.created)
)

SELECT
  datetime_trunc(du.created, MONTH)                       AS date,
  count(DISTINCT du.user_id)                              AS monthly_unique_users,
  count(DISTINCT du.user_id) - count(DISTINCT ru.user_id) AS monthly_new_users,
  count(DISTINCT ru.user_id)                              AS monthly_returning_users
FROM
  total_daily_users AS du
LEFT JOIN
  returning_monthly_users AS ru
  ON
    du.created = ru.created
GROUP BY
  date
ORDER BY
  date
```
