## 2. Identify sources and relationships

Six tables are involved:

| Table | Role | Join Key |
|---|---|---|
| `stg_user_knowledge_check_answers` (ukca) | Fact table — one row per answer submission | Central table |
| `stg_question_sets` (qs) | Maps questions to their correct option | `qs.question_id = ukca.question_id` |
| `stg_questions` (q) | Question names/text | `q.id = qs.question_id` |
| `stg_question_options` (ua) | Option names — used for the user's answer | `ua.id = ukca.answer_option_id` |
| `stg_question_options` (qo) | Option names — used for the correct answer | `qo.id = qs.correct_option_id` |
| `stg_courses` (c) | Course names and container | `c.id = ukca.course_id` |
| `stg_users` (u) | User metadata — used only for filtering test users | `u.id = ukca.user_id` |

Key observations:
- `stg_question_options` is joined **twice** under different aliases (`ua` and `qo`) — once to resolve the user's chosen answer text, once to resolve the correct answer text.
- `stg_users` is a **LEFT JOIN** used purely for filtering (`test_user = 0`). The LEFT JOIN combined with `user_id IS NOT NULL` ensures we keep only rows that have a user_id and where that user is not a test user.
- The correct option is resolved indirectly: `ukca` → `qs` (via question_id) → `qo` (via correct_option_id). The `qs` table acts as a glue between the question and its correct option.

## 3. Identify aggregation depth

There is **no aggregation** in this query. It is a pure join-and-filter query that produces one output row per input row from the fact table.

Single-level shape in arrow notation:


No `->` transitions — no GROUP BY, no aggregate functions.

## 4. Identify dependencies

- **Driving table:** `ukca` — every output row corresponds to one row in this fact table
- **Dimension lookups (INNER JOIN):** `qs`, `q`, `ua`, `qo`, `c` — all INNER JOINs, meaning rows without matching dimension records are excluded
- **Filter dependency:** `u` is LEFT JOINed but used in the WHERE clause to exclude test users. The `user_id IS NOT NULL` guard handles the case where `ukca.user_id` is null (those rows are dropped). The `u.test_user = 0` condition further restricts to real users — rows where the LEFT JOIN finds no match (u.test_user would be NULL) are also excluded, effectively making this behave like an INNER JOIN for non-test users
- **Indirect lookup chain:** The correct answer text requires two hops: `ukca.question_id → qs.correct_option_id → qo.name`

## 7. Compose the final query

```sql
SELECT
  ukca.user_id,
  ukca.course_id,
  c.name AS course_name,
  c.container_id,
  ukca.lesson_id,
  ukca.question_id,
  q.name AS question,
  ukca.answer_option_id,
  ua.name AS user_answer,
  ukca.correct,
  ukca.created AS date_answered,
  qs.correct_option_id,
  qo.name AS correct_answer
FROM
  ${ref("stg_user_knowledge_check_answers")} ukca
JOIN
  ${ref("stg_question_sets")} qs
ON
  qs.question_id = ukca.question_id
JOIN
  ${ref("stg_questions")} q
ON
  q.id = qs.question_id
JOIN
  ${ref("stg_question_options")} ua
ON
  ua.id = ukca.answer_option_id
JOIN
  ${ref("stg_question_options")} qo
ON
  qo.id = qs.correct_option_id
JOIN
  ${ref("stg_courses")} c
ON
  c.id = ukca.course_id
LEFT JOIN
  ${ref("stg_users")} u
ON
  u.id = ukca.user_id
WHERE
  user_id IS NOT NULL
  AND u.test_user = 0
```
