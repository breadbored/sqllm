--- USER ---

For each client and day, give me the list of active users, broken down into new users (first time participating with that client) and returning users (participated with that client on a previous day). Include the client name.

--- AGENT ---

## 1. Identify shapes

**Input shape (joined):**
- `agg_client_transactions`: `user_id, day, client_id` (where `client_id` is an array that must be unnested)
- `stg_clients`: `id, name`

**Target shape:**
```
day, client_id, client_name, active_users[], new_users[], returning_users[]
```

One row per day per client, with three user-id arrays classifying each active user as new or returning.

## 2. Identify sources and relationships

- `agg_client_transactions.client_id` is an array column — each row can relate to multiple clients, so it must be unnested with `UNNEST(ct.client_id) AS cid` to produce one row per user × day × client.
- `stg_clients.id` joins to the unnested `cid` value to resolve client names.
- To classify users as new vs returning, we need each user's first day per client. This is derived from `agg_client_transactions` itself: `MIN(day)` grouped by `user_id, client_id`. That first-day lookup joins back to the main table on `(user_id, client_id)`.

Join chain:
```
agg_client_transactions (unnested) 
  → client_transactions_min ON (user_id, client_id)   -- for new/returning classification
  → stg_clients ON (client_id = id)                    -- for client name
```

## 3. Identify aggregation depth

Two levels of aggregation:

```
user_id, cid, MIN(day) as first_day
-> day, cid, client_name, ARRAY_AGG(DISTINCT user_id) as active_users,
                          ARRAY_AGG(DISTINCT CASE new) as new_users,
                          ARRAY_AGG(DISTINCT CASE returning) as returning_users
```

Level 1 (CTE): Pre-aggregate to find each user's first day per client — this must be computed before the main query can classify users.

Level 2 (final SELECT): Group by `(day, client_id, client_name)` and collect user arrays with conditional CASE logic driven by the first-day lookup.

## 4. Identify dependencies

- **Grouping columns (final):** `day`, `cid`, `sc.name`
- **Derived (CTE):** `first_day` — `MIN(day)` per `(user_id, client_id)` — required before the final SELECT can evaluate the CASE expressions
- **Derived (final):** `active_users`, `new_users`, `returning_users` — all are `ARRAY_AGG(DISTINCT ...)` over `user_id`, distinguished by comparing `first_day` to `ct.day`
- The CASE classification depends on `ctm.first_day` being available at the row level, which requires the CTE to be joined in before final aggregation

## 5. Plan CTE layers bottom-up

1. **`client_transactions_min`** — Computes each user's earliest participation day per client by unnesting the client_id array and grouping by `(user_id, client_id)` with `MIN(day)`.

2. **Final SELECT** — Unnests the client_id array again on the main table, joins to the first-day CTE and the clients dimension, then groups by `(day, client_id, client_name)` to produce the three user classification arrays.

## 6. Write each CTE with justification

**`client_transactions_min`:** This CTE exists because the new/returning classification requires knowing each user's first day per client *before* the final aggregation. Without pre-computing `first_day`, there is no row-level value to compare against `ct.day` in the CASE expressions.

```sql
SELECT
  ct.user_id,
  cid AS client_id,
  MIN(ct.day) AS first_day
FROM agg_client_transactions AS ct,
  UNNEST(ct.client_id) AS cid
GROUP BY ct.user_id, cid
```

**Final SELECT:** Groups by `(day, client_id, client_name)` and uses conditional `ARRAY_AGG(DISTINCT ... IGNORE NULLS)` to split users into three buckets — all active, new (first_day equals current day), and returning (first_day before current day).

## 7. Compose the final query

```sql
WITH
  client_transactions_min AS (
    -- Get the first day each user participated with each client
  SELECT
    ct.user_id,
    cid AS client_id,
    MIN(ct.day) AS first_day
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS cid
    -- Flatten client_id array
  GROUP BY
    ct.user_id,
    cid)
SELECT
  ct.day,
  cid AS client_id,
  sc.name AS client_name,
  ARRAY_AGG(DISTINCT ct.user_id) AS active_users,
  ARRAY_AGG(DISTINCT
    CASE
      WHEN ctm.first_day = ct.day THEN ct.user_id
  END
    IGNORE NULLS) AS new_users,
  ARRAY_AGG(DISTINCT
    CASE
      WHEN ctm.first_day < ct.day THEN ct.user_id
  END
    IGNORE NULLS) AS returning_users
FROM
  ${ref("agg_client_transactions")} AS ct
LEFT JOIN
  UNNEST(ct.client_id) AS cid
LEFT JOIN
  client_transactions_min AS ctm
ON
  ct.user_id = ctm.user_id
  AND cid = ctm.client_id
LEFT JOIN
  ${ref("stg_clients")} AS sc
ON
  cid = sc.id
GROUP BY
  ct.day,
  cid,
  sc.name
```

## 8. Verify output shape

Target shape: `day, client_id, client_name, active_users[], new_users[], returning_users[]`

Final SELECT produces:
- `ct.day` ✓
- `cid AS client_id` ✓
- `sc.name AS client_name` ✓
- `ARRAY_AGG(DISTINCT ct.user_id) AS active_users` ✓
- `ARRAY_AGG(DISTINCT CASE ... END IGNORE NULLS) AS new_users` ✓
- `ARRAY_AGG(DISTINCT CASE ... END IGNORE NULLS) AS returning_users` ✓

One row per `(day, client_id, client_name)`. Output shape matches target. The three arrays are mutually exclusive by construction: a user's `first_day` is either equal to `ct.day` (new) or before it (returning), never both for the same day. The `active_users` array is the union of both. `IGNORE NULLS` ensures the CASE-miss rows (where the condition doesn't match) don't pollute the arrays with NULL entries.
