## Question
Give me a list of user badge completions enriched with the client and badge metadata. I need the user, learning path, badge name, completion date, whether they completed it, the client ID and name, and whether it's a legacy badge.

--- AGENT ---

## 1. Identify shapes

**Input shape:**
- `agg_user_badge_completion`: `user_id, learning_path_id, badge_name, completed_date, did_complete`
- `agg_client_badges`: `badge_id, client_id, client_name, legacy_badge`

**Target shape:**
`user_id, learning_path_id, badge_name, date, did_complete, client_id, client_name, legacy_badge`

This is a row-level enrichment — every row from the user badge completion table is decorated with client-level metadata from the client badges table. No aggregation, no filtering, no reshaping beyond the join itself.

## 2. Identify sources and relationships

Two tables:

- `agg_user_badge_completion` (aliased `ubc`) — one row per user per badge/learning path, carrying completion status and date.
- `agg_client_badges` (aliased `cb`) — one row per badge per client, carrying the client identity and a legacy flag.

**Join key:** `cb.badge_id = ubc.learning_path_id`. The `learning_path_id` in the user completion table maps to `badge_id` in the client badges table. This is the sole join condition — a standard foreign-key lookup.

**Cardinality note:** If a `badge_id` appears for multiple clients in `agg_client_badges`, the join will fan out user completion rows accordingly. The query accepts this — it wants every client association for every user completion.

## 3. Identify aggregation depth

There is no aggregation in this query. The output is row-level from the join.

Arrow notation for the single-level pass-through shape:

```
user_id, learning_path_id, badge_name, completed_date, did_complete, client_id, client_name, legacy_badge
```

No `->` — zero aggregation levels, zero CTEs required.

## 4. Identify dependencies

- All output columns are direct column references; none are derived or computed.
- `completed_date` is aliased to `date` in the SELECT — a cosmetic rename only.
- `client_id`, `client_name`, and `legacy_badge` come exclusively from the `cb` side of the join.
- `user_id`, `learning_path_id`, `badge_name`, `completed_date`, and `did_complete` come exclusively from `ubc`.
- No intermediate steps are needed.

## 5. Plan CTE layers bottom-up

No CTEs required. The query is a single SELECT over a two-table inner join.

## 6. Write each CTE with justification

Not applicable — no CTEs.

## 7. Compose the final query

```sql
SELECT
  user_id,
  learning_path_id,
  badge_name,
  completed_date AS date,
  did_complete,
  cb.client_id,
  cb.client_name,
  legacy_badge
FROM
  ${ref("agg_user_badge_completion")} ubc
JOIN
  ${ref("agg_client_badges")} cb
ON
  cb.badge_id = ubc.learning_path_id
```
