--- USER ---

For a specific client (Meta Quest, client_id = 1), produce a monthly summary of user activity: total unique users, new vs returning users (at both daily and monthly granularity), session counts, sessions per user, and a count of users who appear as both new and returning within the same month. Each row represents one month.

--- AGENT ---

## 1. Identify shapes

**Input shape (observable from query):**

- `agg_client_transactions`: `user_id`, `client_id` (ARRAY of INT), `day` (DATE or TIMESTAMP — used as a date column), plus implicit session-level rows (one row per user per day or per session)
- The same table is used twice — once to derive first-seen dates, once as the fact table for the final aggregation.

**Target shape (one row per month):**

`unique_users, new_users, returning_users, new_users_returning, monthly_new_users, monthly_returning_users, total_sessions, returning_user_sessions, sessions_per_user, year_month, month`

## 2. Identify sources and relationships

Two logical data sources, both derived from the same physical table `agg_client_transactions`:

1. **Fact rows** — `agg_client_transactions AS ct`, filtered to rows where `1 IN UNNEST(ct.client_id)`. Each row is a session/transaction event. The `client_id` column is an ARRAY, so the filter uses `UNNEST` in a `WHERE` clause rather than equality.

2. **First-seen date per user** — derived through two CTEs:
   - `client_transactions_min`: UNNEST the `client_id` array via a cross join, then `GROUP BY user_id, cid` with `MIN(day)` to get each user's earliest transaction date per client.
   - `meta_quest_min`: filters `client_transactions_min` to `client_id = 1`, yielding one row per user with their `min_day` for Meta Quest.

**Join relationship:** `ct.user_id = mqm.user_id` — INNER JOIN, so only users who have a first-seen record for client 1 appear. Since `meta_quest_min` is guaranteed to have one row per `user_id` (it's grouped on `user_id, client_id` and filtered to a single `client_id`), this join does not duplicate fact rows.

## 3. Identify aggregation depth

Two levels of aggregation, but the first is a pre-aggregation in a CTE (not a stacked rollup):

```
[CTE: client_transactions_min]
user_id, cid, MIN(day) as min_day

[CTE: meta_quest_min — filter only, no aggregation]
-> filter to client_id = 1

[Final query: fact rows joined to first-seen date]
year_month, month,
  COUNT(DISTINCT user_id) as unique_users,
  COUNT(DISTINCT CASE WHEN <new_day>) as new_users,
  COUNT(DISTINCT CASE WHEN <returning_day>) as returning_users,
  <new_users> + <returning_users> - unique_users as new_users_returning,
  COUNT(DISTINCT CASE WHEN <new_month>) as monthly_new_users,
  COUNT(DISTINCT CASE WHEN <returning_month>) as monthly_returning_users,
  COUNT(user_id) as total_sessions,
  COUNT(CASE WHEN <returning_day>) as returning_user_sessions,
  ROUND(total_sessions / unique_users, 2) as sessions_per_user
```

Arrow notation for the CTE pre-aggregation:

```
user_id, cid, MIN(day) as min_day -> (filter cid=1) -> joined to fact rows, grouped by month with COUNT DISTINCT conditional aggregates
```

## 4. Identify dependencies

- **Grouping drivers:** `format_date('%b %y', date_trunc(DAY, MONTH))` and `date_trunc(DAY, MONTH)` — these define the monthly grain of the output. `DAY` here is a *column name*, not a date part keyword. `date_trunc(DAY, MONTH)` truncates the `DAY` column to month granularity.

- **Derived columns — new/returning logic:** The query compares `date_trunc(mqm.min_day, DAY)` (the user's first-seen date, truncated to day precision) against `date_trunc(DAY, DAY)` (the current transaction's `DAY` column, truncated to day precision). If they match, the user is "new" on that day. If they don't match, the user is "returning."

- **The `new_users_returning` metric:** This is `(returning_users + new_users) - unique_users`. A user who appears as both new (on their first day in a month) and returning (on subsequent days in the same month) will be counted in *both* `new_users` and `returning_users` but only once in `unique_users`. The difference captures exactly these overlap users.

- **Monthly new/returning:** Same logic but at `MONTH` granularity — `date_trunc(mqm.min_day, MONTH) = date_trunc(DAY, MONTH)` checks if the user's first-ever month matches the transaction's month.

- **Intermediate dependency:** `meta_quest_min` must be computed before the final join. It depends on `client_transactions_min`.

## 5. Plan CTE layers bottom-up

1. **`client_transactions_min`** — Unnests the `client_id` array and finds the earliest `day` per user per client across all transactions.

2. **`meta_quest_min`** — Filters to client_id = 1 only, producing one row per user with their first-ever transaction date for Meta Quest.

3. **Final SELECT** — Joins the filtered fact rows (client_id 1 transactions) to first-seen dates, groups by month, and computes new/returning user counts at both daily and monthly granularity plus session metrics.

## 6. Write each CTE with justification

**`client_transactions_min`:** Exists because `client_id` is stored as an ARRAY, so we must UNNEST before we can GROUP BY user + client to find the minimum day. This is the only way to derive a per-client first-seen date from an array-valued column.

**`meta_quest_min`:** Exists as a filtering step to isolate client_id = 1. Could be inlined as a WHERE clause in the first CTE, but separating it makes the intent explicit and the CTE reusable if other client IDs were needed.

**Final query:** Exists to compute the monthly aggregated metrics. The INNER JOIN to `meta_quest_min` enriches each transaction row with the user's first-seen date, enabling the CASE WHEN comparisons that classify users as new or returning.

## 7. Compose the final query

```sql
WITH
client_transactions_min AS (
  SELECT
    ct.user_id,
    cid           AS client_id,
    min(ct.`day`) AS min_day
  FROM
     ${ref("agg_client_transactions")} AS ct,
    unnest(ct.client_id) AS cid
  GROUP BY
    ct.user_id,
    cid
),

meta_quest_min AS (
  SELECT
    ctm.user_id,
    ctm.client_id,
    ctm.min_day
  FROM
    client_transactions_min AS ctm
  WHERE
    ctm.client_id = 1
)

SELECT
  count(DISTINCT ct.user_id)                               AS unique_users,
  count(
    DISTINCT
    CASE
      WHEN date_trunc(mqm.min_day, DAY) = date_trunc(DAY, DAY) THEN ct.user_id
    END
  )                                                        AS new_users,
  count(
    DISTINCT
    CASE
      WHEN date_trunc(mqm.min_day, DAY) != date_trunc(DAY, DAY) THEN ct.user_id
    END
  )                                                        AS returning_users,
  (count(
    DISTINCT
    CASE
      WHEN date_trunc(mqm.min_day, DAY) != date_trunc(DAY, DAY) THEN ct.user_id
    END
  ) + count(
    DISTINCT
    CASE
      WHEN date_trunc(mqm.min_day, DAY) = date_trunc(DAY, DAY) THEN ct.user_id
    END
  )) - count(DISTINCT ct.user_id)                          AS new_users_returning,
  count(
    DISTINCT
    CASE
      WHEN date_trunc(mqm.min_day, MONTH) = date_trunc(DAY, MONTH) THEN ct.user_id
    END
  )                                                        AS monthly_new_users,
  count(
    DISTINCT
    CASE
      WHEN date_trunc(mqm.min_day, MONTH) != date_trunc(DAY, MONTH) THEN ct.user_id
    END
  )                                                        AS monthly_returning_users,
  count(ct.user_id)                                        AS total_sessions,
  count(
    CASE
      WHEN date_trunc(mqm.min_day, DAY) != date_trunc(DAY, DAY) THEN ct.user_id
    END
  )                                                        AS returning_user_sessions,
  round(count(ct.user_id) / count(DISTINCT ct.user_id), 2) AS sessions_per_user,
  format_date('%b %y', date_trunc(DAY, MONTH))             AS year_month,
  date_trunc(DAY, MONTH)                                   AS month
FROM
   ${ref("agg_client_transactions")} AS ct
INNER JOIN
  meta_quest_min AS mqm
  ON
    ct.user_id = mqm.user_id
WHERE
  1 IN unnest(ct.client_id)
GROUP BY
  format_date('%b %y', date_trunc(DAY, MONTH)),
  date_trunc(DAY, MONTH)
ORDER BY
  month
```
