## Question
Build a document-course mapping that includes the merged set of client IDs associated with each record, combining clients linked through the document and clients linked through the course container. If only one side has client associations, use that side's clients (defaulting to client ID 1 if neither side has any).

--- AGENT ---

## Step 1: Identify shapes

**Input tables:**
- `documents_clients` (dc): `doc_id`, `client_id`
- `clients_course_containers` (ccc): `course_container_id`, `client_id`
- `documents_courses` (dco): `id`, `document_id`, `created`, `updated`, `order`, `course_container_id`

**Target shape:**
`id, document_id, created, updated, order, course_container_id, client_ids`

One row per `documents_courses` record, enriched with a merged array of distinct client IDs drawn from two independent sources.

## Step 2: Identify sources and relationships

Three tables feed the output, with `documents_courses` as the spine:

- `documents_clients.doc_id` → `documents_courses.document_id` — clients associated with the document
- `clients_course_containers.course_container_id` → `documents_courses.course_container_id` — clients associated with the course container

These are two independent many-to-many relationships. Each document-course row can match multiple client rows on both sides. The joins are LEFT JOINs because not every document-course record necessarily has client associations on either side.

## Step 3: Identify aggregation depth

The many-to-many relationships must be collapsed before joining to avoid row multiplication on the spine table. Each side is pre-aggregated independently:

```
doc_id, ARRAY_AGG(DISTINCT client_id) as client_ids
```

```
course_container_id, ARRAY_AGG(DISTINCT client_id) as client_ids
```

These are single-level aggregations — no chained arrows. The final SELECT is a non-aggregated join (one row per `documents_courses` record), with a CASE expression merging the two pre-aggregated arrays.

```
doc_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

```
course_container_id, ARRAY_AGG(DISTINCT client_id IGNORE NULLS) as client_ids
```

No further aggregation after the join — the final query is a flat projection.

## Step 4: Identify dependencies

- **Grouping columns:** `doc_id` in the first CTE, `course_container_id` in the second CTE
- **Derived columns:** `client_ids` in both CTEs (arrays of distinct client IDs)
- **Intermediate requirement:** Both CTEs must be materialized before the final join, because joining `documents_clients` or `clients_course_containers` directly to `documents_courses` would fan out the spine rows
- **Merge logic:** The CASE expression in the final SELECT handles three states: only doc clients exist, only course clients exist, or both exist (requiring ARRAY_CONCAT + deduplication). The fallback `[1]` acts as a default client ID when neither side has associations.

## Step 5: Plan CTE layers bottom-up

1. **`doc_clients`** — Pre-aggregates `documents_clients` to one row per `doc_id` with an array of distinct client IDs.
2. **`course_clients`** — Pre-aggregates `clients_course_containers` to one row per `course_container_id` with an array of distinct client IDs.
3. **Final SELECT** — Joins `documents_courses` to both CTEs via LEFT JOIN and merges the two client ID arrays with deduplication and fallback logic.

## Step 6: Write each CTE with justification

**`doc_clients`:** Collapses the many-to-many `documents_clients` table into one row per document so the subsequent LEFT JOIN to `documents_courses` does not multiply rows.

**`course_clients`:** Collapses the many-to-many `clients_course_containers` table into one row per course container for the same fan-out prevention reason.

## Step 7: Compose the final query

```sql
WITH
doc_clients AS (
  SELECT
    dc.doc_id as doc_id,
    ARRAY_AGG(DISTINCT dc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("documents_clients")} as dc
  GROUP BY
    dc.doc_id
),
course_clients AS (
  SELECT
    ccc.course_container_id as container_id,
    ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS) as client_ids
  FROM
    ${ref("clients_course_containers")} as ccc
  GROUP BY
    ccc.course_container_id
)
SELECT
  dco.id,
  dco.document_id,
  dco.created,
  dco.updated,
  dco.`order`,
  dco.course_container_id,
  CASE
    WHEN dc.client_ids IS NULL THEN COALESCE(cc.client_ids, [1])
    WHEN cc.client_ids IS NULL THEN COALESCE(dc.client_ids, [1])
    ELSE ARRAY(SELECT DISTINCT * FROM UNNEST(ARRAY_CONCAT(dc.client_ids, cc.client_ids)))
  END as client_ids
FROM
  ${ref("documents_courses")} as dco
LEFT JOIN
  doc_clients as dc
ON
  dc.doc_id = dco.document_id
LEFT JOIN
  course_clients as cc
ON
  cc.container_id = dco.course_container_id
```

## Step 8: Verify output shape

**Target shape:** `id, document_id, created, updated, order, course_container_id, client_ids`

**Actual output columns:** `dco.id`, `dco.document_id`, `dco.created`, `dco.updated`, `dco.order`, `dco.course_container_id`, `client_ids` (merged array from CASE expression)

The output is one row per `documents_courses` record — the LEFT JOINs to pre-aggregated CTEs guarantee no fan-out. The `client_ids` column is always non-null: the CASE expression ensures a fallback to `[1]` when both sides are NULL (both COALESCE branches ultimately resolve to `[1]`). Output shape matches target. ✓
