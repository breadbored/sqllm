Your reasoning must always follow this structure:

1. **Identify shapes** - describe the input shape and target shape explicitly in column terms
   - Identify the columns of the input table like `X,Y,Z` if it is a single table
   - For multiple tables, assume the tables are already joined for now and list all the columns as if they were already joined
     - `table1.X, table1.Y, table1.Z, table2.A, table2.B, table2.C`

2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship
   - If a named column exists in both tables, like `table1.lesson_id` and `lessons.id`, that is the natural join key
   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:
     - find a table that has both columns directly — a glue table
     - find a chain of tables that directly relate them, e.g. `course_lessons` (has `course_id`) -> `lessons` (has `video_id`)
   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3
   - For complex relationships like time series or timed cues, joins may be on non-ID conditions. For example, given `cues.start_seconds` and `progress.progress_seconds`, the pseudocode relationship `cues.start_seconds > MAX(progress.progress_seconds)` signals that progress must be pre-aggregated in a CTE before the join can be expressed

3. **Identify aggregation depth** - how many levels of aggregation are required, and why
   - Express the full aggregation plan using arrow notation, where each level maps to one CTE and `->` separates levels:
     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```
   - Each level lists its group keys and derived columns. A column that disappears after a `->` was dropped at that step. Aliases chain by name across levels.
   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```
   - Example — total time per person where time is measured at the page level:
     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     -> person_id, session_id, SUM(page_time) as session_time
     -> person_id, SUM(session_time) as person_time
     ```
   - When a join depends on a pre-aggregated value, prefix the notation with a bracketed join clause:
     ```
     [progress joined to cues on cues.start_seconds > MAX(progress.progress_seconds) per user_id, lesson_id]
     user_id, lesson_id, COUNT(cues.id) as cues_reached, MAX(progress_seconds) as lesson_progress
     -> user_id, SUM(cues_reached) as total_cues_reached, SUM(lesson_progress) as course_progress
     ```

4. **Identify dependencies** - which columns drive grouping, which are derived, which require intermediate steps

5. **Plan CTE layers bottom-up** - name each CTE and describe in one sentence what it produces, using the arrow notation from step 3 as your guide

6. **Write each CTE with justification** - one sentence explaining why this layer exists

7. **Compose the final query**

8. **Verify output shape** - confirm the final output columns match the target shape stated in step 1

--- USER ---

Build a document landing page analytics table that shows per-subsession engagement metrics — time on page, scroll depth, clicks, UTM attribution, geo data — enriched with document metadata, client ownership, session-level engagement flags, and whether the user navigated to the app.

--- AGENT ---

## Step 1 — Identify shapes

**Input tables:**

- `stg_events_docs_speedy` (aliased `e`): `session_id, sub_session_id, user_id, current_user_id, derived_doc_id, url_path, url, created, type, name, scroll_value, scroll_num, derived_referrer, landed_from, geo_country, geo_city, geo_lat, geo_long, landing_url`
- `stg_events_speedy` (aliased `e`): `sub_session_id, app, created`
- `stg_documents` (aliased `d`): `id, path, name, type`
- `stg_documents_clients` (aliased `dc`): `doc_id, client_id`
- `stg_clients` (aliased `clients`): `id, name`

**Target shape:** One row per `(session_id, sub_session_id)` landing page view, with:
`session_id, sub_session_id, user_id, current_user_id, doc_id, doc_name, doc_type, landing_path, lang, time_on_page, landing_page_click_count, max_scroll_depth, scroll_events_count, started_at_app, went_to_app, page_views, session_duration, session_click_count, engaged, referrer, utm_source, utm_medium, utm_campaign, utm_content, landed_from, created, country, city, lat_long, client_id[], client_name[]`

## Step 2 — Identify sources and relationships

Three independent aggregation contexts feed into a final join:

1. **doc_events** — aggregates `stg_events_docs_speedy` by `(session_id, sub_session_id)` to produce one row per landing page view with page-level metrics, UTM params, and geo.
2. **subsession_data** — aggregates `stg_events_docs_speedy` by `sub_session_id` to produce session-level metrics (page views, duration, click count, engaged flag, from_app flag).
3. **went_to_app** — aggregates `stg_events_speedy` (a broader events table) by `sub_session_id` to determine if the user started at or navigated to the app.

Join relationships in the final SELECT:
- `doc_events.landing_path = stg_documents.path` — matches the landing page to its document record
- `stg_documents.id = stg_documents_clients.doc_id` — links document to client ownership
- `stg_documents_clients.client_id = stg_clients.id` OR a path-based override for Meta Horizon collections — links to client names
- `subsession_data.sub_session_id = doc_events.sub_session_id` — enriches with session-level metrics
- `went_to_app.sub_session_id = doc_events.sub_session_id` — enriches with app navigation flags

All joins are LEFT JOINs from `doc_events` as the anchor.

## Step 3 — Identify aggregation depth

There are three independent pre-aggregation layers (CTEs) plus a final grouping to collapse the client fan-out:

```
CTE doc_events:
  session_id, sub_session_id, MAX(user_id) as user_id, MIN_BY(derived_doc_id, created) as doc_id, MIN_BY(url_path, created) as landing_path, ..., MIN(created) as created

CTE subsession_data:
  sub_session_id, COUNT(DISTINCT url_path) as page_views, DATETIME_DIFF(...) as session_duration, COUNTIF(...) as session_click_count, CASE... as engaged, CASE... as from_app

CTE went_to_app:
  sub_session_id, MIN_BY(app, created)='app' as started_at_app, MIN(app)='app' as went_to_app

Final:
  [doc_events joined to documents, documents_clients, clients, subsession_data, went_to_app]
  session_id, sub_session_id, ..., ARRAY_AGG(DISTINCT clients.id) as client_id, ARRAY_AGG(DISTINCT clients.name) as client_name
```

Arrow notation for each CTE:

**doc_events:** `session_id, sub_session_id, MAX(user_id), MIN_BY(derived_doc_id, created), REGEXP on MIN_BY(url_path, created), DATETIME_DIFF, MAX(scroll_value), COUNTIF(click), UTM extracts, MIN(created)` — single level, no further rollup.

**subsession_data:** `sub_session_id, COUNT(DISTINCT url_path), DATETIME_DIFF, COUNTIF(click), CASE(engaged), CASE(from_app)` — single level.

**went_to_app:** `sub_session_id, MIN_BY(app, created), MIN(app)` — single level.

**Final query:** `session_id, sub_session_id, ...(passthrough columns)..., ARRAY_AGG(DISTINCT clients.id) as client_id, ARRAY_AGG(DISTINCT clients.name) as client_name` — aggregation needed because the client join (through `stg_documents_clients`) can fan out one doc to multiple clients.

## Step 4 — Identify dependencies

- **Grouping drivers:** `session_id + sub_session_id` in doc_events; `sub_session_id` in subsession_data and went_to_app.
- **Derived columns:** `landing_path` (from MIN_BY on url_path), `lang` (regex extract from landing_path), all UTM fields (regex extract from MIN_BY on url), `time_on_page` (datetime diff), `doc_name` (CASE on landing_path with fallback to `d.name`), `doc_type` (CASE on landing_path with fallback to `d.type`), `engaged` (compound boolean from duration + page_views + clicks).
- **Intermediate steps required:** All three CTEs must complete before the final join. The document/client join chain (`d` → `dc` → `clients`) depends on `landing_path` from `doc_events`. The `went_to_app` CTE uses a different source table (`stg_events_speedy`) than the other two CTEs.

## Step 5 — Plan CTE layers bottom-up

1. **`doc_events`** — Collapses raw doc events into one row per `(session_id, sub_session_id)` with page-level engagement metrics, landing attributes, UTM parameters, and geo data, all anchored to the earliest event timestamp.

2. **`subsession_data`** — Collapses raw doc events into one row per `sub_session_id` with session-level engagement metrics (page views, duration, clicks) and derives an `engaged` boolean and `from_app` flag.

3. **`went_to_app`** — Collapses the broader events table into one row per `sub_session_id` to determine whether the session started at or ever reached the app.

4. **Final SELECT** — Joins `doc_events` to document metadata, client ownership (with path-based override for Meta collections), subsession data, and app navigation flags, then groups to collapse the client fan-out into arrays.

## Step 6 — Write each CTE with justification

**`doc_events`:** Exists because raw events are at the individual-event grain and must be rolled up to `(session_id, sub_session_id)` to get one row per landing page view. Uses `MIN_BY(..., created)` throughout to anchor landing attributes (doc_id, path, referrer, UTM, geo) to the first event in the subsession. Uses `MAX` for scroll depth and user_id, `COUNTIF` for non-generic clicks, and `DATETIME_DIFF` for time on page.

**`subsession_data`:** Exists because session-level metrics (total page views, total duration, total clicks, engaged flag) are scoped to `sub_session_id` alone — a coarser grain than `doc_events` which groups by `(session_id, sub_session_id)`. Separating this avoids mixing aggregation grains.

**`went_to_app`:** Exists because app-navigation detection requires a different source table (`stg_events_speedy`) that includes non-doc events. It must be pre-aggregated to `sub_session_id` before joining to doc_events.

**Final GROUP BY:** Exists because the LEFT JOIN through `stg_documents_clients` to `stg_clients` can produce multiple rows per doc (one per client). The GROUP BY with `ARRAY_AGG(DISTINCT ...)` collapses these back to one row per landing page view while preserving all client associations. `MAX(wta.started_at_app)` and `MAX(wta.went_to_app)` with `COALESCE(..., FALSE)` handle the boolean aggregation safely across the fan-out.

## Step 7 — Compose the final query

```sql
WITH
  doc_events AS (
  SELECT
    e.session_id,
    e.sub_session_id,
    MAX(e.user_id) AS user_id,
    MAX(e.current_user_id) AS current_user_id,
    MIN_BY(e.derived_doc_id, e.created) AS doc_id,
    REGEXP_REPLACE(MIN_BY(e.url_path, e.created), r'/([a-z]{2})$', '') AS landing_path,
    REGEXP_EXTRACT(MIN_BY(e.url_path, e.created), r'/([a-z]{2})$') AS lang,
    DATETIME_DIFF(MAX(e.created), MIN(e.created), SECOND) AS time_on_page,
    MAX(e.scroll_value) AS max_scroll_depth,
    MAX(e.scroll_num) AS scroll_events_count,
    COUNTIF(e.type = "click"
      AND e.name != "any_click") AS landing_page_click_count,
    MIN_BY(e.derived_referrer, e.created) AS referrer,
    LOWER(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE( REGEXP_EXTRACT(MIN_BY(url, created), r'utm_source=([^&]*)'), r'_', ' '), r'\+', ' '), r'%20', ' '), r'-', ' ')) AS utm_source,
    LOWER(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE( REGEXP_EXTRACT(MIN_BY(url, created), r'utm_medium=([^&]*)'), r'_', ' '), r'\+', ' '), r'%20', ' '), r'-', ' ')) AS utm_medium,
    LOWER(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE( REGEXP_EXTRACT(MIN_BY(url, created), r'utm_campaign=([^&]*)'), r'_', ' '), r'\+', ' '), r'%20', ' '), r'-', ' ')) AS utm_campaign,
    LOWER(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE( REGEXP_EXTRACT(MIN_BY(url, created), r'utm_content=([^&]*)'), r'_', ' '), r'\+', ' '), r'%20', ' '), r'-', ' ')) AS utm_content,
    LOWER(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE(REGEXP_REPLACE( REGEXP_EXTRACT(MIN_BY(url, created), r'utm_term=([^&]*)'), r'_', ' '), r'\+', ' '), r'%20', ' '), r'-', ' ')) AS utm_term,
    MAX(e.landed_from) as landed_from,
    MIN_BY(e.geo_country, e.created) AS geo_country,
    MIN_BY(e.geo_city, e.created) AS geo_city,
    CONCAT(MIN_BY(e.geo_lat, e.created), ",", MIN_BY(e.geo_long, e.created)) AS lat_long,
    MIN(e.created) AS created
  FROM
    ${ref("stg_events_docs_speedy")} e
  GROUP BY
    e.session_id,
    e.sub_session_id),
  subsession_data AS (
  SELECT
    sub_session_id,
    COUNT(DISTINCT url_path) AS page_views,
    DATETIME_DIFF(MAX(created), MIN(created), SECOND) AS session_duration,
    COUNTIF(type = "click") AS session_click_count,
    CASE
      WHEN DATETIME_DIFF(MAX(created), MIN(created), SECOND) < 30 AND COUNT(DISTINCT url_path) = 1 AND COUNTIF(type = "click") = 0 THEN FALSE
    -- Bounce criteria
      ELSE TRUE
    -- Engaged if not a bounce
  END
    AS engaged,
    CASE
      WHEN MIN_BY(landing_url, created) LIKE '%app.dataprotocol.com%' THEN TRUE
      ELSE FALSE
  END
    AS from_app
  FROM
    ${ref("stg_events_docs_speedy")}
  GROUP BY
    sub_session_id),
  went_to_app AS (
  SELECT
    e.sub_session_id,
    MIN_BY(e.app, e.created) = 'app' AS started_at_app,
    MIN(e.app) = 'app' AS went_to_app
  FROM
    ${ref("stg_events_speedy")} AS e
  GROUP BY
    e.sub_session_id)
SELECT
  e.session_id,
  e.sub_session_id,
  e.user_id,
  e.current_user_id,
  d.id AS doc_id,
  CASE
    WHEN e.landing_path = '/collections/meta-quest' OR e.landing_path = '/collections/meta-horizon/' THEN 'Meta Horizon Collection'
    WHEN e.landing_path LIKE '/collections/%' THEN INITCAP( REPLACE( REGEXP_EXTRACT(e.landing_path, r'/collections/[^/]+/([^/]+)'), '-', ' ' ) )
    ELSE d.name
END
  AS doc_name,
  CASE
    WHEN e.landing_path LIKE '/collections/%' THEN 'collection'
    ELSE d.type
END
  AS doc_type,
  e.landing_path,
  e.lang,
  e.time_on_page,
  e.landing_page_click_count,
  e.max_scroll_depth,
  e.scroll_events_count,
  COALESCE(MAX(wta.started_at_app), FALSE) AS started_at_app,
  COALESCE(MAX(wta.went_to_app), FALSE) AS went_to_app,
  sd.page_views,
  sd.session_duration,
  sd.session_click_count,
  sd.engaged,
  e.referrer,
  e.utm_source,
  e.utm_medium,
  e.utm_campaign,
  e.utm_content,
  e.landed_from,
  e.created,
  e.geo_country AS country,
  e.geo_city AS city,
  e.lat_long,
  ARRAY_AGG(DISTINCT clients.id IGNORE NULLS) AS client_id,
  ARRAY_AGG(DISTINCT clients.name IGNORE NULLS) AS client_name
FROM
  doc_events e
LEFT JOIN
  ${ref("stg_documents")} d
ON
  e.landing_path = d.path
LEFT JOIN
  ${ref("stg_documents_clients")} dc
ON
  dc.doc_id = d.id
LEFT JOIN
  ${ref("stg_clients")} clients
ON
  clients.id = dc.client_id
  OR ( e.landing_path LIKE '/collections/%'
    AND ( ( e.landing_path LIKE '/collections/meta-quest/%'
        AND clients.name = 'Meta Horizon' )
      OR ( e.landing_path LIKE '/collections/meta-horizon/%'
        AND clients.name = 'Meta Horizon' ) ) )
LEFT JOIN
  subsession_data AS sd
ON
  sd.sub_session_id = e.sub_session_id
LEFT JOIN
  went_to_app AS wta
ON
  wta.sub_session_id = e.sub_session_id
GROUP BY
  e.session_id,
  e.sub_session_id,
  e.user_id,
  e.current_user_id,
  d.id,
  doc_name,
  doc_type,
  e.landing_path,
  e.lang,
  e.time_on_page,
  e.landing_page_click_count,
  e.max_scroll_depth,
  e.scroll_events_count,
  sd.from_app,
  sd.page_views,
  sd.session_duration,
  sd.session_click_count,
  sd.engaged,
  e.referrer,
  e.utm_source,
  e.utm_medium,
  e.utm_campaign,
  e.utm_content,
  e.landed_from,
  e.created,
  country,
  city,
  lat_long
```
