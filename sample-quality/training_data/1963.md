--- USER ---

Build a user-course detail report that shows, for each user enrolled in a course: course metadata (name, type, version), the user's progress and time-on-course metrics, their best assessment score and assessment completion counts, and the client(s) associated with the course's container. Include the user's account creation date.

## Step 1 — Identify shapes

**Input tables and their observable columns:**

- `stg_user_courses` (uc): user_id, course_id, created, updated, status
- `stg_user_course_assessments` (uca): id, user_id, course_id, assessment_id, completed, passed, score
- `stg_clients_course_containers` (ccc): course_container_id, client_id
- `stg_clients` (sc): id, name
- `agg_user_course_progress` (aucp): user_id, course_id, course_updated, course_started, course_status, course_progress_minutes, course_length_minutes, course_progress_percentage, time_on_course_minutes, time_on_course
- `stg_courses` (c): id, container_id, version, name, course_type
- `stg_users` (u): id, created

**Target shape (one row per user-course enrollment):**

user_id, course_id, container_id, version, course_name, course_type, course_started, course_updated, course_status, course_progress_minutes, time_on_course_seconds, course_length_minutes, course_progress_percentage, time_on_course_minutes, assessments_started, assessments_finished, assessments_passed, best_assessment_score, client_id (array), client_name (array), user_created

## Step 2 — Identify sources and relationships

The spine table is `stg_user_courses` (uc) — one row per user-course enrollment. All other tables attach to it:

- **uc → stg_courses (c):** `uc.course_id = c.id` — course metadata lookup (INNER JOIN, every enrollment must reference a valid course)
- **c → container_clients (cc):** `c.container_id = cc.container_id` — course belongs to a container, container maps to clients (INNER JOIN, implying every course has a container with at least one client mapping)
- **uc → user_assessments (ua):** `ua.user_id = uc.user_id AND ua.course_id = uc.course_id` — LEFT JOIN because not every enrollment has an assessment attempt
- **uc → user_course_progress (ucp):** `ucp.user_id = uc.user_id AND ucp.course_id = uc.course_id` — LEFT JOIN because not every enrollment has progress records
- **uc → stg_users (u):** `uc.user_id = u.id` — user metadata lookup (INNER JOIN)

Note: `stg_clients_course_containers` → `stg_clients` is a secondary join inside the `container_clients` CTE, joined on `ccc.client_id = sc.id` (LEFT JOIN, allowing containers with client IDs that have no matching client record).

## Step 3 — Identify aggregation depth

There are two independent pre-aggregations, each a single level:

**Assessment rollup (per user, course, assessment):**
```
user_id, course_id, assessment_id, COUNT(DISTINCT id) as assessments_started, SUM(completed) as assessments_finished, SUM(completed) as courses_completed, SUM(passed) as course_passed, MAX(score) as scored
```

This is a single-level aggregation. The grain is user_id + course_id + assessment_id. Note: the query groups by assessment_id but the final SELECT joins on only user_id + course_id — this means if a user has multiple assessments for a course, the LEFT JOIN will produce multiple rows. This appears to be an intentional or accepted fan-out at the assessment level.

**Container-client rollup (per container):**
```
course_container_id, ARRAY_AGG(DISTINCT client_id) as client_id, ARRAY_AGG(DISTINCT name) as client_names
```

Single-level aggregation collapsing multiple client associations per container into arrays.

**User course progress:** No aggregation — this CTE is a pure rename/passthrough from `agg_user_course_progress`, which is already pre-aggregated upstream at the user_id + course_id grain.

## Step 4 — Identify dependencies

- **Grouping drivers:** `user_id + course_id` is the intended final grain (from `stg_user_courses`), though the assessment join on `user_id + course_id` against a CTE grouped by `user_id + course_id + assessment_id` may fan out rows per assessment.
- **Derived columns:** `assessments_started`, `assessments_finished`, `course_passed`, `scored` are derived from aggregation over `stg_user_course_assessments`. `client_id` and `client_names` arrays are derived from aggregation over the container-client relationship.
- **Intermediate steps required:** Assessments must be pre-aggregated before joining to the spine. Container-client relationships must be collapsed into arrays before joining. Progress is already pre-aggregated in the upstream model.

## Step 5 — Plan CTE layers bottom-up

1. **`user_assessments`** — Aggregates assessment-level data per user + course + assessment, producing attempt counts, completion counts, pass counts, and best score.
2. **`container_clients`** — Collapses the many-to-many container-client relationship into one row per container with client IDs and names as arrays.
3. **`user_course_progress`** — Renames and passes through the already-aggregated progress model at user + course grain.

Final SELECT joins the spine (`stg_user_courses`) to all three CTEs plus `stg_courses` and `stg_users` for metadata.

## Step 6 — Write each CTE with justification

**`user_assessments`:** Pre-aggregates raw assessment records to one row per user + course + assessment so that attempt counts, completion flags, pass flags, and best scores are available for the final join. Without this, the raw assessment rows would fan out the spine uncontrollably.

**`container_clients`:** Collapses the container-to-client mapping into arrays so that the final join on `container_id` produces exactly one row per container, carrying all associated client IDs and names without duplicating the spine.

**`user_course_progress`:** A passthrough/rename CTE that normalizes column names from the upstream `agg_user_course_progress` model. No aggregation is performed here because the upstream model is already at the correct grain.

## Step 7 — Compose the final query

```sql
WITH 
user_assessments AS (
  SELECT
    uca.user_id as user_id,
    uca.course_id as course_id,
    uca.assessment_id as assessment_id,
    COUNT(DISTINCT uca.id) as assessments_started,
    SUM(uca.completed) as assessments_finished,
    SUM(uca.completed) as courses_completed,
    SUM(uca.passed) as course_passed,
    MAX(uca.score) as scored
  FROM
    ${ref("stg_user_course_assessments")} as uca
  GROUP BY
    uca.user_id,
    uca.course_id,
    uca.assessment_id
),

container_clients AS (
  SELECT
    ccc.course_container_id as container_id,
    ARRAY_AGG(DISTINCT ccc.client_id IGNORE NULLS ORDER BY ccc.client_id ASC) as client_id,
    ARRAY_AGG(DISTINCT sc.name IGNORE NULLS) as client_names

  FROM
    ${ref("stg_clients_course_containers")} as ccc

  LEFT JOIN
    ${ref("stg_clients")} as sc
  ON
    ccc.client_id = sc.id

  GROUP BY
    ccc.course_container_id
),

user_course_progress AS (
    SELECT
        aucp.user_id as user_id,
        aucp.course_id as course_id,
        aucp.course_updated as course_update,
        aucp.course_started as course_started,
        aucp.course_updated as course_updated_on,
        aucp.course_status as course_status,
        aucp.course_progress_minutes as course_progress_minutes,
        aucp.course_length_minutes as course_length_minutes,
        aucp.course_progress_percentage as course_complete_percent,
        aucp.time_on_course_minutes as time_on_course_minutes,
        aucp.time_on_course as time_on_course_seconds
    FROM
        ${ref("agg_user_course_progress")} as aucp
)

SELECT
  uc.user_id as user_id,
  uc.course_id as course_id,
  cc.container_id as container_id,
  c.version as version,
  c.name as course_name,
  c.course_type as course_type,
  uc.created as course_started,
  uc.updated as course_updated,
  uc.status as course_status,
  ucp.course_progress_minutes as course_progress_minutes,
  ucp.time_on_course_seconds as time_on_course_seconds,
  ucp.course_length_minutes as course_length_minutes,
  ucp.course_complete_percent as course_progress_percentage,
  ucp.time_on_course_minutes as time_on_course_minutes,
  COALESCE(ua.assessments_started, 0) as assessments_started,
  ua.assessments_finished as assessments_finished,
  ua.course_passed as assessments_passed,
  ua.scored as best_assessment_score,
  cc.client_id as client_id,
  cc.client_names as client_name,
  u.created as user_created,

FROM
  ${ref("stg_user_courses")} as uc

LEFT JOIN
  user_assessments as ua
ON
  ua.user_id = uc.user_id
AND
  ua.course_id = uc.course_id

JOIN
  ${ref("stg_courses")} as c
ON
  uc.course_id = c.id

JOIN
  container_clients as cc
ON
  cc.container_id = c.container_id

LEFT JOIN
  user_course_progress as ucp
ON
  ucp.user_id = uc.user_id
AND
  ucp.course_id = uc.course_id

JOIN
  ${ref("stg_users")} as u
ON
  u.id = uc.user_id
```

However, the `user_assessments` CTE groups by `user_id, course_id, assessment_id`, and the LEFT JOIN matches on only `user_id, course_id`.
