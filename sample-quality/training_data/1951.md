2. **Identify sources and relationships** - assuming this is not a CROSS JOIN, identify what columns define the join relationship

   - If there is no discernible direct relationship between two tables, find a 2nd or 3rd+ degree glue table. If you need `course_id, video_id` but no table contains both:

   - If two tables share multiple columns like `lesson_id` and `user_id`, join on all matching columns to keep the relationship as tight as possible. If this risks row duplication or broken aggregation, either find a stronger relationship or deepen the aggregation as described in step 3

3. **Identify aggregation depth** - how many levels of aggregation are required, and why

     ```
     group_keys, AGG(expr) as alias -> group_keys, AGG(alias) as alias -> final_group_keys, AGG(alias)
     ```

   - Example — per-user per-course progress where progress is recorded at the lesson level:
     ```
     user_id, course_id, lesson_id, MAX(progress_seconds) as lesson_progress
     -> user_id, course_id, SUM(lesson_progress) as course_progress
     ```

     ```
     person_id, session_id, page_id, MAX(created)-MIN(created) as page_time
     ```

## Tables
### agg_client_transactions
| column | type |
|---|---|
| user_id | INT64 |
| user_created | DATETIME |
| day | DATETIME |
| client_id | ARRAY<INT64> |

## Question
For each client and half-year period, show the count of unique users, new users, returning users, users who appear in both categories, half-year-scoped new and returning users, total sessions, returning user sessions, and average sessions per user. A user is "new" on the day of their first-ever transaction for that client, and "returning" on any subsequent day.

--- AGENT ---

## Step 1 — Identify shapes

**Input shape:**
The source table is `agg_client_transactions` with at least these columns:
- `user_id` — identifies the user
- `client_id` — a repeated/array field of client IDs associated with the transaction
- `day` — the date of the transaction/session

**Target shape (one row per client per half-year period):**
```
unique_users, new_users, returning_users, new_users_returning,
half_year_new_users, half_year_returning_users,
total_sessions, returning_user_sessions, sessions_per_user,
half_start, client_id
```

## Step 2 — Identify sources and relationships

There is a single source table (`agg_client_transactions`) used twice — once to compute each user's first-ever transaction day per client, and once to produce the per-row event stream with half-year period assignment.

The `client_id` column is an ARRAY that must be unnested with `UNNEST(ct.client_id) AS client_id_expanded` to produce one row per user × client × day.

The join relationship is between the flattened event rows and the per-user-per-client minimum day, joined on both `user_id` and `client_id`. This is a tight two-column join — both columns are required to correctly attribute "new" vs "returning" status per client.

There is also an intermediate CTE (`meta_quest_min`) that passes through `client_transactions_min` unchanged — it acts as a naming alias (likely a vestige of a filter that was removed, or a placeholder for a specific client filter). It does not transform the data.

## Step 3 — Identify aggregation depth

The aggregation has two conceptual levels:

**Level 1 — Pre-aggregation:** compute each user's earliest day per client.

**Level 2 — Final aggregation:** join the event stream to the min-day lookup table and aggregate per half-year period per client, computing conditional distinct counts.

Arrow notation:

```
user_id, client_id, MIN(day) as min_day
-> [joined to event rows on user_id, client_id]
   half_start, client_id,
   COUNT(DISTINCT user_id) as unique_users,
   COUNT(DISTINCT CASE WHEN min_day = day THEN user_id END) as new_users,
   COUNT(DISTINCT CASE WHEN min_day != day THEN user_id END) as returning_users,
   (new_users + returning_users) - unique_users as new_users_returning,
   COUNT(DISTINCT CASE WHEN min_day within half-year THEN user_id END) as half_year_new_users,
   COUNT(DISTINCT CASE WHEN min_day outside half-year THEN user_id END) as half_year_returning_users,
   COUNT(user_id) as total_sessions,
   COUNT(CASE WHEN min_day != day THEN user_id END) as returning_user_sessions,
   ROUND(total_sessions / unique_users, 2) as sessions_per_user
```

## Step 4 — Identify dependencies

- **Grouping keys (final):** `half_start`, `client_id`
- **Derived (pre-aggregation):** `min_day` per `(user_id, client_id)` — required before the final join/aggregation can classify users as new or returning
- **Derived (inline):** `half_start` is computed from `day` using a CASE expression that maps months 1–6 to Jan 1 and months 7–12 to Jul 1 of the same year
- **Conditional counts depend on:** comparing `min_day` to `day` (lifetime new/returning) and comparing `min_day` to the `half_start`/`half_start + 6 months` window (period-scoped new/returning)
- `new_users_returning` is a derived arithmetic column: `(new_users + returning_users) - unique_users`, which captures users who had their first-ever day in this period AND also had subsequent days in this same period — they appear in both the new and returning counts

## Step 5 — Plan CTE layers bottom-up

1. **`client_transactions_min`** — Unnests the `client_id` array and computes each user's earliest transaction day per client.
2. **`meta_quest_min`** — Pass-through alias of `client_transactions_min` (no transformation; likely a naming convention or placeholder for a filter).
3. **`half_year_periods`** — Unnests the `client_id` array on the full event stream and computes the `half_start` period marker for each row.
4. **Final SELECT** — Joins `half_year_periods` to `meta_quest_min` on `(user_id, client_id)`, then aggregates per `(half_start, client_id)` with conditional distinct counts for user classification.

## Step 6 — Write each CTE with justification

**`client_transactions_min`:** Exists to pre-aggregate the earliest transaction day per user per client. This is required before the final query can classify each event row as "new" or "returning" by comparing the row's day to the user's first-ever day for that client.

**`meta_quest_min`:** Exists as a pass-through renaming layer. It selects all three columns from `client_transactions_min` without transformation. This is likely a remnant of a design where different client segments (e.g., Meta Quest devices) would be filtered here.

**`half_year_periods`:** Exists to flatten the event stream (unnesting `client_id`) and compute the half-year period bucket (`half_start`) for each row. This provides the granularity and grouping key for the final aggregation.

## Step 7 — Compose the final query

```sql
WITH
  client_transactions_min AS (
  SELECT
    ct.user_id,
    client_id_expanded AS client_id,
    MIN(ct.`day`) AS min_day
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS client_id_expanded
  GROUP BY
    ct.user_id,
    client_id_expanded ),
  meta_quest_min AS (
  SELECT
    ctm.user_id,
    ctm.client_id,
    ctm.min_day
  FROM
    client_transactions_min AS ctm ),
  half_year_periods AS (
  SELECT
    ct.user_id,
    client_id_expanded AS client_id,
    ct.`day`,
    CASE
      WHEN EXTRACT(MONTH FROM `day`) <= 6 THEN DATE(EXTRACT(YEAR FROM `day`), 1, 1)
      ELSE DATE(EXTRACT(YEAR
      FROM
        `day`), 7, 1)
  END
    AS half_start
  FROM
    ${ref("agg_client_transactions")} AS ct,
    UNNEST(ct.client_id) AS client_id_expanded )
SELECT
  COUNT(DISTINCT ct.user_id) AS unique_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day = ct.`day` THEN ct.user_id
  END
    ) AS new_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day != ct.`day` THEN ct.user_id
  END
    ) AS returning_users,
  ( COUNT( DISTINCT
      CASE
        WHEN mqm.min_day != ct.`day` THEN ct.user_id
    END
      ) + COUNT( DISTINCT
      CASE
        WHEN mqm.min_day = ct.`day` THEN ct.user_id
    END
      ) ) - COUNT(DISTINCT ct.user_id) AS new_users_returning,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day >= half_start AND mqm.min_day < DATE_ADD(half_start, INTERVAL 6 MONTH) THEN ct.user_id
  END
    ) AS half_year_new_users,
  COUNT( DISTINCT
    CASE
      WHEN mqm.min_day < half_start OR mqm.min_day >= DATE_ADD(half_start, INTERVAL 6 MONTH) THEN ct.user_id
  END
    ) AS half_year_returning_users,
  COUNT(ct.user_id) AS total_sessions,
  COUNT(
    CASE
      WHEN mqm.min_day != ct.`day` THEN ct.user_id
  END
    ) AS returning_user_sessions,
  ROUND(COUNT(ct.user_id) / COUNT(DISTINCT ct.user_id), 2) AS sessions_per_user,
  --   FORMAT_DATE('%b %Y', half_start) AS half_year,
  half_start,
  ct.client_id AS client_id
FROM
  half_year_periods AS ct
INNER JOIN
  meta_quest_min AS mqm
ON
  mqm.user_id = ct.user_id
  AND mqm.client_id = ct.client_id
GROUP BY
  half_start,
  ct.client_id
ORDER BY
  half_start
```

Target shape from Step 1:

```
half_year_new_users, half_year_returning_users,
total_sessions, returning_user_sessions, sessions_per_user,
half_start, client_id
```

Final SELECT produces exactly these 11 columns, grouped by `(half_start, client_id)`, ordered by `half_start`. ✓

Each row represents one half-year period for one client, with user classification metrics computed by comparing each event's day against the user's globally earliest day for that client. The distinction between "lifetime" new/returning (`min_day = day`) and "period-scoped" new/returning (`min_day` within/outside the half-year window) provides two complementary lenses on user acquisition. The `new_users_returning` metric captures overlap — users whose first-ever day falls in this period but who also have additional days in the same period, meaning they count as both new and returning. ✓
