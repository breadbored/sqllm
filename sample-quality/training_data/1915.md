## 2. Identify sources and relationships

**Join chain:**

1. `uc → u` on `u.id = uc.user_id` — INNER JOIN. Filters to users that exist in `stg_users`. No columns are selected from `u`, so this is purely a validity filter (ensures referential integrity — orphaned enrollments are excluded).

2. `uc → c` on `c.id = uc.course_id` — INNER JOIN. Brings in course metadata (`container_id, name, course_type, version`).

3. `c → ccc` on `ccc.course_container_id = c.container_id` — INNER JOIN. Links courses to clients via a container-level glue table. This is a many-to-many relationship: one container can belong to multiple clients. This is the source of the fan-out that requires `ARRAY_AGG(DISTINCT ...)` for client columns.

4. `ccc → clients` on `clients.id = ccc.client_id` — INNER JOIN. Resolves client IDs to names.

5. `uc → ucp` on `(ucp.course_id = uc.course_id AND ucp.user_id = uc.user_id)` — LEFT JOIN. Brings in pre-aggregated daily progress rows. LEFT because a user may have an enrollment with no recorded progress yet.

6. `ucp → uca` on `(ucp.user_id = uca.user_id AND ucp.course_id = uca.course_id AND DATE_TRUNC(uca.created, DAY) >= ucp.progress_date)` — LEFT JOIN. Attaches assessments to progress rows, but only assessments created on or after the progress date. This is a temporal inequality join — each progress-date row accumulates all assessments from that date forward. This means assessment counts are **cumulative from that progress date onward**, not daily snapshots.

**Key observation on the assessment join:** The condition `DATE_TRUNC(uca.created, DAY) >= ucp.progress_date` means every assessment row joins to every progress-date row that precedes it. A single assessment taken on day 10 will appear in the aggregation for progress rows on days 1 through 10. This is intentional — it gives a "from this day forward, how many assessments eventually happened" view.

## 3. Identify aggregation depth

This query has a single level of aggregation. The GROUP BY contains all non-aggregated columns from the joined result, and the aggregates operate directly on the fan-out produced by the joins (multiple clients per container, multiple assessments per user-course-date window).

## 4. Identify dependencies

**Grouping drivers:** The grain is `(user_id, course_id, progress_date)` plus all the scalar progress metrics from `ucp`. Every `ucp` column in the SELECT is also in the GROUP BY, confirming `ucp` contributes exactly one row per `(user_id, course_id, progress_date)`.

**Derived columns:**
- `progress_date` — `IFNULL(ucp.progress_date, uc.created)`: falls back to enrollment date when no progress exists.
- `derived_course_status` — a CASE expression combining `ucp.total_course_progress_percentage` (cast to FLOAT64) with `uc.status`. Four tiers: completed (100% + status match), 100% (≥95%), in_progress (<95%), not_started (NULL progress).

**Fan-out sources requiring aggregation:**
- `clients` via `ccc` — multiple clients per container → `ARRAY_AGG(DISTINCT ... IGNORE NULLS)`
- `uca` via temporal inequality join — multiple assessments per user-course after each progress date → `COUNT`, `SUM`, `MAX`

**Cross-join risk:** The two fan-out sources (clients × assessments) multiply against each other. Using `DISTINCT` on the client arrays and `COUNT/SUM/MAX` on assessments mitigates double-counting for clients but **inflates assessment counts** by the number of distinct clients per container. If a course belongs to 3 clients, each assessment row appears 3× in the pre-GROUP BY result, and `COUNT(uca.completed)` will triple-count. This is a potential correctness issue in the original query, but per the instructions, we include it verbatim.

## 7. Compose the final query

```sql
SELECT
  uc.user_id,
  uc.course_id,
  c.container_id,
  c.name AS course_name,
  c.course_type,
  c.version,
  IFNULL(ucp.progress_date, uc.created) AS progress_date,
  CASE
    WHEN CAST(ucp.total_course_progress_percentage AS FLOAT64) = 1 AND uc.status = "completed" THEN "completed"
    WHEN CAST(ucp.total_course_progress_percentage AS FLOAT64) >= 0.95 THEN "100%"
    WHEN CAST(ucp.total_course_progress_percentage AS FLOAT64) < 0.95 THEN "in_progress"
    WHEN CAST(ucp.total_course_progress_percentage AS FLOAT64) IS NULL THEN "not_started"
END
  AS derived_course_status,
  uc.status AS course_status,
  uc.created AS course_started,
  uc.completion_date AS course_completed,
  ucp.daily_progress_seconds,
  ucp.current_progress_seconds,
  ucp.total_progress_seconds,
  ucp.progress_percentage_made,
  ucp.current_course_progress_percentage,
  ucp.total_course_progress_percentage,
  ucp.course_length,
  ucp.daily_time_on_course,
  ucp.total_time_on_course,
  COUNT(uca.completed) AS assessments_started,
  SUM(uca.completed) AS assessments_finished,
  SUM(uca.passed) AS assessments_passed,
  MAX(uca.score) AS best_assessment_score,
  ARRAY_AGG(DISTINCT clients.id IGNORE NULLS ) AS client_id,
  ARRAY_AGG(DISTINCT clients.name IGNORE NULLS ) AS client_name,
FROM
  ${ref("stg_user_courses")} uc
JOIN
  ${ref("stg_users")} u
ON
  u.id = uc.user_id
JOIN
  ${ref("stg_courses")} c
ON
  c.id = uc.course_id
JOIN
  ${ref("clients_course_containers")} ccc
ON
  ccc.course_container_id = c.container_id
JOIN
  ${ref("stg_clients")} clients
ON
  clients.id = ccc.client_id
LEFT JOIN
  ${ref("agg_daily_user_course_progress")} ucp
ON
  ucp.course_id = uc.course_id
  AND ucp.user_id = uc.user_id
LEFT JOIN
  ${ref("stg_user_course_assessments")} AS uca
ON
  ucp.user_id = uca.user_id
  AND ucp.course_id = uca.course_id
  AND DATE_TRUNC(uca.created, DAY) >= ucp.progress_date
GROUP BY
  uc.user_id,
  uc.course_id,
  c.container_id,
  c.name,
  c.course_type,
  c.version,
  ucp.progress_date,
  uc.status,
  uc.created,
  uc.completion_date,
  ucp.daily_progress_seconds,
  current_progress_seconds,
  ucp.total_progress_seconds,
  ucp.progress_percentage_made,
  ucp.total_course_progress_percentage,
  ucp.course_length,
  ucp.daily_time_on_course,
  ucp.total_time_on_course,
  ucp.current_course_progress_percentage
ORDER BY
  user_id,
  course_id,
  progress_date
```
